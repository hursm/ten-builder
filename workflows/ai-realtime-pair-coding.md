# AI 에이전트 실시간 페어 코딩 워크플로우

> 개발자 2명 + AI 에이전트 1명으로 구성된 실시간 페어 코딩 세션 설계 및 운영 방법

## 개요

전통적인 페어 프로그래밍은 한 명이 타이핑하고 다른 한 명이 리뷰하는 방식이에요. 여기에 AI 에이전트를 세 번째 멤버로 추가하면 생산성이 크게 올라가요.

2026년 기준으로 AI 페어 코딩은 개발 속도를 평균 40~55% 높이는 것으로 측정되고 있어요. 단, 이 수치는 AI를 제대로 역할 배분해서 쓸 때의 이야기예요.

이 워크플로우는 **드라이버(Driver)**, **네비게이터(Navigator)**, **AI 에이전트** 세 역할을 명확히 나누고, 각자가 언제 무엇을 해야 하는지 구체적으로 정리해요.

## 사전 준비

- AI 코딩 도구: Claude Code, Cursor, Windsurf 중 하나 (팀 공통으로 통일)
- 공유 편집 환경: VS Code Live Share 또는 Cursor 공유 세션
- 세션 컨텍스트 파일: `PAIR_SESSION.md` (아래 설명 참고)
- 최소 1시간 확보 (집중 세션 기준)

## 역할 정의

| 역할 | 담당 | 핵심 책임 |
|------|------|----------|
| **드라이버** | 개발자 A | 실제 코드 입력, AI 프롬프트 작성 |
| **네비게이터** | 개발자 B | 방향 결정, AI 출력 검토, 다음 태스크 준비 |
| **AI 에이전트** | Claude Code 등 | 코드 생성, 에러 분석, 리팩토링 제안 |

역할은 25~30분마다 드라이버↔네비게이터를 교체해요.

## 설정

### Step 1: 세션 컨텍스트 파일 생성

세션 시작 전에 `PAIR_SESSION.md`를 프로젝트 루트에 만들어요. AI 에이전트가 이 파일을 읽고 세션 맥락을 파악해요.

```markdown
# 페어 코딩 세션

## 오늘 목표
- [ ] 장바구니 API 엔드포인트 추가
- [ ] 유닛 테스트 작성
- [ ] PR 초안 생성

## 기술 스택
- Node.js 22, Express, PostgreSQL
- 테스트: Jest + Supertest

## 제약 조건
- 기존 auth 미들웨어 변경 금지
- 응답 포맷: { data, error, meta } 구조 유지

## 현재 진행
드라이버: 김개발 / 네비게이터: 이코딩
```

### Step 2: AI 에이전트 세션 시작 프롬프트

```
PAIR_SESSION.md를 읽고 오늘 목표를 확인해줘.
현재 codebase 구조를 파악한 뒤 장바구니 API 구현 계획 3단계로 제안해줘.
각 단계별 예상 소요 시간도 포함해줘.
```

### Step 3: 채팅 채널 설정

팀 Slack/Discord에 `#pair-session-log` 채널을 만들고, AI가 중요한 결정을 내릴 때마다 네비게이터가 요약을 올려요.

```
[10:35] AI가 제안한 DB 스키마 변경 → 검토 후 채택
[10:52] 에러 처리 패턴: throw vs return { error } → return 방식으로 결정
```

## 사용 방법

### 기본 루프 (25분 사이클)

```
[드라이버] 태스크 선택 → AI 에이전트에 구현 요청
[AI 에이전트] 코드 생성 + 설명
[네비게이터] 출력 검토 → 피드백 제공
[드라이버] 수정 반영 또는 AI에 재요청
[양쪽] 실행/테스트 확인
25분 후 → 역할 교체
```

### 네비게이터의 AI 출력 검토 체크리스트

```
□ 기존 코드 스타일과 일치하는가?
□ 엣지 케이스가 처리되어 있는가?
□ 보안 취약점(SQL 인젝션, 입력값 검증 등)이 없는가?
□ 테스트 코드도 함께 생성되었는가?
□ TODO/FIXME가 남겨지지 않았는가?
```

### AI에게 효과적인 페어 코딩 프롬프트 패턴

```bash
# 패턴 1: 드라이버가 구현 요청
"cartService에 addItem(userId, productId, qty) 함수 추가해줘.
DB는 PostgreSQL, 이미 있는 pool.query 패턴 따라줘."

# 패턴 2: 네비게이터가 리뷰 요청
"방금 추가한 addItem 함수에서 동시성 문제 가능한 부분 분석해줘.
race condition이 발생할 수 있는 시나리오를 설명하고
트랜잭션 처리 방법 제안해줘."

# 패턴 3: 역할 교체 시 컨텍스트 이전
"지금까지 한 작업 요약해줘.
다음 드라이버가 바로 이어받을 수 있도록
현재 미완성 부분과 다음 태스크를 정리해줘."
```

## 커스터마이징

| 설정 | 기본값 | 설명 |
|------|--------|------|
| 사이클 시간 | 25분 | 팀 집중력에 맞게 15~40분 조정 |
| AI 도구 | Claude Code | 팀 공통 도구 통일 권장 |
| 컨텍스트 파일 | PAIR_SESSION.md | 프로젝트당 하나 유지 |
| 리뷰 채널 | 팀 채팅 | 세션 당일 아카이브 |

## 고급 패턴: 비동기 페어 코딩

팀원이 다른 시간대에 있을 때도 AI 에이전트를 중간에 두고 비동기로 페어 코딩할 수 있어요.

```markdown
# PAIR_SESSION.md (비동기 버전)

## 현재 상태 (오후 6시 기준, 김개발 작성)
- addItem 구현 완료, 테스트 코드 80% 작성
- 미완: removeItem 함수, 재고 검증 로직

## 이코딩에게 (내일 아침 이어받기)
AI에게 이 파일 읽히고 removeItem부터 시작해줘.
재고 검증은 inventory_service.checkStock() 참고.
```

이코딩이 다음날 작업 시작 시:

```
PAIR_SESSION.md 읽고 어제 작업 이어받아줘.
removeItem 구현부터 시작하면 돼.
```

## 문제 해결

| 문제 | 해결 |
|------|------|
| AI가 기존 패턴과 다른 코드를 생성 | PAIR_SESSION.md에 제약 조건 구체적으로 추가 |
| 두 개발자가 AI 출력에 의견 충돌 | 네비게이터가 최종 결정권, 이유를 세션 로그에 기록 |
| 컨텍스트 윈도우 부족 | 25분마다 역할 교체 시 AI에게 요약 요청 후 새 대화 시작 |
| 한쪽이 AI에만 의존하는 경향 | 타이머 기반 역할 교체 엄수, 네비게이터가 AI 없이 방향 결정 |
