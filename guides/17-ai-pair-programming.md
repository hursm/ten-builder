# 17. AI 페어 프로그래밍 실전 가이드

> 혼자 코딩하지만 혼자가 아닌 — AI와 짝을 이뤄 코드를 작성하는 실전 패턴

---

## 왜 AI 페어 프로그래밍인가?

전통적인 페어 프로그래밍은 두 명의 개발자가 한 화면 앞에서 함께 작업하는 방식이에요. 한 명이 코드를 작성하고(Driver), 한 명이 방향을 잡아주죠(Navigator). AI 페어 프로그래밍도 같은 구조를 따르되, Navigator 역할을 AI가 수행해요.

실제로 이 방식을 적용한 개발자들의 작업 완료율은 21~60% 향상되었다는 조사 결과가 있어요. 단순히 "코드 생성"을 넘어서, AI를 페어로 쓰는 건 **사고 과정 자체를 바꾸는 일**이에요.

---

## 핵심 패턴 3가지

### 패턴 1: Driver-Navigator 모드

**내가 코드를 작성하고, AI가 방향을 잡아주는 모드.**

이 패턴이 적합한 경우:
- 새로운 기능을 설계할 때
- 아키텍처 결정이 필요할 때
- 코드를 작성하면서 실시간으로 피드백 받고 싶을 때

```
[내 역할: Driver]
코드를 직접 작성. 구현 세부사항을 결정.

[AI 역할: Navigator]  
설계 방향 제안. 빠뜨린 엣지 케이스 지적.
대안 접근법 제시.
```

**실전 프롬프트 예시:**

```
지금 사용자 인증 미들웨어를 작성하고 있어.
JWT 토큰 검증 → 사용자 정보 조회 → req.user에 할당하는 흐름이야.

내가 작성한 코드를 보고:
1. 빠뜨린 에러 케이스가 있는지
2. 보안상 주의할 점
3. 더 나은 구조가 있다면

알려줘. 코드를 대신 작성하지 말고 방향만 잡아줘.
```

### 패턴 2: Ping-Pong 모드

**AI가 코드를 작성하고, 내가 리뷰하는 모드. 번갈아가며 진행.**

이 패턴이 적합한 경우:
- 보일러플레이트가 많은 작업
- 테스트 코드 작성
- 잘 알려진 패턴 구현 (CRUD, 폼 처리 등)

```
[Round 1] AI가 초안 작성 → 내가 리뷰 & 수정 요청
[Round 2] AI가 수정본 작성 → 내가 리뷰 & 승인
[Round 3] 다음 단위로 이동
```

**실전 프롬프트 예시:**

```
이 스키마 기반으로 CRUD API를 작성해줘:
- User: id, email, name, createdAt
- 엔드포인트: GET /users, GET /users/:id, POST /users, PUT /users/:id, DELETE /users/:id
- 검증: email 형식, name 필수

한 엔드포인트씩 작성해. 내가 리뷰하고 OK 하면 다음으로 넘어가자.
```

### 패턴 3: Rubber Duck+ 모드

**AI에게 설명하면서 스스로 답을 찾는 모드. 전통적인 러버덕 디버깅의 진화형.**

이 패턴이 적합한 경우:
- 버그 원인을 찾고 있을 때
- 설계 결정에서 고민이 있을 때
- 복잡한 로직을 정리할 때

```
[내 역할] 문제를 설명
[AI 역할] 정리된 질문으로 되묻기
[결과] 설명하는 과정에서 답을 발견
```

고무 오리와 다른 점은 — AI가 실제로 유용한 질문을 던져준다는 거예요.

## 도구별 페어 프로그래밍 패턴

### Claude Code (터미널)

터미널 기반이라 **대화형 페어 프로그래밍에 적합**해요.

| 상황 | 접근법 |
|------|--------|
| 새 기능 개발 | 프로젝트 구조 설명 → 단계별 구현 요청 |
| 버그 수정 | 에러 로그 + 관련 파일 공유 → 원인 분석 |
| 리팩토링 | 현재 코드 의도 설명 → 개선 방향 논의 |

```bash
# CLAUDE.md에 페어 프로그래밍 규칙 추가
cat >> CLAUDE.md << 'EOF'

## Pair Programming Rules
- 코드를 생성하기 전에 항상 계획을 먼저 설명할 것
- 한 번에 한 파일만 수정할 것
- 수정 전후의 차이를 명확히 설명할 것
EOF
```

### Cursor (IDE)

IDE 내장이라 **인라인 편집과 채팅을 구분해서 쓰는 게 핵심**이에요.

| 상황 | 도구 | 이유 |
|------|------|------|
| 단일 함수 수정 | `CMD+K` (인라인) | 컨텍스트 전환 없이 빠르게 |
| 여러 파일 변경 | Chat + `@file` | 전체 맥락을 보면서 |
| 설계 논의 | Chat | 긴 대화가 필요하니까 |
| 타입 수정 | `CMD+K` (인라인) | 간단한 변경이니까 |

```
// Cursor .cursorrules 페어 프로그래밍 설정
{
  "rules": [
    "한 번에 하나의 변경사항만 제안해주세요",
    "변경 이유를 항상 설명해주세요",
    "테스트가 필요한 경우 테스트 코드도 함께 제안해주세요"
  ]
}
```

## 컨텍스트 전달의 기술

AI 페어 프로그래밍에서 가장 중요한 건 **AI에게 적절한 맥락을 전달하는 것**이에요.

### 컨텍스트 3계층

```
[Layer 1: 프로젝트 컨텍스트]
CLAUDE.md, .cursorrules 등 → 한 번 설정하면 매번 자동 적용

[Layer 2: 세션 컨텍스트]  
이번 작업의 목표, 관련 파일, 제약 조건 → 세션 시작 시 공유

[Layer 3: 턴별 컨텍스트]
현재 고민, 선택지, 질문 → 각 대화에서 구체적으로 전달
```

### 효과적인 컨텍스트 전달 예시

```
❌ 나쁜 예:
"이 코드 고쳐줘"

✅ 좋은 예:
"이 미들웨어에서 토큰 만료 시 401 대신 500이 리턴되는 문제가 있어.
jwt.verify()에서 TokenExpiredError를 잡지 못하는 것 같은데,
catch 블록에서 에러 타입 분기가 빠져있는 건지 확인해줘.
코드를 바로 수정하지 말고 원인부터 설명해줘."
```

### 규칙 파일 활용

```markdown
# CLAUDE.md 페어 프로그래밍 섹션

## 대화 규칙
- 코드를 작성하기 전에 계획을 설명하세요
- 불확실한 부분은 가정하지 말고 질문하세요
- 한 번에 100줄 이상 수정하지 마세요

## 프로젝트 관례
- 에러 처리: 커스텀 에러 클래스 사용 (src/errors/)
- 로깅: winston logger, 절대 console.log 사용 금지
- 테스트: 모든 퍼블릭 함수에 단위 테스트 필수
```

## 커밋 전략

AI와 함께 작업할 때 커밋을 어떻게 가져갈지도 중요해요.

### 짧은 주기 커밋

```bash
# 한 기능의 한 단계가 끝날 때마다 커밋
git add -p  # 변경사항을 하나씩 확인하며 스테이징
git commit -m "feat(auth): add JWT token validation middleware"

# AI가 생성한 코드를 리뷰 후 커밋
git diff --cached  # 스테이징된 변경사항 최종 확인
git commit -m "feat(auth): add token refresh endpoint"
```

### 커밋 전 체크리스트

| 단계 | 확인 사항 |
|------|----------|
| 1. 코드 읽기 | AI가 생성한 코드를 한 줄씩 읽었는가 |
| 2. 테스트 실행 | 기존 테스트가 깨지지 않았는가 |
| 3. 의도 확인 | 내가 요청한 것과 결과가 일치하는가 |
| 4. 하드코딩 | 시크릿이나 민감 정보가 포함되지 않았는가 |
| 5. 스타일 | 프로젝트 관례와 맞는가 |

## 흔한 실수와 해결법

| 실수 | 문제 | 해결 |
|------|------|------|
| AI 출력을 그대로 커밋 | 이해하지 못한 코드가 프로덕션에 들어감 | 모든 라인을 설명할 수 있어야 커밋 |
| 너무 큰 단위 요청 | AI가 맥락을 놓치고 엉뚱한 코드 생성 | 한 함수, 한 파일씩 진행 |
| 채팅만 사용 | 간단한 수정에도 긴 대화 | 인라인 편집 활용 |
| 규칙 파일 미설정 | 매번 같은 설명 반복 | CLAUDE.md / .cursorrules 한 번 설정 |
| 장시간 세션 | 컨텍스트가 오래되어 품질 저하 | 2시간마다 새 세션 시작 |
| AI에게 모든 결정 위임 | 아키텍처 일관성 깨짐 | 설계 결정은 내가, 구현은 AI와 함께 |

## 페어 프로그래밍 세션 구조

실전에서 쓰는 30분 단위 세션 구조예요.

```
[0-5분] 컨텍스트 세팅
  - 오늘 작업 목표 공유
  - 관련 파일/코드 AI에게 전달
  - 제약 조건 명시

[5-20분] 구현
  - Driver-Navigator 또는 Ping-Pong 모드로 진행
  - 단계별 구현 + 즉시 리뷰

[20-25분] 리뷰 & 테스트
  - 생성된 코드 전체 리뷰
  - 테스트 실행
  - 엣지 케이스 확인

[25-30분] 정리 & 커밋
  - 변경사항 스테이징
  - 커밋 메시지 작성
  - 다음 세션 메모
```
