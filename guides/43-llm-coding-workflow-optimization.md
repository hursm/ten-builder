# 가이드 43: LLM 코딩 워크플로우 최적화

> AI 코딩 도구를 매일 쓰면서 결과물 품질을 끌어올리는 5단계 워크플로우 — 스펙 먼저, 작게 나누고, 검증하고, 반복하는 실전 루프

## 왜 워크플로우가 중요한가

AI 코딩 도구가 일상이 된 2026년, 도구 자체보다 **사용 방식**이 생산성을 결정해요. 같은 Claude Code나 Cursor를 써도 워크플로우에 따라 결과물 품질 차이가 크게 나요.

핵심 원칙은 단순해요: **코드를 직접 짜는 시간을 줄이고, 방향을 잡는 시간을 늘리는 것**.

## Step 1: 스펙 먼저 작성하기

코드부터 생성하면 방향이 흔들려요. AI에게 작업을 맡기기 전에 짧은 스펙을 먼저 정리하세요.

### 스펙 템플릿

```markdown
## 목표
- 무엇을 만들/수정할 것인가

## 제약 조건
- 사용할 기술 스택
- 성능/보안 요구사항
- 기존 코드와의 호환성

## 완료 기준
- [ ] 어떤 상태가 되면 끝인가
- [ ] 테스트 통과 기준
```

| 스펙 요소 | 좋은 예 | 나쁜 예 |
|-----------|---------|---------|
| 목표 | "사용자 프로필 API에 이미지 업로드 엔드포인트 추가" | "API 개선" |
| 제약 | "S3 버킷 사용, 파일 크기 5MB 제한, WebP 변환" | "적당히 빠르게" |
| 완료 기준 | "업로드/조회 통합 테스트 통과, 에러 핸들링 포함" | "잘 되면 됨" |

> 스펙을 CLAUDE.md나 프로젝트 문서에 넣어두면 AI가 매 세션마다 참조할 수 있어요.

## Step 2: 작업을 작은 단위로 분해하기

한 번에 큰 기능을 맡기면 AI가 컨텍스트를 잃거나 방향이 틀어져요. **하나의 프롬프트에 하나의 작업**이 기본이에요.

### 분해 전략

```bash
# 나쁜 예: 한 번에 모든 것을 요청
"사용자 인증 시스템을 만들어줘"

# 좋은 예: 단계별로 분리
"1단계: User 모델과 마이그레이션 파일을 만들어줘"
"2단계: JWT 토큰 발급/검증 유틸리티를 만들어줘"  
"3단계: 로그인/회원가입 API 엔드포인트를 만들어줘"
"4단계: 인증 미들웨어를 만들어줘"
```

### 프롬프트 플랜 파일 활용

여러 단계를 미리 정리한 **프롬프트 플랜 파일**을 만들어두면 일관된 작업 흐름을 유지할 수 있어요.

```markdown
<!-- prompts/auth-feature.md -->
## Phase 1: 데이터 모델
- User 스키마 정의 (email, password_hash, created_at)
- 마이그레이션 파일 생성

## Phase 2: 유틸리티
- JWT 발급/검증 (access + refresh)
- 비밀번호 해싱 (bcrypt)

## Phase 3: API
- POST /auth/register
- POST /auth/login  
- POST /auth/refresh

## Phase 4: 미들웨어
- requireAuth 미들웨어
- 라우트별 적용
```

## Step 3: 풍부한 컨텍스트 제공하기

AI의 출력 품질은 입력 컨텍스트에 비례해요. 코드만 던지지 말고, **왜** 그렇게 해야 하는지 맥락을 함께 전달하세요.

### 효과적인 컨텍스트 구성

| 컨텍스트 유형 | 방법 | 효과 |
|-------------|------|------|
| 프로젝트 규칙 | CLAUDE.md, .cursorrules | 스타일/패턴 일관성 유지 |
| 참조 코드 | 기존 파일 경로 지정 | 코드 스타일 매칭 |
| 에러 로그 | 전체 스택 트레이스 포함 | 정확한 디버깅 |
| 아키텍처 문서 | 다이어그램 또는 설명 | 설계 의도 이해 |

```bash
# 컨텍스트를 명시적으로 전달하는 예
"src/auth/middleware.ts를 참조해서 동일한 패턴으로 
rate-limiting 미들웨어를 만들어줘.
기존 에러 핸들링 방식(AppError 클래스)을 따라야 해."
```

### 모델별 사용 시나리오

작업에 따라 적절한 모델을 선택하면 비용과 속도를 모두 잡을 수 있어요.

| 작업 유형 | 적합한 모델 | 이유 |
|-----------|-----------|------|
| 빠른 수정, 포맷팅 | Haiku / Flash | 응답 속도 빠르고 비용 저렴 |
| 로직 구현, 리팩토링 | Sonnet / GPT-4o | 코드 품질과 속도의 균형 |
| 아키텍처 설계, 복잡한 분석 | Opus / o3 | 추론 깊이가 필요한 작업 |
| 멀티 파일 대규모 변경 | Claude Code (에이전틱) | 파일 탐색과 수정을 자율 수행 |

## Step 4: 생성된 코드 검증하기

AI가 만든 코드를 그대로 커밋하면 안 돼요. **생성 → 검증 → 수정**의 루프를 반드시 거치세요.

### 검증 체크리스트

```bash
# 1. 타입 체크
npx tsc --noEmit

# 2. 린트
npx eslint src/ --fix

# 3. 테스트 실행
npm test -- --coverage

# 4. 빌드 확인
npm run build
```

### AI에게 자기 검증 요청하기

```
"방금 작성한 코드에서 다음을 확인해줘:
1. 에러 핸들링이 빠진 곳은 없는지
2. 엣지 케이스(빈 배열, null, 큰 입력)는 처리했는지
3. 기존 코드와 네이밍 컨벤션이 일치하는지"
```

| 검증 단계 | 자동화 가능 | 방법 |
|-----------|-----------|------|
| 타입 체크 | ✅ | CI/pre-commit 훅 |
| 유닛 테스트 | ✅ | AI가 테스트도 함께 생성 |
| 코드 리뷰 | 부분적 | AI 리뷰 + 사람 최종 확인 |
| 통합 테스트 | ✅ | 배포 파이프라인에 포함 |
| 보안 검토 | 부분적 | 정적 분석 도구 + AI 보조 |

## Step 5: 반복하고 커밋 자주 하기

작은 단위로 작업하고, 검증이 끝날 때마다 바로 커밋하세요. 되돌리기가 쉬워져요.

### 커밋 리듬

```bash
# 기능 하나 완성될 때마다 커밋
git add -A
git commit -m "feat(auth): add JWT token utility"

# 다음 단계로 넘어가기 전에 반드시 커밋
git add -A  
git commit -m "feat(auth): add login endpoint"
```

### 반복 개선 프롬프트 패턴

```
"현재 login 엔드포인트를 리뷰하고 개선해줘.
특히 다음 관점에서:
- 입력 검증이 빠진 필드
- 에러 메시지의 구체성
- 로깅 추가 필요한 곳"
```

## 실전 일일 워크플로우 요약

```
09:00  오늘 할 작업 스펙 정리 (10분)
09:10  스펙을 프롬프트 플랜으로 분해 (5분)
09:15  Phase 1 생성 → 검증 → 커밋
09:45  Phase 2 생성 → 검증 → 커밋
10:15  Phase 3 생성 → 검증 → 커밋
10:45  전체 통합 테스트 + 코드 리뷰
11:00  PR 생성, 다음 작업 스펙 시작
```

| 단계 | 시간 비중 | 주체 |
|------|---------|------|
| 스펙/계획 | 20% | 사람 |
| 코드 생성 | 15% | AI |
| 검증/수정 | 40% | 사람 + AI |
| 커밋/통합 | 15% | 사람 |
| 리뷰/개선 | 10% | AI + 사람 |

## 흔한 실수와 해결법

| 실수 | 증상 | 해결 |
|------|------|------|
| 스펙 없이 바로 코딩 요청 | AI가 방향을 잘못 잡음 | 3줄짜리 스펙이라도 먼저 작성 |
| 한 프롬프트에 너무 많은 요청 | 앞부분은 맞는데 뒷부분이 틀림 | 하나의 프롬프트에 하나의 작업 |
| 생성된 코드를 그대로 수락 | 미묘한 버그, 보안 취약점 | 반드시 검증 루프를 거침 |
| 커밋을 미루다가 큰 변경 | 되돌리기 어려움 | 검증 통과할 때마다 즉시 커밋 |
| 항상 같은 모델 사용 | 비용 낭비 또는 품질 부족 | 작업 복잡도에 따라 모델 전환 |
