# 가이드 22: 스펙 기반 AI 개발

> AI에게 코드를 시키기 전에 스펙을 먼저 쓰세요 — 결과물의 품질이 완전히 달라져요

## 소요 시간

20-30분

## 사전 준비

- AI 코딩 도구 사용 경험 (Claude Code, Cursor, Copilot 등)
- 소프트웨어 설계 기본 개념
- 프로젝트 요구사항 정리 경험

## 왜 스펙부터 써야 하나요?

AI 코딩 에이전트에게 "채팅 앱 만들어줘"라고 하면, 동작하는 코드는 나와요. 그런데 내가 원하는 채팅 앱이 맞는지는 다른 문제예요.

```
❌ 일반적인 접근
"채팅 앱 만들어줘" → AI가 알아서 결정 → 원하는 결과가 아님 → 수정 반복

✅ 스펙 기반 접근
스펙 작성 → AI가 스펙대로 구현 → 검증 기준으로 확인 → 정확한 결과
```

스펙 기반 개발(Spec-Driven Development)은 코드를 작성하기 전에 "무엇을 만들 것인지"를 문서로 먼저 정의하는 방법이에요. AI 시대에 이 접근이 더 중요해진 이유가 있어요:

| 이유 | 설명 |
|------|------|
| **방향 고정** | AI가 임의로 결정하는 부분을 줄여요 |
| **검증 가능** | "맞게 만들었는지" 판단할 기준이 생겨요 |
| **재현 가능** | 같은 스펙으로 다시 시켜도 비슷한 결과가 나와요 |
| **팀 공유** | 동료에게 "이거 리뷰해줘"가 가능해져요 |

## Step 1: 스펙 문서 구조 잡기

좋은 스펙은 AI가 모호함 없이 이해할 수 있는 구조를 가져요. 전체를 한 번에 쓸 필요 없이, 단계별로 채워가면 돼요.

### 기본 스펙 템플릿

```markdown
# Feature: {기능명}

## 목적
- 이 기능이 해결하는 문제
- 대상 사용자

## 동작 정의
### 정상 케이스
1. 사용자가 {행동}하면
2. 시스템이 {반응}한다
3. 결과로 {상태 변화}가 발생한다

### 엣지 케이스
- {조건} 일 때 → {기대 동작}
- {조건} 일 때 → {기대 동작}

## 기술 제약
- 사용 언어/프레임워크: {명시}
- 성능 요구: {명시}
- 호환성: {명시}

## 검증 기준
- [ ] {테스트 케이스 1}
- [ ] {테스트 케이스 2}
- [ ] {테스트 케이스 3}
```

### 실전 예시: 댓글 기능 스펙

```markdown
# Feature: 게시물 댓글 시스템

## 목적
- 사용자가 게시물에 댓글을 남기고 소통할 수 있게 한다
- 대상: 로그인한 사용자

## 동작 정의
### 정상 케이스
1. 로그인한 사용자가 댓글 입력란에 텍스트를 작성한다
2. "등록" 버튼을 클릭하면 댓글이 저장된다
3. 페이지 새로고침 없이 댓글 목록에 즉시 표시된다
4. 댓글 작성자에게 본인 댓글 삭제 버튼이 보인다

### 엣지 케이스
- 빈 문자열 전송 시 → "댓글을 입력해주세요" 에러 표시
- 5000자 초과 시 → "5000자 이내로 작성해주세요" 에러 표시
- 비로그인 사용자 → 입력란 비활성화 + "로그인 후 댓글을 남길 수 있어요" 안내

## 기술 제약
- React + TypeScript (프론트)
- Express.js + Prisma (백엔드)
- 실시간 업데이트: Optimistic UI (SWR mutate)

## 검증 기준
- [ ] 댓글 CRUD API 동작 (POST/GET/DELETE)
- [ ] 빈 문자열/초과 길이 밸리데이션
- [ ] 비로그인 시 입력 차단
- [ ] 본인 댓글만 삭제 가능
- [ ] Optimistic UI로 즉시 반영
```

## Step 2: 스펙을 AI에게 전달하는 방법

스펙을 쓰는 것만으로 끝이 아니에요. AI가 스펙을 제대로 참조하도록 전달하는 방식도 중요해요.

### 방법 1: CLAUDE.md에 스펙 참조 경로 등록

```markdown
# CLAUDE.md

## 현재 작업 스펙
- 댓글 기능: `docs/specs/comment-system.md`
- 알림 기능: `docs/specs/notification.md`

## 구현 규칙
- 스펙에 정의된 동작만 구현할 것
- 스펙에 없는 기능은 추가하지 말 것
- 검증 기준의 모든 항목이 통과해야 완료
```

### 방법 2: 프롬프트에 스펙 직접 포함

```
아래 스펙대로 댓글 API를 구현해줘.
스펙에 정의되지 않은 동작은 추가하지 마.

[스펙 내용 붙여넣기]
```

### 방법 3: speckit 같은 도구 사용

```bash
# GitHub의 speckit 활용
npx speckit init
npx speckit add comment-system
npx speckit generate  # 스펙 → 이슈 → 코드 흐름 자동화
```

| 방법 | 장점 | 단점 |
|------|------|------|
| CLAUDE.md 참조 | 프로젝트 전체에 일관 적용 | 스펙이 여러 파일로 분산될 수 있음 |
| 프롬프트 직접 포함 | 즉시 적용, 맥락 명확 | 매번 붙여넣기 필요 |
| speckit/도구 활용 | 워크플로우 자동화 | 초기 설정 필요 |

## Step 3: 스펙을 점진적으로 구체화하기

처음부터 완벽한 스펙을 쓸 필요 없어요. AI와 대화하면서 스펙을 다듬어가는 것도 좋은 방법이에요.

### 3단계 구체화 패턴

```
Level 1: 한 줄 요약
"사용자가 게시물에 댓글을 남기는 기능"

Level 2: 동작 정의
"로그인 사용자만 댓글 작성 가능, 최대 5000자, 
본인 댓글만 삭제, 실시간 UI 업데이트"

Level 3: 전체 스펙
기술 스택, 엣지 케이스, API 엔드포인트,
테이블 스키마, 테스트 케이스까지 포함
```

AI에게 Level 1만 주고 "이 기능의 스펙 초안을 작성해줘"라고 하면, AI가 빠뜨린 부분을 확인하고 보완할 수 있어요. 이 방식이 특히 유용한 경우:

- 도메인을 잘 모르는 새 프로젝트
- 요구사항이 아직 확정되지 않은 초기 단계
- 기존 시스템에 기능을 추가할 때

## Step 4: 검증 기준을 먼저 쓰기

스펙 기반 개발에서 가장 중요한 부분이에요. 검증 기준이 곧 "완료 조건"이에요.

### 검증 기준 작성 패턴

```markdown
## 검증 기준

### 기능 검증
- [ ] 정상 입력으로 댓글 생성 → 201 응답 + DB 저장 확인
- [ ] GET /comments?postId=1 → 해당 게시물 댓글 목록 반환
- [ ] DELETE /comments/:id → 본인 댓글만 삭제 가능 (403 검증)

### 엣지 케이스 검증
- [ ] 빈 body 전송 → 400 + 적절한 에러 메시지
- [ ] 5001자 입력 → 400 + 길이 제한 에러
- [ ] 존재하지 않는 postId → 404

### 비기능 검증
- [ ] 댓글 조회 API 응답 < 200ms (100개 기준)
- [ ] SQL 인젝션 방어 확인
- [ ] XSS 필터링 확인
```

이 검증 기준을 AI에게 같이 전달하면, AI가 스스로 테스트 코드를 작성하고 검증까지 해줄 수 있어요.

```
아래 스펙의 검증 기준을 모두 통과하는 코드를 작성해줘.
각 검증 기준에 대응하는 테스트 코드도 함께 작성해.

[스펙 + 검증 기준]
```

## Step 5: 스펙 관리 워크플로우

프로젝트가 커지면 스펙도 체계적으로 관리해야 해요.

### 디렉토리 구조

```
project/
├── docs/
│   └── specs/
│       ├── _template.md          # 스펙 템플릿
│       ├── auth-system.md        # 인증 스펙 (구현 완료)
│       ├── comment-system.md     # 댓글 스펙 (구현 중)
│       └── notification.md       # 알림 스펙 (계획 단계)
├── CLAUDE.md                     # 현재 작업 스펙 참조
└── src/
```

### 스펙 상태 관리

| 상태 | 의미 | CLAUDE.md 포함 여부 |
|------|------|---------------------|
| `draft` | 초안, 아직 확정 안 됨 | ❌ |
| `ready` | 확정, 구현 가능 | ✅ |
| `in-progress` | 구현 진행 중 | ✅ |
| `done` | 구현 + 검증 완료 | ❌ (아카이브) |

### 스펙 변경 추적

```bash
# 스펙 파일도 git으로 버전 관리
git log --oneline docs/specs/comment-system.md
# a1b2c3d 스펙: 댓글 길이 제한 5000자로 변경
# d4e5f6g 스펙: 초안 작성
```

## 흔한 실수와 해결

| 실수 | 해결 |
|------|------|
| 스펙 없이 바로 코딩 시작 | 5분이라도 동작 정의를 먼저 적기 |
| 스펙이 너무 추상적 | 구체적인 입출력 예시 추가 |
| 엣지 케이스 누락 | "이걸 잘못 쓰면 어떻게 되지?" 질문하기 |
| 검증 기준 없음 | "어떻게 완료를 판단할까?" 먼저 답하기 |
| 스펙과 코드가 따로 놀기 | 스펙 변경 시 관련 코드도 함께 업데이트 |
| 한 번에 완벽하게 쓰려 함 | Level 1 → 2 → 3 점진 구체화 |

## 도구 활용 팁

### GitHub speckit

```bash
# 설치
npm install -g @anthropic/speckit

# 프로젝트 초기화
speckit init

# 스펙 작성
speckit add user-authentication

# 스펙 → GitHub Issue 변환
speckit plan

# 스펙 기반 코드 생성
speckit implement
```

### Kiro IDE의 스펙 기반 접근

Kiro는 스펙 파일을 자동으로 감지해서, 코딩 전에 "이 스펙이 맞는지" 확인하는 단계를 거쳐요. 수동으로 하던 스펙 전달 과정이 IDE 레벨에서 자동화되는 거예요.

### CI에 스펙 검증 통합

```yaml
# .github/workflows/spec-check.yml
name: Spec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check spec coverage
        run: |
          # 변경된 코드에 대응하는 스펙이 있는지 확인
          changed_files=$(git diff --name-only origin/main)
          for file in $changed_files; do
            dir=$(dirname "$file")
            if [ ! -f "docs/specs/${dir##*/}.md" ]; then
              echo "Warning: No spec found for $file"
            fi
          done
```

## 실전 체크리스트

새 기능을 구현할 때 이 순서로 진행하세요:

- [ ] 한 줄 요약 작성 ("이 기능은 ___를 한다")
- [ ] 정상 동작 시나리오 3개 이상 정의
- [ ] 엣지 케이스 2개 이상 정의
- [ ] 기술 제약 명시 (언어, 프레임워크, 성능)
- [ ] 검증 기준 작성 (기능 + 엣지 + 비기능)
- [ ] CLAUDE.md에 스펙 경로 등록
- [ ] AI에게 스펙 전달 + 구현 요청
- [ ] 검증 기준 전체 통과 확인
- [ ] 스펙 상태를 `done`으로 변경
