# AI 코딩 에이전트 프롬프트 체이닝 고급 패턴

> 복잡한 개발 태스크를 단계별 프롬프트 체인으로 분해하여 AI 에이전트의 정확도와 일관성을 높이는 실전 가이드

## 프롬프트 체이닝이 필요한 이유

"이 프로젝트 전체를 리팩터링해줘"라고 한 번에 던지면 AI 에이전트가 중간에 방향을 잃거나, 앞뒤가 안 맞는 코드를 만들어요. 컨텍스트 윈도우가 아무리 커져도 **한 번에 모든 것을 처리하는 것**과 **단계별로 나눠서 처리하는 것**은 결과 품질이 완전히 달라요.

프롬프트 체이닝은 복잡한 태스크를 작은 단위로 쪼개고, 각 단계의 출력을 다음 단계의 입력으로 연결하는 패턴이에요. 사람이 큰 프로젝트를 할 때 TODO 리스트를 만들고 하나씩 체크하는 것과 같은 원리예요.

### 단일 프롬프트 vs 체이닝 비교

| | 단일 프롬프트 | 프롬프트 체이닝 |
|---|---|---|
| 적합한 태스크 | 함수 1개 수정, 간단한 버그 픽스 | 모듈 리팩터링, 피처 구현, 마이그레이션 |
| 에러 복구 | 전체 재시도 | 실패한 단계만 재시도 |
| 디버깅 | 어디서 틀렸는지 파악 어려움 | 단계별 출력 확인 가능 |
| 일관성 | 뒷부분으로 갈수록 품질 저하 | 각 단계가 집중된 컨텍스트에서 동작 |
| 비용 효율 | 한 번에 많은 토큰 소비 | 필요한 컨텍스트만 각 단계에 전달 |

## 패턴 1: Sequential Chain (순차 체인)

가장 기본적인 패턴이에요. 각 단계가 순서대로 실행되고, 이전 단계의 결과가 다음 단계의 입력이 돼요.

### 적용 시나리오

- API 엔드포인트 추가 (스키마 → 타입 → 핸들러 → 테스트)
- 데이터베이스 마이그레이션 (스키마 분석 → 마이그레이션 파일 → 모델 업데이트 → 테스트)

### 실전 예제: REST API 엔드포인트 추가

```bash
# Step 1: 스키마 정의
"users 테이블에 subscription_tier 컬럼을 추가하는 마이그레이션 파일을 생성해줘.
타입: enum('free', 'pro', 'enterprise'), 기본값: 'free', NOT NULL"

# Step 2: 타입/인터페이스 업데이트 (Step 1 결과 참조)
"방금 만든 마이그레이션에 맞춰서 User 타입과 관련 DTO를 업데이트해줘.
CreateUserDto, UpdateUserDto, UserResponseDto 모두 반영"

# Step 3: 비즈니스 로직 구현 (Step 2 결과 참조)
"업데이트된 타입을 기반으로 subscription-tier 변경 API 핸들러를 구현해줘.
- PUT /users/:id/subscription
- 권한 체크: admin만 변경 가능
- 변경 이력 로깅 포함"

# Step 4: 테스트 작성 (Step 3 결과 참조)
"방금 만든 subscription 핸들러에 대한 통합 테스트를 작성해줘.
정상 케이스 3개, 에러 케이스 3개 (권한 없음, 잘못된 tier, 존재하지 않는 유저)"
```

### 체인 설계 팁

```
[스키마] → [타입] → [로직] → [테스트]
   ↓          ↓         ↓         ↓
 .sql       .ts      .ts       .test.ts
```

각 단계에서 **출력 파일을 명시**하면 다음 단계에서 자동으로 참조할 수 있어요. "방금 만든 파일"이라는 모호한 참조 대신 `src/types/user.dto.ts`처럼 경로를 지정하세요.

## 패턴 2: Verification Chain (검증 체인)

생성 → 검증 → 수정을 반복하는 패턴이에요. AI가 만든 코드를 AI 스스로 검증하게 하면 품질이 올라가요.

### 적용 시나리오

- 보안이 중요한 코드 (인증, 결제, 권한)
- 레거시 코드 리팩터링 (기존 동작 보존 필수)
- 타입 안전성이 중요한 변환 작업

### 실전 예제: 보안 코드 생성 + 검증

```bash
# Step 1: 구현
"JWT 토큰 검증 미들웨어를 구현해줘.
- RS256 알고리즘
- 토큰 만료 체크
- refresh token rotation"

# Step 2: 보안 검증 (같은 에이전트 또는 다른 에이전트)
"방금 만든 JWT 미들웨어를 보안 관점에서 검토해줘.
체크리스트:
- [ ] 타이밍 공격 방어 (constant-time comparison)
- [ ] 토큰 재사용 공격 방어
- [ ] 에러 메시지에 민감 정보 노출 여부
- [ ] rate limiting 적용 여부
발견된 문제를 수정 코드와 함께 알려줘"

# Step 3: 수정 적용
"보안 리뷰 결과를 반영해서 미들웨어를 수정해줘.
변경사항을 주석으로 표시해줘 // SECURITY FIX: ..."
```

### 검증 체인 자동화

```yaml
# .claude/commands/secure-implement.md
1. 구현: $TASK
2. 보안 검증: OWASP Top 10 기준 체크
3. 수정: 발견된 취약점 즉시 패치
4. 테스트: 취약점별 방어 테스트 추가
5. 최종 확인: 전체 보안 체크리스트 통과 여부
```

## 패턴 3: Fan-out / Fan-in (병렬 분산 체인)

독립적인 서브태스크를 병렬로 처리하고, 결과를 모아서 통합하는 패턴이에요. 서로 의존성이 없는 작업들을 동시에 진행할 수 있어요.

### 적용 시나리오

- 여러 마이크로서비스에 동일 변경 적용
- 프론트엔드 + 백엔드 + 인프라 동시 작업
- 다국어 지원 추가 (각 언어 파일 독립 생성)

### 실전 예제: 풀스택 피처 구현

```bash
# Fan-out: 3개 태스크 병렬 실행

# Agent A (백엔드)
"POST /api/notifications 엔드포인트를 구현해줘.
- 스키마, 핸들러, 서비스 레이어 모두 포함
- WebSocket 브로드캐스트 로직 포함"

# Agent B (프론트엔드)
"NotificationBell 컴포넌트를 만들어줘.
- WebSocket 연결로 실시간 알림 수신
- 읽지 않은 알림 개수 뱃지
- 드롭다운 알림 리스트"

# Agent C (인프라)
"Redis pub/sub 설정을 추가해줘.
- docker-compose.yml에 Redis 서비스 추가
- 환경변수 설정
- health check 엔드포인트"

# Fan-in: 결과 통합
"Agent A, B, C의 결과를 통합해줘.
- import 경로 연결
- 환경변수 일관성 확인
- 통합 테스트 작성"
```

### 병렬 실행 팁

| 조건 | 병렬 가능 | 순차 필요 |
|------|----------|----------|
| 같은 파일 수정 | ❌ | ✅ |
| 다른 디렉토리 작업 | ✅ | ❌ |
| 공유 타입/인터페이스 | ❌ | ✅ (타입 먼저) |
| API 스펙 기반 작업 | ✅ (스펙 공유) | ❌ |

## 패턴 4: Conditional Chain (조건 분기 체인)

이전 단계의 결과에 따라 다음 단계가 달라지는 패턴이에요. 에러 핸들링이나 분석 결과에 따른 분기 처리에 유용해요.

### 실전 예제: 코드 분석 → 조건별 대응

```bash
# Step 1: 분석
"이 모듈의 복잡도를 분석해줘.
출력 형식:
- cyclomatic_complexity: (숫자)
- lines_of_code: (숫자)
- dependency_count: (숫자)
- recommendation: 'refactor' | 'split' | 'keep'"

# Step 2: 분기
# recommendation = 'refactor' → 리팩터링 체인
# recommendation = 'split' → 모듈 분리 체인
# recommendation = 'keep' → 테스트 보강만

# Step 2a: refactor 경로
"복잡도가 높은 함수를 추출 리팩터링해줘.
단일 책임 원칙 적용, 함수당 20줄 이하"

# Step 2b: split 경로
"이 모듈을 3개 하위 모듈로 분리해줘.
- core: 핵심 비즈니스 로직
- utils: 헬퍼 함수
- types: 타입 정의
각 모듈의 public API를 index.ts로 내보내기"
```

### 조건 분기 자동화 패턴

```bash
# CLAUDE.md 또는 커스텀 커맨드에 분기 로직 정의
분석 결과에 따라:
- 복잡도 > 15 → 리팩터링 우선
- 테스트 커버리지 < 60% → 테스트 추가 우선
- 둘 다 해당 → 테스트 먼저, 리팩터링 다음
```

## 패턴 5: Iterative Refinement Chain (반복 정제 체인)

동일한 출력을 여러 번 정제하는 패턴이에요. 초안 → 리뷰 → 개선을 반복하면서 품질을 올려요.

### 적용 시나리오

- 복잡한 알고리즘 구현 (정확성 → 성능 → 가독성)
- API 설계 (기능 → 에러 핸들링 → 문서화)
- 프롬프트 자체를 개선할 때

### 실전 예제: 알고리즘 3단 정제

```bash
# Round 1: 정확성 우선
"주어진 DAG에서 최장 경로를 찾는 함수를 구현해줘.
정확하게 동작하는 것이 최우선. 성능은 나중에 개선"

# Round 2: 성능 개선
"방금 구현한 최장 경로 알고리즘을 최적화해줘.
- 메모이제이션 적용
- 불필요한 재계산 제거
- 시간/공간 복잡도를 주석으로 명시"

# Round 3: 가독성 + 엣지 케이스
"최적화된 코드에 다음을 추가해줘:
- 빈 그래프, 단일 노드, 순환 감지 등 엣지 케이스 처리
- JSDoc 주석
- 각 주요 로직 블록에 한 줄 설명"
```

## 실전 조합: CLAUDE.md에 체인 패턴 정의하기

프로젝트의 CLAUDE.md 파일에 자주 쓰는 체인 패턴을 정의해두면 반복 작업이 일관되게 동작해요.

```markdown
# CLAUDE.md

## 커스텀 워크플로우

### 새 API 엔드포인트 추가 체인
1. OpenAPI 스펙에 엔드포인트 정의 추가
2. 스펙에서 타입 자동 생성 (`pnpm codegen`)
3. 서비스 레이어 구현
4. 컨트롤러/핸들러 구현
5. 통합 테스트 작성
6. API 문서 업데이트

### 버그 픽스 체인
1. 버그 재현 테스트 작성 (실패하는 테스트)
2. 원인 분석 + 최소 범위 수정
3. 테스트 통과 확인
4. 관련 엣지 케이스 테스트 추가
5. 리그레션 테스트 추가
```

## 체이닝 안티패턴

피해야 할 패턴도 알아두세요.

| 안티패턴 | 문제 | 해결 |
|----------|------|------|
| **과도한 분할** | 단순 작업을 10단계로 쪼개면 오버헤드만 증가 | 2~5단계가 적정. 단순 작업은 단일 프롬프트로 |
| **컨텍스트 누락** | "이전 단계에서 만든 파일" 같은 모호한 참조 | 파일 경로를 명시적으로 전달 |
| **검증 없는 연결** | 이전 단계 실패를 무시하고 다음 진행 | 각 단계 후 출력 확인 단계 추가 |
| **순환 체인** | 수정 → 검증 → 수정이 무한 반복 | 최대 반복 횟수 설정 (보통 3회) |
| **전역 상태 의존** | 단계 간 암묵적 상태 공유 | 각 단계에 필요한 정보를 명시적으로 전달 |

## 체인 길이 가이드라인

| 태스크 규모 | 권장 체인 길이 | 예시 |
|------------|--------------|------|
| 소 | 1-2단계 | 함수 수정, 타입 추가 |
| 중 | 3-5단계 | API 엔드포인트, 컴포넌트 구현 |
| 대 | 5-8단계 | 피처 전체 구현, 리팩터링 |
| 특대 | Fan-out 병렬 | 마이크로서비스 변경, 대규모 마이그레이션 |
