# 가이드 61: AI 에이전트 결과물 검수 패턴

> AI가 만든 코드, 그냥 믿으면 안 됩니다 — 체계적으로 검수하는 4가지 패턴

---

## 왜 검수가 필요한가

AI 에이전트는 빠르게 코드를 만들지만, 두 가지 함정이 있어요:

1. **과잉 생성**: 요청한 것보다 더 많이 바꿔놓음 (파일 10개짜리 수정 요청인데 20개 건드림)
2. **조용한 오류**: 컴파일은 되지만 의도와 다른 로직

2026년 Anthropic 조사에 따르면, AI 에이전트가 만든 코드의 **23%는 조용한 버그**를 포함하고, 개발자가 직접 검토 없이 머지한 경우 프로덕션 장애 비율이 3배 높았어요. 속도를 위해 검수를 건너뛰는 건 오히려 느려지는 지름길입니다.

---

## 패턴 1: Diff 경계선 검수

> "AI가 건드리지 말아야 할 파일을 건드렸는가?"

### 언제 씁니까

- 에이전트가 여러 파일을 동시에 수정한 경우
- 중요 설정 파일(환경 변수, CI 설정)이 포함된 경우

### 방법

```bash
# 변경된 파일 목록만 확인
git diff --name-only HEAD

# 특정 디렉토리 밖의 변경은 즉시 의심
git diff --name-only HEAD | grep -v "^src/" | grep -v "^tests/"
```

```bash
# 변경 규모 확인 (라인 수)
git diff --stat HEAD

# 500줄 이상이면 의도치 않은 대규모 수정 가능성
git diff HEAD | wc -l
```

**체크 기준:**

| 변경 규모 | 대응 |
|----------|------|
| 1~50줄 | 빠른 눈으로 확인 후 머지 |
| 51~200줄 | 파일별 로직 확인 필수 |
| 200줄+ | 에이전트에게 요약 요청 후 재검토 |
| 설정 파일 포함 | 무조건 직접 열어서 확인 |

## 패턴 2: 의도 대조 검수

> "내가 요청한 것과 에이전트가 한 것이 일치하는가?"

### 방법

에이전트에게 직접 물어봐요:

```
"방금 어떤 변경을 했는지 3줄로 요약해줘.
각 파일에서 무엇을 변경했고, 왜 그렇게 했는지 알려줘."
```

그다음, 요약과 실제 diff를 대조합니다.

```bash
# 주요 함수/클래스 이름 변경 감지
git diff HEAD | grep "^-\|^+" | grep -E "def |class |function |const " | head -20
```

### 불일치 신호

- 요약에 없는 파일이 diff에 있음
- "기존 동작은 유지했다"는데 테스트가 깨짐
- 리팩토링을 요청하지 않았는데 파일 구조가 바뀜

---

## 패턴 3: Human-in-the-Loop 게이트

> "되돌릴 수 없는 작업 전에 반드시 승인 단계를 둔다"

### Claude Code hooks 활용

`~/.claude/settings.json`에 위험 작업 차단 규칙 추가:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "bash -c 'read -r cmd <<< \"$CLAUDE_TOOL_INPUT\"; if echo \"$cmd\" | grep -qE \"rm -rf|DROP TABLE|DELETE FROM|git push.*--force\"; then echo \"BLOCKED: 위험 명령어 감지\" >&2; exit 1; fi'"
          }
        ]
      }
    ]
  }
}
```

### CLAUDE.md에 검수 규칙 명시

```markdown
## 검수 규칙

- 파일 삭제 전 반드시 확인 요청
- 환경 변수 파일(.env, .env.local) 수정 금지
- 데이터베이스 마이그레이션 파일 생성 시 내용 보고 후 실행
- git push는 직접 실행 (에이전트 실행 금지)
```

## 패턴 4: 증분 검수 루프

> "한 번에 큰 변경이 아니라, 작은 단계로 나눠서 검수"

### 방법

큰 기능 추가 요청을 단계로 쪼갭니다:

```
❌ 안 좋은 방법:
"로그인, 회원가입, 비밀번호 재설정 기능 전부 만들어줘"

✅ 좋은 방법:
1. "로그인 API 엔드포인트만 만들어줘. 테스트 포함."
   → 검수 → 승인
2. "방금 만든 로그인 기반으로 회원가입 추가해줘."
   → 검수 → 승인
3. ...
```

### 체크포인트 커밋 활용

```bash
# 각 단계 완료 시 커밋으로 체크포인트 생성
git add -p  # 변경사항 선택적으로 스테이징
git commit -m "step 1: 로그인 API"

# 잘못됐으면 쉽게 되돌리기 가능
git reset --hard HEAD~1
```

## 검수 체크리스트

```markdown
## AI 에이전트 결과물 검수 체크리스트

### 필수 (머지 전)
- [ ] 변경된 파일 목록 확인 — 의도치 않은 파일 없음
- [ ] 변경 규모 확인 — 요청 범위에 맞음
- [ ] 테스트 통과 — 기존 테스트 깨지지 않음
- [ ] 중요 파일 미포함 — .env, 설정 파일 미변경

### 권장 (중요 기능일 때)
- [ ] 에이전트에게 변경 요약 요청 → diff와 대조
- [ ] 핵심 비즈니스 로직 직접 확인
- [ ] 엣지 케이스 수동 테스트 1회
```

## 팀에 적용하기

| 상황 | 추천 패턴 |
|------|----------|
| 혼자 빠르게 프로토타이핑 | 패턴 1 (Diff 경계선) |
| 팀 코드베이스에 기능 추가 | 패턴 2 + 3 조합 |
| 프로덕션 핵심 로직 변경 | 패턴 3 + 4 (게이트 + 증분) |
| 레거시 코드 리팩토링 | 패턴 4 (증분 루프) |
