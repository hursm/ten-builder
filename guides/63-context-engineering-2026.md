# 가이드 63: 컨텍스트 엔지니어링 — AI 에이전트 성능을 결정하는 진짜 기술

> 프롬프트 엔지니어링의 시대는 끝났다. 2026년 AI 에이전트 성능의 90%는 모델이 어떤 정보를 얼마나 잘 받았느냐로 결정됩니다.

## 왜 프롬프트보다 컨텍스트인가

Andrej Karpathy는 이렇게 정리했습니다: LLM은 CPU, 컨텍스트 윈도우는 RAM, 개발자는 운영체제입니다.

프롬프트를 아무리 정교하게 다듬어도, 모델이 접근하는 정보가 부실하면 출력이 달라집니다. 반대로 컨텍스트가 잘 설계되어 있으면, 평범한 프롬프트로도 좋은 결과가 나옵니다.

실제로 출력 품질에 영향을 주는 요소를 분류하면:

| 요소 | 영향 비중 |
|------|---------|
| 컨텍스트 설계 (어떤 정보를 넣을지) | ~60% |
| 컨텍스트 구조 (어떻게 정리할지) | ~20% |
| 프롬프트 지시문 | ~10% |
| 모델 선택 | ~10% |

프롬프트 엔지니어링은 지시문을 다듬는 작업입니다. 컨텍스트 엔지니어링은 모델이 접근하는 정보 전체를 설계하는 작업입니다.

## 컨텍스트 엔지니어링의 5가지 레이어

컨텍스트는 단순히 대화 내용만이 아닙니다. 여러 레이어로 구성됩니다.

### 레이어 1: 시스템 컨텍스트 (CLAUDE.md / AGENTS.md)

에이전트가 모든 세션에서 공통으로 참조하는 기반 정보입니다.

**포함해야 할 것:**
- 프로젝트 목적 한 문장 요약
- 기술 스택 목록 (언어, 프레임워크, 주요 라이브러리)
- 코드 스타일 원칙 (3~5개)
- 자주 실수하는 패턴 주의사항
- 디렉토리 구조 설명

**포함하지 말아야 할 것:**
- 완료된 태스크 기록
- 임시 결정 사항
- 500줄 초과 내용

```markdown
# CLAUDE.md 최적화 예시

## 프로젝트
Next.js 14 기반 SaaS 대시보드 (App Router, TypeScript strict)

## 스택
- Frontend: Next.js 14, Tailwind CSS, shadcn/ui
- Backend: Supabase (PostgreSQL + Auth + Storage)
- 배포: Vercel

## 코드 규칙
- 컴포넌트는 'use client' 최소화 (기본 Server Component)
- 에러는 typed error만 사용 (Error 클래스 상속)
- API 응답은 Zod로 검증 필수

## 금지 패턴
- any 타입 사용 금지
- 직접 DOM 조작 금지
- 민감 정보 하드코딩 금지
```

### 레이어 2: 태스크 컨텍스트 (세션 시작 시 주입)

현재 세션에서만 필요한 정보입니다. 태스크마다 다릅니다.

```
# 태스크 컨텍스트 주입 패턴
"지금 인증 기능을 리팩토링한다.
- 현재 파일: src/lib/auth.ts (레거시 next-auth v4)
- 목표: next-auth v5로 마이그레이션
- 변경 범위: auth 관련 파일만, DB 스키마는 건드리지 않음
- 테스트: src/tests/auth/ 폴더 기준"
```

범위를 명확히 하면 에이전트가 필요 없는 파일을 탐색하는 시간을 줄입니다.

### 레이어 3: 도구 결과 컨텍스트 (Tool Call Results)

에이전트가 파일을 읽거나 명령을 실행한 결과입니다.

컨텍스트 길이가 늘어날수록 오래된 도구 결과는 노이즈가 됩니다. Claude는 이를 위해 **도구 결과 압축(tool result clearing)** 기능을 지원합니다.

```
/clear tools  # 오래된 도구 결과만 삭제 (대화 내용 유지)
```

Anthropic의 연구에 따르면 도구 결과 압축만으로도 컨텍스트 사용량을 30~40% 줄일 수 있습니다.

### 레이어 4: 메모리 컨텍스트 (에이전트 노트)

긴 세션에서 에이전트가 발견한 패턴이나 결정을 저장하는 파일입니다.

```bash
# .claude/session-notes.md 예시
## 2026-04-18 인증 리팩토링 세션

### 발견한 것
- src/lib/auth.ts:47 — 토큰 갱신 로직에 race condition 있음
- middleware.ts — 보호된 라우트 목록이 하드코딩됨

### 결정
- 토큰 갱신은 refresh-mutex 패턴으로 처리
- 보호 라우트는 config 파일로 분리
```

세션이 길어지면 에이전트에게 주기적으로 노트를 업데이트하도록 지시합니다:

```
"지금까지 발견한 것과 결정한 내용을 .claude/session-notes.md에 정리해줘"
```

### 레이어 5: 외부 컨텍스트 (RAG / 문서 검색)

레포 범위 밖의 정보를 실시간으로 주입합니다. MCP 서버나 웹 검색 도구를 활용합니다.

```
# MCP를 통한 외부 문서 주입 예시
"next-auth v5의 공식 마이그레이션 가이드를 참고해서 작업해줘"
```

## 컨텍스트 설계 실전 패턴 4가지

### 패턴 1: 선택적 로딩 (Selective Loading)

전체 레포를 에이전트에 노출하지 말고, 관련 파일만 먼저 보여줍니다.

```
# BAD: 에이전트가 알아서 탐색하게 두기
"인증 버그 고쳐줘"

# GOOD: 관련 파일을 미리 지정
"다음 파일들의 인증 버그를 고쳐줘:
- src/lib/auth.ts
- src/middleware.ts
- src/app/api/auth/[...nextauth]/route.ts"
```

| 방식 | 탐색 시간 | 토큰 사용 | 정확도 |
|------|---------|---------|------|
| 자유 탐색 | 길다 | 많다 | 낮다 |
| 선택적 로딩 | 짧다 | 적다 | 높다 |

### 패턴 2: 컨텍스트 압축 (Context Compression)

긴 코드 파일을 요약해서 넘기면 토큰을 아끼면서 핵심 정보를 전달할 수 있습니다.

```
# 파일 전체 대신 핵심만 요약해서 주입
"auth.ts의 구조는 이렇다:
- authenticateUser(email, password): 사용자 인증, JWT 반환
- refreshToken(token): 토큰 갱신, 만료 30분 전 자동 실행
- invalidateSession(userId): 강제 로그아웃

버그: refreshToken에서 만료 체크 로직이 UTC 기준이 아니라 로컬 타임 기준임"
```

### 패턴 3: 점진적 공개 (Progressive Disclosure)

태스크 복잡도에 따라 컨텍스트를 단계별로 추가합니다.

```
# 1단계: 최소 컨텍스트로 시작
"auth.ts 파일을 분석해줘"

# 2단계: 에이전트 분석 결과 보고 추가 컨텍스트 주입
"방금 분석한 내용을 바탕으로, 이 파일과 연결된 middleware.ts도 확인해줘"

# 3단계: 필요한 경우에만 추가
"테스트 파일까지 확인이 필요하면 tests/auth/ 폴더를 봐도 돼"
```

### 패턴 4: 컨텍스트 격리 (Context Isolation)

서로 다른 태스크는 컨텍스트를 공유하지 않습니다. 서브에이전트나 `/clear`를 활용합니다.

```
# Claude Code에서 태스크 격리
/clear  # 이전 태스크 컨텍스트 전부 제거

# 서브에이전트로 격리
"백그라운드 서브에이전트를 하나 만들어서, 인증과 독립적으로
결제 모듈을 리뷰하게 해줘"
```

## CLAUDE.md 최적화 체크리스트

파일이 비대해지면 오히려 성능이 떨어집니다. 주기적으로 점검합니다.

```
□ 500줄 이하인가?
□ 완료된 태스크 기록이 남아있지 않은가?
□ 항상 적용되는 규칙만 있는가?
□ 헤딩과 목록 위주로 구성되어 있는가?
□ 코드 예시는 최소한인가? (핵심만 남기기)
□ 6개월 이상 지난 내용은 없는가?
```

### 컨텍스트 압박 신호

에이전트가 이런 행동을 보이면 컨텍스트가 포화 상태입니다:

| 신호 | 의미 |
|------|------|
| 이전에 설명한 내용을 다시 묻는다 | 초기 컨텍스트가 밀려남 |
| 같은 코드를 반복 생성한다 | 자신의 이전 출력을 참조 못함 |
| 파일 경로를 혼동한다 | 파일 탐색 결과가 뒤섞임 |
| 응답이 갑자기 짧아진다 | 내부 압축 시작 |

## 팀 단위 컨텍스트 엔지니어링

개인이 아니라 팀 전체의 AI 성능을 높이려면, 컨텍스트 설계를 팀의 자산으로 관리합니다.

```
project-root/
  CLAUDE.md              # 팀 공통 규칙 (git 관리)
  .claude/
    guides/
      auth-patterns.md   # 인증 관련 패턴 모음
      db-patterns.md     # DB 쿼리 최적화 패턴
    prompts/
      code-review.md     # 코드 리뷰 시작 프롬프트
      onboarding.md      # 신규 개발자 온보딩 프롬프트
```

시니어가 발견한 최적 패턴을 `CLAUDE.md`에 추가하면 팀 전체의 AI 성능이 올라갑니다.
