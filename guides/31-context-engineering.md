# 컨텍스트 엔지니어링 실전 가이드

> 프롬프트를 잘 쓰는 것보다, 맥락을 잘 설계하는 게 더 중요해졌어요

## 프롬프트 엔지니어링 vs 컨텍스트 엔지니어링

프롬프트 엔지니어링은 "어떻게 물어볼까"에 집중해요. 컨텍스트 엔지니어링은 "무엇을 보여줄까"에 집중합니다.

| 구분 | 프롬프트 엔지니어링 | 컨텍스트 엔지니어링 |
|------|---------------------|---------------------|
| 초점 | 질문 방식 최적화 | 제공 정보 최적화 |
| 핵심 | 지시문 구조, 톤, 포맷 | 데이터, 도구, 메모리, 구조 |
| 한계 | 모델이 모르는 건 못 답함 | 필요한 정보를 적시에 주입 |
| 스케일 | 프롬프트마다 수동 조정 | 시스템 레벨 자동화 가능 |

2026년 현재, AI 코딩 에이전트의 성능은 모델 자체보다 **어떤 컨텍스트를 주느냐**에 더 크게 좌우돼요.

## 기법 1: 계층적 지시 파일

CLAUDE.md(또는 .cursorrules)를 프로젝트 구조에 맞게 계층화하면, 에이전트가 현재 작업 위치에 맞는 규칙만 로드해요.

```
my-project/
├── CLAUDE.md              # 전역 규칙 (코딩 스타일, 커밋 규칙)
├── frontend/
│   └── CLAUDE.md          # 프론트엔드 전용 (React 패턴, 컴포넌트 규칙)
├── backend/
│   └── CLAUDE.md          # 백엔드 전용 (API 설계, DB 규칙)
└── infra/
    └── CLAUDE.md          # 인프라 전용 (Terraform, Docker 규칙)
```

**핵심 원칙:**

- 루트 CLAUDE.md: 프로젝트 전체에 적용되는 규칙만 (50줄 이내 권장)
- 하위 CLAUDE.md: 해당 디렉토리 작업 시에만 적용되는 구체적 규칙
- 충돌 시: 하위 디렉토리 규칙이 우선

```markdown
# frontend/CLAUDE.md
- React 18+ 함수형 컴포넌트만 사용
- CSS-in-JS 대신 Tailwind CSS 사용
- 컴포넌트 파일명: PascalCase
- 테스트: React Testing Library + Vitest
- 상태 관리: Zustand (Redux 사용 금지)
```

## 기법 2: 선택적 컨텍스트 제외

`.claudeignore`(또는 `.cursorignore`)로 에이전트가 불필요한 파일을 읽지 않게 해요. 컨텍스트 윈도우에 노이즈가 줄어들수록 응답 품질이 올라갑니다.

```gitignore
# .claudeignore
node_modules/
dist/
build/
*.lock
*.min.js
*.min.css

# 대용량 데이터
data/
fixtures/
__snapshots__/

# 자동 생성 파일
generated/
.next/
coverage/
```

**1M 토큰 윈도우의 함정:** 컨텍스트 윈도우가 크다고 다 채우면 오히려 성능이 떨어져요. 연구에 따르면 모델은 윈도우 크기와 무관하게 약 200K 토큰 이후 성능이 눈에 띄게 저하됩니다.

| 컨텍스트 크기 | 정확도 (상대값) | 권장 용도 |
|--------------|----------------|----------|
| ~50K 토큰 | 95%+ | 단일 파일/함수 작업 |
| ~200K 토큰 | 85-90% | 모듈 단위 작업 |
| 500K+ 토큰 | 70% 이하 | 피할 것 — 서브에이전트로 분할 |

## 기법 3: 동적 도구 선택

MCP 서버를 10개 연결해 놓으면 에이전트가 매번 모든 도구 스키마를 읽느라 토큰을 낭비해요. 작업에 필요한 도구만 활성화하는 전략이 필요합니다.

```json
{
  "mcpServers": {
    "github": {
      "command": "gh-mcp",
      "env": { "GITHUB_TOKEN": "${GITHUB_TOKEN}" }
    },
    "postgres": {
      "command": "pg-mcp",
      "disabled": true,
      "_comment": "DB 작업 시에만 활성화"
    }
  }
}
```

**작업별 도구 세트 예시:**

| 작업 유형 | 활성화할 도구 | 비활성화 |
|----------|-------------|---------|
| 코드 리뷰 | GitHub, 파일시스템 | DB, 브라우저 |
| DB 마이그레이션 | PostgreSQL, 파일시스템 | GitHub, 브라우저 |
| UI 개발 | 브라우저, 파일시스템 | DB, GitHub |
| 배포 | GitHub Actions, SSH | DB, 브라우저 |

## 기법 4: 세션 메모리와 핸드오프

긴 작업을 여러 세션에 걸쳐 진행할 때, 세션 간 컨텍스트를 구조화해서 전달하면 새 세션이 바로 이전 작업을 이어받을 수 있어요.

```markdown
# .claude/memory.md (자동 또는 수동 업데이트)

## 현재 작업 상태
- [x] API 스키마 설계 완료
- [x] 인증 미들웨어 구현
- [ ] 페이지네이션 추가 (다음 세션에서)

## 주요 결정사항
- JWT 대신 세션 기반 인증 채택 (이유: SSR 호환)
- PostgreSQL 선택 (이유: JSON 쿼리 필요)

## 알려진 이슈
- rate limiter 테스트가 CI에서 가끔 실패 (타이밍 이슈)
```

**핸드오프 패턴:**

```bash
# 세션 종료 시
# "현재 작업 상태를 .claude/memory.md에 정리해줘"

# 새 세션 시작 시
# ".claude/memory.md 읽고 이어서 작업해줘"
```

## 기법 5: 서브에이전트 격리

대규모 프로젝트에서는 하나의 에이전트에 모든 컨텍스트를 주는 대신, 서브에이전트에게 각각 좁은 범위의 컨텍스트만 줘서 병렬 처리하는 게 더 좋아요.

```bash
# 메인 에이전트가 태스크 분배
# 서브에이전트 1: 프론트엔드만 담당 (frontend/ 컨텍스트)
# 서브에이전트 2: 백엔드만 담당 (backend/ 컨텍스트)
# 서브에이전트 3: 테스트만 담당 (tests/ 컨텍스트)
```

**서브에이전트 격리의 장점:**

- 각 에이전트의 컨텍스트 윈도우가 작아져 정확도 상승
- 한 에이전트가 실패해도 다른 에이전트에 영향 없음
- 병렬 실행으로 전체 작업 시간 단축

| 방식 | 컨텍스트 크기 | 정확도 | 소요 시간 |
|------|-------------|--------|----------|
| 단일 에이전트 | 500K+ 토큰 | 낮음 | 30분+ |
| 서브에이전트 3개 | 각 100K 토큰 | 높음 | 10분 (병렬) |

## 기법 6: RAG 기반 동적 주입

프로젝트 문서, 이전 PR, 관련 코드를 벡터 DB에 저장해 두고, 에이전트가 필요할 때 검색해서 컨텍스트에 주입하는 방식이에요.

```yaml
# 컨텍스트 소스 설정 예시
context_sources:
  - type: codebase
    path: ./src
    index: semantic  # 시맨틱 검색 인덱스

  - type: docs
    path: ./docs
    format: markdown

  - type: history
    source: git_log
    depth: 100  # 최근 100 커밋
```

**실전 적용 사례:**

1. **코드 리뷰 시**: 변경된 파일과 관련된 다른 파일을 자동으로 찾아 컨텍스트에 포함
2. **버그 수정 시**: 유사한 이전 버그 수정 PR을 검색해서 패턴 참조
3. **새 기능 구현 시**: 기존 유사 기능의 구현 패턴을 자동으로 제공

## 실전 체크리스트

작업 시작 전 이 체크리스트로 컨텍스트를 점검하세요:

- [ ] CLAUDE.md가 현재 작업 디렉토리에 맞게 설정되어 있는가?
- [ ] .claudeignore로 불필요한 파일이 제외되어 있는가?
- [ ] 사용하지 않는 MCP 도구가 비활성화되어 있는가?
- [ ] 이전 세션의 메모리/핸드오프 파일이 최신인가?
- [ ] 작업 범위가 서브에이전트로 분할할 만큼 큰가?
- [ ] 참조할 문서나 코드가 검색 가능한 상태인가?
