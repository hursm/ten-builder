# 34. AI 코딩 에이전트 벤치마크 해석 가이드

> 숫자만 보면 착각해요. 벤치마크를 올바르게 읽는 방법을 알아봅시다.

## 왜 벤치마크 해석이 중요한가

새로운 AI 코딩 도구가 나올 때마다 "SWE-bench 72% 달성!" 같은 숫자가 화제가 됩니다. 하지만 그 숫자가 실무에서 어떤 의미인지 정확히 아는 개발자는 많지 않아요. 벤치마크를 제대로 읽을 수 있으면 도구 선택에서 시간과 비용을 절약할 수 있어요.

## 주요 벤치마크 한눈에 보기

| 벤치마크 | 측정 대상 | 언어 | 태스크 수 | 난이도 |
|----------|----------|------|-----------|--------|
| HumanEval | 단일 함수 생성 | Python | 164 | 낮음 |
| MBPP | 단일 함수 생성 | Python | 974 | 낮음~중간 |
| SWE-bench Verified | GitHub 이슈 해결 | Python | ~500 | 중간~높음 |
| SWE-Bench Pro | 멀티 언어 이슈 해결 | 다국어 | 1,865 | 높음 |
| ProjDevBench | 프로젝트 전체 개발 | 다국어 | 다양 | 매우 높음 |

## HumanEval과 MBPP — 기초 체력 측정

### 특징

```python
# HumanEval 태스크 예시
def has_close_elements(numbers: List[float], threshold: float) -> bool:
    """
    주어진 리스트에서 threshold보다 가까운 두 수가 있는지 확인
    """
    # AI가 이 함수를 완성하는 것이 과제
```

단일 함수 하나를 완성하는 테스트예요. 현재 대부분의 최신 모델이 90% 이상을 달성하기 때문에 모델 간 변별력이 거의 없어요.

### 한계

| 한계 | 설명 |
|------|------|
| Python 전용 | 다른 언어의 능력을 알 수 없음 |
| 단일 함수 | 실무의 멀티 파일 작업과 거리가 멀음 |
| 테스트 부족 | 엣지케이스 검증이 빈약 |
| 데이터 오염 | 학습 데이터에 포함되었을 가능성이 높음 |

**결론:** HumanEval/MBPP 점수만으로 도구를 선택하는 건 의미가 없어요. "90% 이상이면 기본기는 된다" 정도로만 참고하세요.

## SWE-bench — 현실적인 코딩 능력 측정

### SWE-bench Verified vs SWE-Bench Pro

```
SWE-bench Verified (기존)
├── Python 전용
├── ~500개 태스크
├── 데이터 오염 문제 제기됨
└── 점수가 과대평가될 가능성

SWE-Bench Pro (2026~)
├── Python, JavaScript, TypeScript, Go, Rust, Java
├── 1,865개 태스크
├── 오염 방지 설계
└── 더 까다로운 평가 기준
```

### 점수 읽는 법

SWE-bench Verified에서 80%인 모델이 SWE-Bench Pro에서는 50% 미만일 수 있어요. 같은 이름이라고 같은 난이도가 아닙니다.

| 벤치마크 | 좋은 점수 기준 | 현재 최고 성능 |
|----------|---------------|---------------|
| SWE-bench Verified | 60% 이상 | ~81% |
| SWE-Bench Pro | 40% 이상 | ~58% |

**핵심:** 두 벤치마크의 점수를 직접 비교하면 안 돼요. SWE-Bench Pro 40%가 Verified 80%보다 실무에 가까운 지표일 수 있어요.

### 데이터 오염 문제

```
오염 시나리오:
1. AI 모델 학습 데이터에 SWE-bench 태스크의 정답이 포함
2. 모델이 "이해"가 아닌 "기억"으로 문제를 풀게 됨
3. 벤치마크 점수는 높지만 새로운 문제에서는 성능 하락

SWE-Bench Pro가 이를 해결하는 방법:
- 최신 이슈 사용 (학습 데이터 이후 생성)
- 다국어 태스크로 범위 확장
- 주기적 태스크 갱신
```

## 벤치마크 외에 확인할 것

### 실무 적합성 체크리스트

벤치마크 점수 외에 실제로 확인해야 할 항목이에요:

```yaml
# 실무 평가 체크리스트
technical:
  - 내 주 사용 언어에서 코드 품질은 어떤가?
  - 멀티 파일 변경을 잘 처리하는가?
  - 기존 코드 스타일을 따르는가?
  - 에러 발생 시 자체 수정이 가능한가?

workflow:
  - 컨텍스트 윈도우가 프로젝트 규모에 충분한가?
  - IDE/터미널 통합이 자연스러운가?
  - 반복 작업 자동화가 가능한가?

cost:
  - 토큰당 비용이 예산 내인가?
  - 구독 플랜 vs API 과금 중 어떤 게 유리한가?
  - 캐싱/프롬프트 최적화를 지원하는가?
```

### 직접 테스트 방법

벤치마크를 신뢰하기 어려울 때, 직접 확인하는 방법이에요:

```bash
# 1. 내 프로젝트에서 최근 머지된 PR을 기반으로 테스트
git log --oneline -10

# 2. 특정 커밋의 변경사항을 되돌린 뒤 AI에게 같은 작업 요청
git diff HEAD~1 HEAD -- src/

# 3. AI가 생성한 결과와 실제 커밋 비교
# - 코드 품질, 테스트 포함 여부, 스타일 일관성 체크
```

이 방법은 벤치마크 점수보다 훨씬 실질적인 판단 기준을 제공해요.

## 벤치마크 비교 시 흔한 실수

| 실수 | 올바른 해석 |
|------|------------|
| "A 모델이 80%니까 B 모델 60%보다 무조건 좋다" | 같은 벤치마크인지, 버전이 같은지 확인 |
| "HumanEval 95%니까 코딩 잘 함" | 단일 함수 생성과 실무 코딩은 다른 영역 |
| "SWE-bench 1위니까 최고" | 데이터 오염 가능성과 비용 효율도 고려 |
| "벤치마크 점수 = 실무 생산성" | 벤치마크는 특정 조건의 테스트일 뿐 |
| "최신 벤치마크가 항상 더 정확" | 태스크 설계와 평가 방법론을 함께 확인 |

## 실전 의사결정 프레임워크

```
1단계: 기본 필터링
  └── HumanEval/MBPP 90% 이상? → 기본기 확인 완료

2단계: 심층 비교
  └── SWE-Bench Pro 점수 비교 (같은 버전 기준)
  └── 가성비 (점수/비용) 계산

3단계: 실무 검증
  └── 내 프로젝트에서 3개 이상 태스크 직접 테스트
  └── 코드 품질, 속도, 비용 종합 평가

4단계: 최종 선택
  └── 벤치마크 30% + 실무 테스트 50% + 비용 20% 가중치
```

## 2026년 벤치마크 동향

- **SWE-Bench Pro**가 사실상 표준 벤치마크로 자리잡는 추세예요
- 멀티 언어 지원으로 Python 편향 문제가 줄어들었어요
- 모델별 가성비(점수 대비 비용) 비교가 점점 중요해지고 있어요
- ProjDevBench 같은 프로젝트 단위 벤치마크가 새로운 평가 축으로 떠오르고 있어요

## 정리

| 핵심 포인트 | 설명 |
|------------|------|
| 같은 벤치마크끼리 비교 | Verified와 Pro를 섞어서 비교하면 안 됨 |
| 데이터 오염 의심 | 비정상적으로 높은 점수는 오염 가능성 체크 |
| 실무 테스트 병행 | 벤치마크만으로 도구를 결정하지 않기 |
| 비용 대비 성능 | 점수 1% 차이보다 비용 효율이 더 중요할 수 있음 |
| 내 사용 패턴 기준 | 주 사용 언어와 작업 유형에 맞는 벤치마크 참고 |
