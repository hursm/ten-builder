# AI 코딩 에이전트 트러블슈팅 실전 가이드

> AI 에이전트가 막혔을 때 빠르게 원인을 찾고 해결하는 7가지 패턴

## 이 가이드에서 다루는 것

AI 코딩 에이전트를 쓰다 보면 누구나 겪는 문제들이 있어요. 무한 루프에 빠지거나, 엉뚱한 파일을 수정하거나, 컨텍스트 윈도우가 넘쳐서 앞부분 지시를 까먹거나. 이 가이드는 실무에서 자주 만나는 7가지 문제 상황과 각각의 해결 패턴을 정리합니다.

## 문제 1: 무한 루프 — 같은 수정을 반복한다

에이전트가 같은 에러를 고치려고 동일한 코드를 반복 수정하는 현상이에요. 특히 타입 에러나 린트 에러가 서로 충돌할 때 자주 발생합니다.

### 증상

- 같은 파일을 3회 이상 연속 편집
- 에러 메시지가 A → B → A 순환 패턴
- 토큰 소비가 급격히 증가

### 해결 패턴

```bash
# 1. 즉시 중단하고 현재 상태 확인
git diff --stat

# 2. 에이전트에게 새로운 접근법을 지시
# Bad: "이 에러 고쳐줘" (같은 접근 반복)
# Good: "현재 접근이 순환하고 있어. 다른 방법을 시도해줘.
#        먼저 에러의 근본 원인을 분석하고, 3가지 대안을 제시한 후
#        가장 적합한 방법으로 진행해."
```

### 예방 설정

```yaml
# CLAUDE.md에 추가
rules:
  - 같은 파일을 3회 이상 연속 수정하면 멈추고 상황을 보고할 것
  - 에러 수정 시 먼저 근본 원인을 분석하고 수정 계획을 설명할 것
```

| 도구 | 루프 감지 기능 |
|------|---------------|
| Claude Code | `--max-turns` 옵션으로 턴 수 제한 |
| Cursor | Agent 모드에서 자동 루프 감지 (5회) |
| Aider | `/clear` 명령으로 컨텍스트 리셋 |

## 문제 2: 컨텍스트 윈도우 오버플로우

긴 대화나 많은 파일을 열면 에이전트가 초기 지시사항을 잊어버려요. 특히 200K+ 토큰 세션에서 발생합니다.

### 증상

- 프로젝트 규칙(CLAUDE.md)을 무시하기 시작
- 이전에 합의한 아키텍처와 다른 코드 생성
- "이전 대화에서 말했듯이..."에 대해 엉뚱한 답변

### 해결 패턴

```bash
# 1. 현재 컨텍스트 사용량 확인
# Claude Code: /cost 명령어
# Cursor: 우측 하단 토큰 카운터

# 2. 새 세션 시작 + 핵심 컨텍스트만 전달
# 이전 세션의 결론만 정리해서 새 세션에 전달

cat > handoff.md << 'EOF'
## 이전 세션 결론
- API 구조: REST, /api/v1 prefix
- 인증: JWT + refresh token
- 남은 작업: 사용자 프로필 엔드포인트
EOF
```

### 예방 전략

```markdown
<!-- CLAUDE.md에서 컨텍스트 관리 규칙 -->
## 세션 관리
- 단일 작업 단위로 세션을 분리할 것
- 파일 전체 내용 대신 관련 부분만 참조할 것
- 200K 토큰 초과 시 새 세션 시작을 제안할 것
```

| 전략 | 효과 | 적용 시점 |
|------|------|----------|
| 세션 분리 | 컨텍스트 오염 방지 | 태스크 변경 시 |
| 선택적 파일 로딩 | 토큰 절약 50%+ | 대규모 코드베이스 |
| 핸드오프 문서 | 지식 연속성 유지 | 세션 교체 시 |

## 문제 3: 엉뚱한 파일 수정

요청하지 않은 파일을 수정하거나, 잘못된 파일에 코드를 추가하는 경우예요.

### 증상

- `git diff`에 예상하지 못한 파일이 포함됨
- 설정 파일이나 패키지 잠금 파일이 변경됨
- 다른 모듈의 코드가 영향을 받음

### 해결 패턴

```bash
# 1. 변경된 파일 확인
git diff --name-only

# 2. 원하지 않는 변경 되돌리기
git checkout -- path/to/unwanted-file.ts

# 3. 에이전트에게 범위를 명확히 제한
# "src/api/users.ts 파일만 수정해. 다른 파일은 절대 건드리지 마."
```

### 예방 설정

```yaml
# .claudeignore 또는 .cursorignore
# 에이전트가 수정하면 안 되는 파일 목록
package-lock.json
yarn.lock
pnpm-lock.yaml
*.generated.ts
migrations/*.sql
```

## 문제 4: 환각(Hallucination) — 존재하지 않는 API 사용

실제로 없는 라이브러리 함수나 API 엔드포인트를 사용하는 코드를 생성하는 문제예요.

### 증상

- `import` 경로가 존재하지 않는 모듈을 참조
- 라이브러리의 없는 메서드를 호출
- 잘못된 시그니처로 함수를 호출

### 해결 패턴

```bash
# 1. 타입 체크로 빠르게 검증
npx tsc --noEmit  # TypeScript
mypy .            # Python
go vet ./...      # Go

# 2. 에이전트에게 검증을 요청
# "방금 사용한 API가 실제로 존재하는지 확인해줘.
#  공식 문서나 타입 정의 파일을 참조해서 검증해."
```

### 예방 전략

```markdown
<!-- CLAUDE.md 규칙 -->
## 코드 생성 규칙
- 외부 라이브러리 사용 시 반드시 node_modules 또는 타입 정의를 확인할 것
- 새 의존성 추가 전 package.json에 존재하는지 확인할 것
- 코드 생성 후 반드시 타입 체크를 실행할 것
```

| 언어 | 검증 명령어 | 확인 범위 |
|------|-----------|----------|
| TypeScript | `npx tsc --noEmit` | 타입, import 경로 |
| Python | `mypy --ignore-missing-imports .` | 타입 힌트, 함수 시그니처 |
| Go | `go vet ./...` | 미사용 변수, 잘못된 호출 |
| Rust | `cargo check` | 전체 컴파일 검증 |

## 문제 5: 불완전한 구현 — 반쪽짜리 코드

함수 시그니처만 만들고 내용은 `// TODO`로 채우거나, 에러 핸들링을 생략하는 경우예요.

### 증상

- `TODO`, `FIXME` 주석만 남기고 넘어감
- `catch` 블록이 비어있음
- 엣지 케이스 처리 누락

### 해결 패턴

```bash
# 1. 불완전 구현 탐지
grep -rn "TODO\|FIXME\|HACK\|XXX" src/
grep -rn "catch.*{}" src/  # 빈 catch 블록

# 2. 에이전트에게 완성을 요청
# "프로젝트에서 TODO/FIXME를 검색하고 전부 구현해줘.
#  빈 catch 블록도 적절한 에러 핸들링으로 채워줘."
```

### 예방 설정

```markdown
<!-- CLAUDE.md -->
## 구현 완결성 규칙
- TODO/FIXME 주석을 남기지 않는다. 즉시 구현하거나 명시적으로 범위 외임을 설명한다.
- 모든 catch 블록에 로깅 또는 에러 전파를 구현한다.
- 함수 stub만 만들지 않는다. 기본 구현까지 완료한다.
```

## 문제 6: 느린 응답 — 토큰 소비만 늘어남

간단한 작업인데 에이전트가 불필요하게 많은 파일을 읽거나, 긴 설명을 추가하는 경우예요.

### 증상

- 한 줄 수정에 5분 이상 소요
- 파일을 하나씩 전부 열어보는 동작
- 불필요한 리팩토링을 함께 수행

### 해결 패턴

```bash
# 1. 정확한 파일과 라인을 지정
# Bad: "로그인 버그 고쳐줘"
# Good: "src/auth/login.ts 45번째 줄의 null 체크를 추가해줘"

# 2. 범위를 명확히 제한
# "이 파일만 수정해. 리팩토링하지 마. 최소한의 변경만."
```

### 효율적인 프롬프팅 패턴

| 상황 | 비효율적 | 효율적 |
|------|---------|--------|
| 버그 수정 | "이 버그 고쳐줘" | "src/api.ts:45에서 null 체크 추가" |
| 기능 추가 | "사용자 기능 만들어줘" | "POST /api/users 엔드포인트 추가. 스키마는 User 모델 참조" |
| 리팩토링 | "코드 개선해줘" | "UserService 클래스에서 중복된 validateEmail을 utils로 추출" |

## 문제 7: 테스트 실패 무시

에이전트가 생성한 코드의 테스트가 실패하는데 무시하고 넘어가거나, 테스트 자체를 삭제하는 경우예요.

### 증상

- 기존 테스트가 깨졌는데 새 코드만 추가
- 실패하는 테스트를 `.skip()`으로 건너뜀
- 테스트 assertion을 현재 (잘못된) 출력에 맞춰 변경

### 해결 패턴

```bash
# 1. 테스트 실행을 강제
npm test         # 전체 테스트
npm test -- --watch  # 변경 감지 모드

# 2. 에이전트에게 테스트 우선 지시
# "먼저 기존 테스트가 전부 통과하는지 확인해.
#  실패하는 테스트가 있으면 코드를 수정해서 통과시켜.
#  테스트를 skip하거나 삭제하지 마."
```

### 예방 설정

```markdown
<!-- CLAUDE.md -->
## 테스트 규칙
- 코드 수정 후 반드시 관련 테스트를 실행할 것
- 기존 테스트를 삭제하거나 skip하지 않을 것
- 테스트 실패 시 테스트를 고치지 말고 코드를 고칠 것
- 새 기능에는 반드시 테스트를 함께 추가할 것
```

## 종합 트러블슈팅 플로차트

```
에이전트가 기대대로 동작하지 않음
    │
    ├─ 같은 동작 반복? → 문제 1: 루프 감지 → 중단 + 접근법 변경
    │
    ├─ 이전 지시 무시? → 문제 2: 컨텍스트 오버플로우 → 새 세션 + 핸드오프
    │
    ├─ 엉뚱한 파일 변경? → 문제 3: 범위 이탈 → git checkout + 범위 지정
    │
    ├─ 없는 API 사용? → 문제 4: 환각 → 타입 체크 + 문서 참조 강제
    │
    ├─ TODO만 남김? → 문제 5: 불완전 구현 → 완결성 규칙 설정
    │
    ├─ 너무 느림? → 문제 6: 비효율 → 정확한 위치 + 범위 제한
    │
    └─ 테스트 깨짐? → 문제 7: 테스트 무시 → 테스트 우선 규칙 설정
```

## CLAUDE.md 통합 템플릿

위 7가지 문제를 예방하는 규칙을 한 곳에 모았어요:

```markdown
## AI 에이전트 안전 규칙

### 루프 방지
- 같은 파일을 3회 연속 수정하면 멈추고 보고할 것
- 에러 수정 시 근본 원인을 먼저 분석할 것

### 범위 제한
- 요청받은 파일만 수정할 것
- package-lock.json, migration 파일은 직접 수정하지 않을 것

### 검증 필수
- 코드 생성 후 타입 체크를 실행할 것
- 외부 라이브러리 API는 타입 정의로 검증할 것

### 테스트 우선
- 기존 테스트를 삭제/skip하지 않을 것
- 새 기능에 테스트를 포함할 것

### 완결성
- TODO/FIXME를 남기지 않을 것
- 에러 핸들링을 생략하지 않을 것
```
