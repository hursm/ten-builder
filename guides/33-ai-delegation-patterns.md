# 가이드 33: AI 코딩 위임 판단 프레임워크

> 어떤 작업을 AI 에이전트에게 맡기고, 어떤 작업은 직접 해야 할까? — 복잡도, 위험도, 반복성 기준 의사결정 매트릭스

## 왜 위임 판단이 중요한가

AI 코딩 에이전트가 점점 유능해지면서 "모든 걸 AI에게 맡기면 되지 않나?"라는 생각이 들 수 있어요. 하지만 실제로는 AI에게 맡겼다가 오히려 시간이 더 걸리는 경우도 많아요. 핵심은 **어떤 작업을 맡길지 판단하는 능력** 자체가 개발자의 새로운 역량이 되었다는 거예요.

2026년 Anthropic의 에이전틱 코딩 트렌드 보고서에 따르면, AI 위임의 성공은 **검증 가능성(verifiability)**, **노력 비율(effort ratio)**, **신뢰 보정(trust calibration)** 세 가지 축으로 결정돼요.

## 의사결정 매트릭스: 3축 모델

### 축 1: 검증 가능성 (Verifiability)

AI가 만든 결과물을 얼마나 쉽게 확인할 수 있는지가 가장 중요한 기준이에요.

| 검증 난이도 | 예시 | AI 위임 적합도 |
|------------|------|---------------|
| 쉬움 — 테스트로 확인 | 유닛 테스트, API 엔드포인트, CRUD | ✅ 적극 위임 |
| 보통 — 리뷰로 확인 | 리팩토링, 문서 생성, 타입 추가 | ⚠️ 조건부 위임 |
| 어려움 — 전문가 판단 필요 | 아키텍처 설계, 보안 로직, 동시성 | ❌ 직접 수행 |

```bash
# 검증 가능성이 높은 작업 = AI 위임에 적합
# 테스트 파일이 있으면 AI가 만든 코드를 바로 검증 가능
claude "이 함수에 대한 유닛 테스트를 추가해줘" --file src/utils/parser.ts

# 검증이 어려운 작업 = 직접 수행
# 인증 플로우 설계는 테스트만으로 정확성 판단이 어려움
```

### 축 2: 노력 비율 (Effort Ratio)

"AI에게 설명하는 시간" 대 "직접 하는 시간"의 비율이에요.

| 비율 | 판단 | 예시 |
|------|------|------|
| 설명 < 직접 작업의 1/3 | AI에게 맡기기 | 100줄짜리 보일러플레이트 생성 |
| 설명 ≈ 직접 작업 | 상황에 따라 판단 | 중간 복잡도 리팩토링 |
| 설명 > 직접 작업 | 직접 하기 | 3줄 수정, 변수명 변경 |

실전에서 자주 하는 실수는 **"간단한 수정인데 AI에게 맡기는 것"**이에요. `git diff`로 3줄 고칠 걸 프롬프트 작성에 5분을 쓰는 건 비효율적이에요.

### 축 3: 위험도 (Risk Level)

실수했을 때 발생하는 영향의 크기예요.

| 위험도 | 특징 | 전략 |
|--------|------|------|
| 낮음 | 되돌리기 쉬움, 프로덕션 영향 없음 | 자율 위임 (yolo 모드) |
| 중간 | 부분적 영향, 리뷰 후 배포 | 위임 후 리뷰 |
| 높음 | 데이터 손실, 보안 이슈, 과금 | 직접 수행 또는 최소 범위만 위임 |

## 실전 판단 플로우차트

```
새 작업 도착
    │
    ├─ Q1: 테스트/린터로 결과를 자동 검증할 수 있는가?
    │   ├─ Yes → Q2로
    │   └─ No → Q4: 전문 판단이 필요한가?
    │       ├─ Yes → 직접 수행
    │       └─ No → 위임 후 수동 리뷰
    │
    ├─ Q2: 프롬프트 작성이 직접 코딩보다 빠른가?
    │   ├─ Yes → Q3로
    │   └─ No → 직접 수행
    │
    └─ Q3: 실수 시 되돌리기 쉬운가?
        ├─ Yes → AI에게 위임
        └─ No → 위임하되, 작은 단위로 분할하여 단계별 검증
```

## 카테고리별 위임 가이드

### 적극 위임 (AI가 잘하는 영역)

| 작업 | 이유 | 프롬프트 팁 |
|------|------|------------|
| 보일러플레이트 생성 | 패턴이 명확하고 반복적 | 기존 파일을 참조로 제공 |
| 유닛 테스트 작성 | 테스트 자체가 검증 수단 | 엣지 케이스 목록 명시 |
| 타입 추가 (JS→TS) | 기계적이고 검증 가능 | 기존 인터페이스 파일 함께 제공 |
| 문서/README 생성 | 구조가 정형화됨 | 톤, 대상 독자 지정 |
| 정규식/쿼리 작성 | 테스트 케이스로 바로 검증 | 입력-출력 예시 3개 이상 제공 |
| 에러 핸들링 추가 | 패턴화되어 있음 | 프로젝트의 에러 처리 컨벤션 명시 |

```bash
# 보일러플레이트: AI가 가장 잘하는 영역
claude "Express 라우터를 만들어줘. 
GET /users, POST /users, GET /users/:id, PUT /users/:id, DELETE /users/:id
기존 라우터 참고: src/routes/products.ts"

# 유닛 테스트: 자체 검증이 가능해서 위임에 적합
claude "src/services/order.ts의 calculateTotal 함수에 대한 테스트를 작성해줘.
경계값: 빈 배열, 할인 100%, 소수점 반올림 케이스 포함"
```

### 조건부 위임 (맥락에 따라 판단)

| 작업 | 위임 조건 | 주의점 |
|------|----------|--------|
| 리팩토링 | 테스트 커버리지 70% 이상일 때 | 한 번에 한 패턴만 변경 |
| 버그 수정 | 재현 조건이 명확할 때 | 로그/스택트레이스 함께 제공 |
| API 통합 | 공식 문서 URL 제공 가능할 때 | 인증 토큰 등 시크릿 주의 |
| DB 마이그레이션 | 스테이징 환경이 있을 때 | 프로덕션 직접 실행 금지 |
| 성능 최적화 | 프로파일링 결과가 있을 때 | 벤치마크 전후 비교 필수 |

```bash
# 리팩토링: 테스트가 있으면 위임 가능
claude "이 파일의 콜백 중첩을 async/await로 변환해줘.
변환 후 기존 테스트가 모두 통과해야 해.
파일: src/legacy/payment-processor.js"

# 버그 수정: 재현 조건이 명확하면 위임 가능
claude "이 에러를 수정해줘:
TypeError: Cannot read property 'map' of undefined at UserList.tsx:42
재현: /users 페이지에서 검색 결과가 0건일 때 발생
기대: 빈 배열이면 '결과 없음' 메시지 표시"
```

### 직접 수행 (AI에게 맡기면 위험한 영역)

| 작업 | 이유 |
|------|------|
| 아키텍처 의사결정 | 비즈니스 맥락 이해 필요 |
| 보안 인증/인가 로직 | 미세한 실수가 치명적 |
| 결제/과금 로직 | 금전적 손실 위험 |
| 데이터 마이그레이션 (프로덕션) | 되돌리기 어려움 |
| 동시성/레이스 컨디션 | AI가 놓치기 쉬운 미묘한 버그 |
| 서드파티 SLA 관련 코드 | 장애 시 외부 영향 |

## 위임 효과를 높이는 5가지 패턴

### 1. 검증 퍼스트 (Verification-First)

AI에게 코드를 맡기기 전에 **검증 수단을 먼저 준비**하세요.

```bash
# 나쁜 예: 바로 구현 위임
claude "결제 모듈을 만들어줘"

# 좋은 예: 테스트를 먼저 작성하고 구현 위임
# 1단계: 내가 직접 테스트 작성
vim tests/payment.test.ts

# 2단계: 구현을 AI에게 위임
claude "tests/payment.test.ts의 모든 테스트를 통과하는 
src/services/payment.ts를 구현해줘"
```

### 2. 점진적 위임 (Incremental Delegation)

큰 작업은 작은 단계로 나눠서 각 단계를 개별 위임하세요.

```bash
# 나쁜 예: 한 번에 전부
claude "이 프로젝트를 모노레포로 전환해줘"

# 좋은 예: 단계별 위임
claude "Step 1: 현재 프로젝트 구조를 분석하고 모노레포 전환 계획을 작성해줘"
# → 계획 리뷰
claude "Step 2: shared 패키지를 packages/shared로 분리해줘"
# → 빌드 테스트
claude "Step 3: 각 앱의 import 경로를 workspace 패키지로 변경해줘"
# → 통합 테스트
```

### 3. 컨텍스트 브리핑 (Context Briefing)

AI가 잘 모르는 프로젝트 특수 사정을 미리 알려주세요.

```bash
# 프로젝트 컨벤션을 명시하면 재작업이 줄어듦
claude "새 API 엔드포인트를 추가해줘.
- 우리 프로젝트는 에러 응답에 항상 { code, message, details } 형식 사용
- 인증은 middleware/auth.ts의 requireAuth 미들웨어 사용
- 로깅은 utils/logger.ts의 structuredLog 사용
- 참고 파일: routes/products.ts (동일 패턴)"
```

### 4. 탈출 기준 설정 (Escape Criteria)

AI가 어려워하면 빠르게 회수하세요. 2~3번 시도해도 안 되면 직접 하는 게 빠릅니다.

| 시도 횟수 | 판단 |
|----------|------|
| 1회 | 프롬프트를 개선해서 재시도 |
| 2회 | 더 작은 단위로 분할하여 재시도 |
| 3회 | 직접 수행으로 전환 |

### 5. 피드백 루프 기록 (Feedback Loop)

어떤 작업을 위임했고 결과가 어땠는지 간단히 기록하면, 점점 판단이 정확해져요.

```markdown
## 위임 로그 (이번 주)
| 작업 | 위임 결과 | 비고 |
|------|----------|------|
| 유닛 테스트 추가 | ✅ 1회 성공 | 엣지 케이스 명시가 핵심 |
| OAuth 통합 | ⚠️ 3회 시도 후 수동 | 리다이렉트 플로우가 복잡 |
| DB 스키마 변경 | ✅ 위임 후 리뷰 | 마이그레이션 SQL 직접 확인 |
| 인증 미들웨어 | ❌ 직접 수행 | 보안 로직은 직접 작성 |
```

## 팀 단위 위임 가이드라인

### 역할별 위임 범위

| 역할 | AI 위임 추천 범위 | 직접 수행 영역 |
|------|-----------------|---------------|
| 주니어 | 보일러플레이트, 테스트, 문서 | 코드 리뷰 결과 이해, 로직 학습 |
| 시니어 | 리팩토링, 프로토타입, 코드 리뷰 초안 | 아키텍처, 설계 의사결정 |
| 리드 | 대규모 마이그레이션, CI/CD | 기술 전략, 보안 정책 |
| DevOps | IaC 생성, 모니터링 설정 | 프로덕션 배포 승인, 인시던트 대응 |

### 팀 공유 위임 정책 예시

```yaml
# .ai-delegation-policy.yaml
delegation_rules:
  auto_approve:  # AI 자율 실행 허용
    - unit_tests
    - type_annotations  
    - documentation
    - linting_fixes
    
  require_review:  # AI 생성 후 리뷰 필수
    - refactoring
    - new_endpoints
    - dependency_updates
    - migration_scripts
    
  manual_only:  # AI 위임 금지
    - auth_logic
    - payment_processing
    - data_deletion
    - production_deployment
    - security_configurations
```

## 자주 하는 실수 3가지

### 1. 과도한 위임

"AI가 다 해줄 거야"라고 기대하며 복잡한 아키텍처 결정까지 맡기는 경우예요. AI는 당신의 비즈니스 맥락을 모릅니다.

### 2. 과도한 불신

"AI 코드는 믿을 수 없어"라며 3줄짜리 유틸 함수도 직접 작성하는 경우예요. 테스트로 검증 가능한 단순 작업은 위임이 효율적입니다.

### 3. 검증 없는 수용

AI가 생성한 코드를 리뷰 없이 그대로 커밋하는 경우예요. 특히 의존성 추가, 환경 변수, 네트워크 호출이 포함된 코드는 반드시 확인하세요.

## 체크리스트: 위임 전 30초 판단

- [ ] 결과를 테스트/린터로 자동 검증할 수 있는가?
- [ ] 프롬프트 작성이 직접 코딩보다 빠른가?
- [ ] 실수해도 쉽게 되돌릴 수 있는가?
- [ ] 보안/결제/데이터 삭제와 무관한가?
- [ ] 필요한 컨텍스트를 AI에게 충분히 전달할 수 있는가?

3개 이상 Yes → AI에게 위임
2개 → 작은 단위로 분할 후 위임
1개 이하 → 직접 수행
