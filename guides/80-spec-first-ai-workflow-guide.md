# 스펙-퍼스트 개발 실전 가이드 2026 — AI 에이전트와 함께 코드 전에 설계하기

> 코드보다 스펙이 먼저다 — 요구사항을 먼저 정의하고 AI가 구현하는 개발 방식의 실전 전략

## 왜 스펙-퍼스트인가

AI 코딩 에이전트 도입률이 90%를 넘어선 지금, 많은 팀이 같은 실수를 반복한다. 막연한 지시로 바로 코드를 생성하고, 결과물이 의도와 다르면 수정 요청을 반복하는 사이클이다.

문제는 AI가 아니라 입력이다. **명확한 스펙이 없으면 AI는 추측으로 코드를 채운다.** 추측이 쌓인 코드는 나중에 더 큰 비용을 낳는다.

스펙-퍼스트(Spec-First)는 새로운 개념이 아니다. 전통적인 소프트웨어 개발에서 오래된 원칙이지만, AI 코딩 에이전트 시대에 다시 주목받고 있다. AI가 강력할수록 좋은 스펙의 가치는 더 커진다.

## 스펙-퍼스트 워크플로우 4단계

### Step 1: AI와 함께 요구사항 정의

코드를 한 줄도 쓰기 전에 AI와 대화로 스펙을 만든다.

```
프롬프트 예시:
"사용자 인증 기능을 만들려고 해. 구현하기 전에
요구사항과 엣지케이스를 질문으로 뽑아줘.
빠진 게 없을 때까지 계속 물어봐."
```

AI가 되묻는 질문들:
- 소셜 로그인이 필요한가, 이메일/비밀번호만인가?
- 비밀번호 재설정 플로우는?
- 세션 만료 정책은?
- 멀티 디바이스 동시 로그인은 허용하는가?

이 대화를 문서로 저장한다. 이게 스펙의 초안이다.

### Step 2: 스펙 문서 작성

대화 결과를 구조화된 문서로 정리한다.

```markdown
# 기능명: 사용자 인증

## 범위
- 이메일/비밀번호 기반 회원가입 및 로그인
- JWT 기반 세션 관리 (액세스 7일, 리프레시 30일)
- 비밀번호 재설정 (이메일 인증)
- 로그인 실패 5회 시 계정 잠금 (15분)

## 제외 범위
- 소셜 로그인 (다음 스프린트)
- 2FA (별도 태스크)

## 엣지케이스
- 만료된 리프레시 토큰 처리
- 동시 로그인 기기 3대 제한
- 탈퇴 후 재가입 시 이전 데이터 처리

## 수용 기준
- 로그인 성공 응답: 200ms 이하
- 비밀번호 최소 8자, 대소문자+숫자 혼합
```

### Step 3: AI에게 스펙 기반 구현 요청

스펙 문서를 컨텍스트로 제공하고 구현을 요청한다.

```bash
# CLAUDE.md 또는 프롬프트에 스펙 파일 포함
cat spec/auth-spec.md | claude "이 스펙을 기반으로 구현해줘.
스펙에 없는 내용은 임의로 추가하지 말고 먼저 물어봐."
```

**핵심 지시사항:**
- 스펙에 없는 내용은 구현하지 않는다
- 불명확한 부분은 코드 생성 전에 질문한다
- 수용 기준을 테스트로 먼저 작성한다

### Step 4: 스펙과 구현 일치 검증

AI가 구현을 마치면 스펙 대비 완성도를 확인한다.

```
"방금 구현한 코드가 spec/auth-spec.md의 모든 요구사항을
커버하는지 항목별로 체크해줘. 누락된 게 있으면 표로 정리해줘."
```

| 요구사항 | 구현 여부 | 파일 |
|----------|----------|------|
| 로그인 실패 잠금 | ✅ | `src/auth/lockout.ts` |
| 리프레시 토큰 만료 처리 | ✅ | `src/auth/token.ts` |
| 동시 로그인 제한 | ❌ 누락 | — |

누락 항목만 추가 구현 요청한다.

## CLAUDE.md에 스펙-퍼스트 규칙 추가

프로젝트 CLAUDE.md에 스펙-퍼스트 원칙을 명시해두면 매번 지시하지 않아도 된다.

```markdown
## 개발 원칙

### 스펙-퍼스트
- 새 기능 구현 전 항상 spec/ 폴더의 해당 스펙 파일을 먼저 확인
- 스펙 파일이 없으면 구현 전에 작성 요청
- 스펙에 명시되지 않은 기능은 임의로 추가하지 않음
- 스펙과 구현이 다를 경우 구현을 수정 (스펙 임의 변경 금지)

### 스펙 파일 위치
- `spec/{기능명}-spec.md` 형식으로 관리
- 변경 시 Git으로 버전 관리
```

## 실전 팁: 스펙 품질이 결과를 결정한다

### 좋은 스펙의 조건

| 항목 | 나쁜 예 | 좋은 예 |
|------|---------|---------|
| 범위 | "검색 기능 만들어줘" | "상품명, 카테고리, 가격 범위로 필터링. 결과는 최대 50개, 페이지네이션 포함" |
| 엣지케이스 | 미정의 | "검색어 0자: 전체 목록 반환, 100자 초과: 에러 반환" |
| 수용 기준 | "잘 작동해야 함" | "응답 500ms 이하, 결과 정확도 95% 이상" |
| 제외 범위 | 미정의 | "자동완성, 오타 교정은 이번 범위 외" |

### 반복 피드백 대신 스펙 보완

AI 결과물이 기대와 다를 때, 재지시 전에 먼저 물어본다:

> 스펙에 이 부분이 충분히 명확하게 적혀 있었나?

명확하지 않았다면 → 스펙을 보완하고 재구현 요청  
명확했는데 틀렸다면 → AI 지시 방식을 개선

## 팀에서 스펙-퍼스트 도입하기

1. **스펙 템플릿 만들기**: 기능 스펙, API 스펙, 데이터 모델 스펙 템플릿 준비
2. **스펙 리뷰를 코드 리뷰만큼 중요하게**: PR 생성 전 스펙 존재 여부 체크
3. **AI가 스펙 작성을 도우도록**: "이 기능의 스펙 초안을 작성해줘, 빠진 항목이 있으면 질문해줘"
4. **스펙과 코드를 같이 버전 관리**: 기능이 바뀌면 스펙도 함께 업데이트

## 자주 겪는 문제

| 문제 | 원인 | 해결 |
|------|------|------|
| AI가 스펙 외 기능을 추가함 | 스펙에 제외 범위 미정의 | 스펙에 "이번 범위 외" 섹션 추가 |
| 스펙 작성이 오히려 느림 | 너무 상세하게 쓰려는 시도 | AI와 대화로 초안 먼저, 세부화는 나중에 |
| 구현하다 스펙이 바뀜 | 초기 요구사항 불명확 | Step 1의 대화 시간을 더 길게 |
| 팀원이 스펙을 안 씀 | 습관 부재 | CLAUDE.md에 규칙 추가, 자동화 체크 |

## 마무리

스펙-퍼스트는 속도를 늦추는 게 아니라 방향을 잡는 과정이다. AI가 빠를수록 잘못된 방향으로도 빠르게 달려간다. 스펙은 그 방향타다.

"코드 전에 스펙, 스펙 전에 대화" — 이 순서를 지키는 팀이 AI 코딩 에이전트를 가장 효과적으로 활용한다.
