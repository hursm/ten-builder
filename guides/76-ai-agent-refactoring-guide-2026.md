# AI 에이전트 코드 리팩토링 가이드 2026 — 레거시 제거에서 모던 패턴까지

> 기술 부채는 쌓이는 속도보다 느리게 갚으면 쌓인다. AI 에이전트는 그 속도를 역전시킨다.

## 왜 지금 AI 리팩토링인가

바이브 코딩이 확산되면서 AI 생성 코드의 기술 부채도 함께 늘고 있다. 코드 중복, 숨겨진 보안 취약점, 테스트 없는 로직이 쌓이는 속도가 빨라졌다. 역설적이게도 AI는 부채의 원인이면서 동시에 해결 도구다.

2026년 기준으로 Claude Code, Gemini CLI 등은 멀티파일 리팩토링을 코드베이스 인덱싱 + 그래프 추론으로 처리한다. 단순 텍스트 치환이 아니라 의존성 맥락을 이해한 구조적 변경이 가능하다.

## 사전 준비

- Claude Code 또는 Gemini CLI 설치
- 리팩토링 대상 레포 git 상태 클린 확인 (`git status`)
- 최소 50% 테스트 커버리지 (없으면 먼저 확보)
- CLAUDE.md에 아키텍처 제약사항 기재

## Step 1: 기술 부채 우선순위 분류

AI에게 코드베이스를 분석시켜 부채를 세 등급으로 나눈다.

```bash
# Claude Code에서 실행
claude -p "이 레포의 기술 부채를 다음 3등급으로 분류해줘:
- Critical: 보안 취약점, 테스트 없는 비즈니스 로직
- High: 중복 코드 30줄 이상, 순환 의존성
- Low: 네이밍 일관성, 불필요한 주석
각 파일별로 등급과 이유를 마크다운 테이블로 정리해줘"
```

| 등급 | 기준 | 처리 순서 |
|------|------|---------|
| Critical | 보안·무테스트 비즈니스 로직 | 즉시 처리 |
| High | 중복 코드, 순환 의존성 | 이번 스프린트 |
| Low | 네이밍, 주석 | 여유 있을 때 |

## Step 2: 안전한 리팩토링 패턴 선택

리팩토링 유형에 따라 AI 접근법이 다르다.

### 패턴 A — 추출 리팩토링 (Extract)

중복 로직을 공통 함수/모듈로 추출할 때 사용한다.

```bash
# 중복 코드 탐지 후 추출 요청
claude -p "src/ 폴더에서 30줄 이상 중복되는 코드 블록을 찾아서,
각각 별도 유틸 함수로 추출하고 기존 호출부를 교체해줘.
변경 전/후 diff를 먼저 보여준 다음 실제 편집해줘"
```

### 패턴 B — 스트랭글러 피그 (Strangler Fig)

레거시 모듈을 즉시 교체하지 않고 새 구현을 옆에 두고 점진적으로 트래픽을 넘긴다.

```
레거시 서비스 → 어댑터 레이어 → 신규 서비스
      ↑               ↓
    점진적 제거    점진적 확대
```

```bash
# 어댑터 레이어 생성
claude -p "OldUserService를 NewUserService로 교체하는 어댑터 클래스를 만들어줘.
인터페이스는 동일하게 유지하고, 내부 구현만 NewUserService로 위임해줘.
피처 플래그 USE_NEW_USER_SERVICE가 true일 때만 신규 서비스를 쓰도록 해줘"
```

### 패턴 C — 의존성 역전 리팩토링

구체 클래스에 직접 의존하는 코드를 인터페이스 의존으로 전환한다.

```bash
claude -p "UserRepository를 직접 의존하는 모든 서비스 클래스를 찾아서,
IUserRepository 인터페이스를 추출하고 생성자 주입으로 변경해줘.
테스트 파일에서 모킹이 쉬워지도록 인터페이스를 설계해줘"
```

## Step 3: 테스트 커버리지 확보 후 진행

리팩토링 전 반드시 기존 동작을 테스트로 고정한다.

```bash
# 테스트 없는 함수 자동 탐지 + 테스트 생성
claude -p "coverage 리포트에서 커버리지 0%인 함수 목록을 추출하고,
각 함수의 엣지 케이스(정상, 빈값, 예외)를 포함한 단위 테스트를 작성해줘.
실제 코드는 건드리지 말고 테스트 파일만 생성해줘"

# 테스트 실행 확인
npm test -- --coverage
```

리팩토링 후에도 동일 테스트가 통과해야 진행한다.

## Step 4: 멀티파일 리팩토링 — 의존성 순서 지키기

AI 에이전트가 여러 파일을 동시에 바꿀 때는 의존성 그래프 순서를 따른다.

```bash
# 의존성 그래프 시각화
claude -p "이 레포의 모듈 의존성 그래프를 mermaid 다이어그램으로 그려줘.
순환 의존성이 있으면 표시해줘"

# 올바른 순서로 리팩토링 계획 수립
claude -p "A → B → C 의존성 구조에서 A를 리팩토링하기 전에
B, C도 영향받는지 분석하고, 안전한 변경 순서를 알려줘"
```

| 원칙 | 이유 |
|------|------|
| 리프 모듈부터 변경 | 의존하는 모듈이 없어 안전 |
| 인터페이스 먼저 확정 | 구현 전 계약 고정 |
| 한 PR에 하나의 관심사 | 리뷰 용이, 롤백 가능 |

## Step 5: 팀 합의 프로세스

리팩토링은 혼자 하면 안 된다. AI가 제안한 변경사항은 팀과 합의해야 한다.

```markdown
## 리팩토링 제안 문서 템플릿

### 배경
기존 코드의 문제점 + 기술 부채 규모

### 변경 범위
영향받는 파일 목록 + 변경 유형

### 위험도
- 사이드이펙트 가능성
- 롤백 계획

### 검증 방법
- 테스트 커버리지 변화
- 성능 영향 측정 방법
```

## 흔한 실수와 해결

| 실수 | 결과 | 해결 |
|------|------|------|
| 테스트 없이 리팩토링 | 버그 감지 불가 | 테스트 먼저 작성 |
| 너무 큰 PR | 리뷰 불가 | 파일 5개 이하로 분리 |
| AI 결과물 무검토 | 숨겨진 버그 | 반드시 diff 검토 |
| 프로덕션에 직접 배포 | 장애 위험 | 스테이징 먼저 확인 |

## 실전 워크플로우 요약

```
1. 부채 분류 → Critical/High/Low 등급 매핑
2. 테스트 확보 → 리팩토링 전 동작 고정
3. 브랜치 생성 → refactor/[scope]-[description]
4. AI 리팩토링 → 패턴 선택 후 단계적 실행
5. 검증 → 테스트 통과 + 성능 비교
6. 팀 리뷰 → PR + 변경 설명 문서
7. 머지 → 스테이징 → 프로덕션
```

## 체크리스트

- [ ] 리팩토링 범위 결정 (파일 단위)
- [ ] 기존 테스트 커버리지 확인
- [ ] 부채 등급 분류 완료
- [ ] AI 리팩토링 계획 문서화
- [ ] 변경 전 모든 테스트 통과 확인
- [ ] 리팩토링 후 테스트 재실행
- [ ] 팀 리뷰 요청 + PR 본문 작성
- [ ] 스테이징 환경 검증
