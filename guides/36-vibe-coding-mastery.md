# 가이드 36: 바이브 코딩 실전 마스터

> 자연어로 설명하면 AI가 코드를 짜준다. 그런데 "짜줘"만으로는 안 된다. 시니어 개발자가 바이브 코딩으로 실제 제품 수준의 결과물을 만드는 방법을 정리했다.

## 바이브 코딩이 뭔가요

Andrej Karpathy가 2025년에 소개한 개념이다. 핵심은 간단하다 — 코드를 직접 타이핑하는 대신, 자연어로 의도를 설명하고 AI가 구현하도록 하는 것이다. 2026년 기준으로 개발자의 절반 이상이 일상 업무에서 어떤 형태로든 바이브 코딩을 하고 있다.

하지만 "그냥 말로 시키는 것"과 "체계적으로 바이브 코딩하는 것"은 결과가 완전히 다르다.

| 접근 방식 | 결과 품질 | 디버깅 시간 |
|-----------|----------|------------|
| 한 번에 전체 앱 요청 | 낮음 | 매우 김 |
| 기능 단위로 분할 요청 | 중간 | 보통 |
| 구조 설계 → 단계별 구현 | 높음 | 짧음 |

## Phase 1: 프로젝트 설계 먼저

코드 작성을 시작하기 전에 **구조를 잡는 시간**이 필요하다. 이 단계를 건너뛰면 나중에 전체를 다시 만드는 일이 생긴다.

### 1-1. 프로젝트 스펙 문서 작성

AI에게 바로 코드를 시키지 말고, 먼저 스펙 문서를 함께 만든다.

```
프롬프트 예시:
"이런 앱을 만들려고 해. [요구사항 설명]
코드 작성은 하지 말고, 먼저 기술 스펙 문서를 만들어줘.
- 디렉토리 구조
- 주요 컴포넌트 목록
- 데이터 모델
- API 엔드포인트 목록
을 포함해줘."
```

이렇게 하면 AI가 구조를 제안하고, 여기서 수정할 부분을 잡은 뒤 코드 작성으로 넘어가면 된다.

### 1-2. CLAUDE.md / .cursorrules 활용

프로젝트 루트에 AI 설정 파일을 먼저 만들어두면 모든 후속 작업의 품질이 올라간다.

```markdown
# CLAUDE.md 예시

## 프로젝트 개요
Next.js 15 + TypeScript + Supabase 풀스택 SaaS

## 코딩 규칙
- 컴포넌트는 함수형, Props 타입은 인터페이스로 분리
- API Route는 /app/api/ 하위에 Route Handler로 구현
- 에러 핸들링은 try-catch + 사용자 친화적 메시지
- 테스트: Vitest + Testing Library

## 디렉토리 구조
src/
  app/        # Next.js App Router
  components/ # UI 컴포넌트
  lib/        # 유틸리티, 타입, 훅
  server/     # 서버 전용 로직
```

## Phase 2: 점진적 구현 패턴

### 2-1. 스켈레톤 → 살 붙이기

한 번에 완성된 코드를 요청하지 않는다. 대신 뼈대를 먼저 만들고, 기능을 하나씩 추가한다.

```
Step 1: "프로젝트 초기화하고 기본 디렉토리 구조만 만들어줘"
Step 2: "메인 레이아웃 컴포넌트를 만들어줘. 헤더, 사이드바, 메인 영역"
Step 3: "사이드바에 네비게이션 메뉴를 추가해줘. 대시보드, 설정, 프로필 3개"
Step 4: "대시보드 페이지에 카드 3개짜리 요약 섹션을 만들어줘"
```

각 단계에서 결과를 확인하고, 문제가 있으면 바로 수정한 뒤 다음으로 넘어간다.

### 2-2. 체크포인트 전략

바이브 코딩에서 가장 중요한 습관은 **자주 커밋하는 것**이다.

```bash
# 기능 하나 완성될 때마다
git add -A && git commit -m "feat: add dashboard summary cards"

# 큰 리팩토링 전에는 반드시 브랜치 분리
git checkout -b experiment/new-layout
```

AI가 예상과 다른 방향으로 코드를 바꿨을 때 `git diff`로 확인하고, 문제가 있으면 `git checkout .`으로 되돌릴 수 있다.

| 시점 | 행동 |
|------|------|
| 기능 1개 완성 | `git commit` |
| 큰 변경 시작 전 | `git checkout -b` |
| AI 응답이 이상할 때 | `git diff` → `git checkout .` |
| 프로토타입 완성 | 태그 생성: `git tag v0.1-prototype` |

## Phase 3: 프롬프트 작성 패턴

### 3-1. 컨텍스트-태스크-제약 (CTC) 패턴

효과적인 프롬프트는 세 가지 요소를 명확히 분리한다.

```
[컨텍스트] Next.js 15 App Router + TypeScript 프로젝트에서
[태스크] 사용자 인증 미들웨어를 만들어줘.
[제약] Supabase Auth를 사용하고,
       /api/public/** 경로는 인증 없이 접근 가능하게,
       인증 실패 시 /login으로 리다이렉트해줘.
```

### 3-2. 레퍼런스 기반 프롬프팅

기존 코드를 참조 포인트로 사용하면 일관성이 올라간다.

```
"src/components/UserCard.tsx 스타일과 동일하게
ProductCard 컴포넌트를 만들어줘.
- Props 구조도 비슷하게
- Tailwind 클래스도 같은 패턴으로"
```

### 3-3. 네거티브 프롬프팅

원하지 않는 것을 명시하면 불필요한 수정 반복을 줄일 수 있다.

```
"로그인 폼을 만들어줘.
- CSS 모듈은 사용하지 말고 Tailwind만 써줘
- 서드파티 폼 라이브러리 없이 네이티브 form으로
- alert() 대신 토스트 메시지로 에러를 보여줘"
```

## Phase 4: 품질 관리

### 4-1. AI 생성 코드 리뷰 체크리스트

AI가 만든 코드를 그대로 쓰면 안 된다. 매번 아래 항목을 확인한다.

```
- [ ] 하드코딩된 값이 환경변수로 분리되어 있는가
- [ ] 에러 핸들링이 빠진 곳은 없는가
- [ ] 타입이 any로 빠져있는 곳은 없는가
- [ ] 불필요한 의존성을 추가하지 않았는가
- [ ] 기존 코드와 네이밍 컨벤션이 일치하는가
- [ ] 보안 취약점 (SQL injection, XSS 등)이 없는가
```

### 4-2. 테스트 주도 바이브 코딩

기능 코드를 요청하기 전에 테스트부터 만들면 품질이 크게 올라간다.

```
Step 1: "이 API의 테스트 코드를 먼저 작성해줘. 성공, 실패, 엣지 케이스 포함"
Step 2: "이 테스트가 통과하도록 실제 API를 구현해줘"
Step 3: "테스트 실행해서 결과 확인해줘"
```

이 패턴은 AI가 테스트 가능한 구조로 코드를 작성하게 유도한다.

### 4-3. 반복 디버깅 패턴

AI가 만든 코드에서 버그가 나오면:

```
"이 에러가 발생해: [에러 메시지]
관련 파일: [파일 경로]
내가 기대한 동작: [설명]
실제 동작: [설명]

원인을 분석하고 수정해줘."
```

에러 메시지 + 기대/실제 동작을 함께 주면 AI가 정확한 원인을 찾을 확률이 높아진다.

## Phase 5: 시나리오별 사용 전략

### 5-1. 프로토타입 (빠르게 만들기)

```
목표: 2시간 안에 동작하는 데모
전략: 완성도보다 속도 우선
도구: Cursor / Bolt / Lovable (시각적 결과 빠름)
주의: 프로토타입 코드를 프로덕션에 올리지 않기
```

### 5-2. 기능 추가 (기존 코드베이스)

```
목표: 기존 코드 스타일에 맞는 새 기능
전략: 기존 파일을 @참조하고, 컨벤션 따라달라고 명시
도구: Claude Code / Cursor (컨텍스트 이해 우수)
주의: 기존 코드를 깨뜨리지 않는지 테스트로 확인
```

### 5-3. 리팩토링 (기존 코드 개선)

```
목표: 구조 개선, 기능 변경 없음
전략: 작은 단위로 분할, 매 단계 테스트 실행
도구: Claude Code (대규모 파일 변경에 강함)
주의: 한 번에 너무 많이 바꾸지 않기 (3파일 이하)
```

| 시나리오 | 추천 도구 | 프롬프트 단위 | 커밋 빈도 |
|---------|----------|-------------|----------|
| 프로토타입 | Cursor, Bolt | 기능 단위 | 완성 시 |
| 기능 추가 | Claude Code | 파일 단위 | 매 기능 |
| 리팩토링 | Claude Code | 함수 단위 | 매 변경 |
| 버그 수정 | Copilot Chat | 에러 단위 | 즉시 |

## 흔한 실수와 해결법

| 실수 | 증상 | 해결 |
|------|------|------|
| 한 번에 전체 앱 요청 | 동작하지 않는 코드 덩어리 | 기능별로 나누어 요청 |
| 리뷰 없이 수락 | 숨은 버그, 보안 취약점 | 매번 diff 확인 |
| 컨텍스트 미제공 | 프로젝트 스타일과 맞지 않는 코드 | CLAUDE.md 설정 |
| 롤백 포인트 미확보 | AI가 기존 코드 덮어쓰기 | 자주 커밋 |
| 프롬프트 반복 수정 | 시간 낭비, 일관성 저하 | CTC 패턴 사용 |
| 테스트 미작성 | 리팩토링 시 기능 파손 | TDD 방식 적용 |

## 정리

바이브 코딩은 "코딩을 안 하는 것"이 아니라 **"코딩의 추상화 레벨을 올리는 것"**이다. 타이핑 대신 설계와 검증에 시간을 쓰고, 반복적인 구현은 AI에 맡기는 방식이다.

핵심을 세 줄로 요약하면:
1. **설계 먼저** — 코드 요청 전에 스펙과 구조를 잡는다
2. **작게 나눠서** — 한 번에 하나씩, 매번 확인하고 커밋한다
3. **사람이 검증** — AI가 만든 코드를 그대로 쓰지 않는다
