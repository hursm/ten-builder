# AI 에이전트 프롬프트 디버깅 치트시트

> AI 에이전트가 예상과 다른 결과를 낼 때 — 원인 유형을 파악하고 프롬프트를 바로잡는 패턴 모음

## 증상으로 원인 찾기

| 증상 | 가능한 원인 | 섹션 |
|------|-----------|------|
| 엉뚱한 파일을 수정함 | 지시 범위 불명확 | A |
| 존재하지 않는 함수/패키지 사용 | 컨텍스트 누락 → 환각 | B |
| 처음 몇 번은 맞다가 점점 이상해짐 | 컨텍스트 누수 | C |
| 요청과 반대로 동작 | 모호한 지시 | D |
| 이전 태스크 내용이 섞임 | 세션 오염 | E |

---

## A. 지시 범위 불명확 — 엉뚱한 파일을 수정할 때

**원인:** "수정해줘"처럼 범위를 지정하지 않으면, 에이전트가 관련 있어 보이는 파일을 자의적으로 선택해요.

```
❌ "로그인 버튼 스타일 수정해줘"
✅ "src/components/LoginButton.tsx 파일의 버튼 배경색만 blue로 변경해줘.
    다른 파일은 건드리지 마."
```

**범위 제한 3요소:**

| 요소 | 예시 |
|------|------|
| 대상 파일 명시 | `src/auth/login.ts만` |
| 변경 범위 | `handleSubmit 함수 내부만` |
| 금지 사항 | `테스트 파일 변경 금지` |

```bash
# 변경된 파일 범위 즉시 확인
git diff --name-only
```

---

## B. 환각 — 없는 함수·패키지를 만들어 쓸 때

**원인:** 에이전트의 학습 데이터에 해당 API의 최신 스펙이 없거나, 컨텍스트에 참고할 문서가 없을 때 그럴듯하게 지어내요.

**대응 패턴:**

```
# 라이브러리 버전을 프롬프트에 명시
"Next.js 15 App Router 기준으로 작성해줘.
 Pages Router 패턴(getServerSideProps 등)은 사용하지 마."

# 공식 문서 스니펫 직접 붙여넣기
"아래 공식 문서 예제를 참고해서 작성해줘:
 [문서 내용 붙여넣기]"
```

**환각 유형별 확인법:**

| 유형 | 확인 명령 |
|------|---------|
| 없는 npm 패키지 | `npm view <패키지명> version` |
| 없는 함수 | `grep -r "functionName" node_modules/<pkg>/dist/` |
| 잘못된 API 엔드포인트 | `curl -I https://api.example.com/path` |

---

## C. 컨텍스트 누수 — 점점 이상해질 때

**원인:** 세션이 길어질수록 이전 대화가 쌓이고, 에이전트가 초기 지시보다 최근 대화 흐름에 끌려요. 컨텍스트 윈도우 끝부분에서 성능이 저하되는 현상이에요.

```
# 징후: 처음 요청한 "Node.js 프로젝트"가 어느 순간 Python 코드로 바뀜
# 징후: "간단하게 만들어줘"라고 했는데 갈수록 복잡해짐
```

**대응 방법:**

```
1. 핵심 제약을 매 요청에 반복
   "앞서 말한 대로, Next.js만 사용해. (Python, Django 사용 금지)"

2. 주요 결정사항 주기적으로 요약 요청
   "지금까지 결정된 내용을 3줄로 요약해줘."

3. 세션 분리 기준 (Claude Code 기준)
   - 다른 기능/컴포넌트 작업 시작 → /clear
   - 컨텍스트 50% 이상 소모 → 새 세션
```

| 컨텍스트 상태 | 권장 액션 |
|------------|----------|
| 초기 (0~30%) | 자유롭게 진행 |
| 중반 (30~60%) | 핵심 제약 재명시 |
| 후반 (60~80%) | 요약 후 새 세션 준비 |
| 포화 (80% 이상) | 즉시 새 세션으로 이전 |

---

## D. 모호한 지시 — 요청과 반대로 동작할 때

**원인:** "더 좋게", "최적화해줘"처럼 기준이 없는 지시는 에이전트가 자체 기준으로 해석해요.

```
❌ "이 쿼리 최적화해줘"
✅ "이 쿼리 실행 시간을 현재 200ms에서 50ms 이하로 줄여줘.
    인덱스 추가는 허용, 테이블 구조 변경은 금지."

❌ "에러 처리 추가해줘"
✅ "fetchUser 함수에 try-catch 추가해줘.
    에러 시 null 반환, console.error로 에러 로깅.
    throw는 하지 마."
```

**지시 명확화 체크리스트:**

- [ ] 기대 결과를 수치/예시로 표현했나?
- [ ] 허용 범위와 금지 사항을 명시했나?
- [ ] 성공 기준을 말했나?

---

## E. 세션 오염 — 이전 태스크 내용이 섞일 때

**원인:** 같은 세션에서 다른 성격의 작업을 연속으로 하면, 에이전트가 이전 작업의 맥락을 끌어와요.

```bash
# Claude Code에서 세션 초기화
/clear

# 또는 특정 파일만 되돌리고 재시작
git checkout -- src/contaminated-file.ts
```

**세션 분리 기준:**

| 상황 | 처리 |
|------|------|
| 다른 기능 작업 시작 | 새 세션 |
| 버그 수정 → 기능 추가 전환 | 새 세션 |
| 에러 3회 이상 반복 | 새 세션 + 다른 접근법 |
| 같은 기능 내 연속 수정 | 같은 세션 유지 가능 |

## 프롬프트 수리 템플릿

에이전트 결과가 틀렸을 때 바로 쓰는 교정 프롬프트 패턴이에요.

```
# 패턴 1: 에러 교정
"방금 결과가 틀렸어.
 기대한 것: [구체적 설명]
 실제 결과: [에이전트 출력]
 차이점: [무엇이 다른지]
 다시 시도해줘. [추가 제약]"

# 패턴 2: 환각 교정
"[함수명]은 존재하지 않아.
 공식 문서 확인 결과: [실제 API]
 이걸 사용해서 다시 작성해줘."

# 패턴 3: 범위 교정
"[파일A], [파일B]는 변경하면 안 돼.
 [파일C]만 수정해서 같은 결과를 만들어줘."
```

## 반복 실패 탈출 시나리오

| 반복 횟수 | 탈출 전략 |
|----------|----------|
| 2회 | 에러 메시지 전문 + 관련 파일 전체 제공 |
| 3회 | "지금까지 시도한 방법을 정리해줘" → 새 접근 요청 |
| 4회 이상 | 새 세션, 핵심 부분 수동 수정 후 나머지 위임 |
