# 스펙 주도 개발(Spec-Driven Development) 치트시트

> 바이브 코딩의 대안 — 코드를 먼저 짜지 말고 스펙을 먼저 써라

## 핵심 원칙

| 바이브 코딩 | 스펙 주도 개발 |
|------------|--------------|
| 모호한 프롬프트 → 즉시 코드 | 명확한 스펙 → 계획 → 코드 |
| AI가 방향을 결정 | 개발자가 방향을 결정 |
| 빠른 시작, 잦은 수정 | 느린 시작, 적은 수정 |
| 프로토타입에 적합 | 프로덕션에 적합 |

---

## 스펙 작성 템플릿

### 기능 스펙 (Feature Spec)

```markdown
## 기능: {기능명}

### 목적
{이 기능이 해결하는 문제 1-2문장}

### 입력/출력
- 입력: {파라미터, 타입, 예시}
- 출력: {반환값, 타입, 예시}

### 정상 동작
1. {케이스 1}
2. {케이스 2}

### 엣지 케이스
- {입력이 null인 경우}
- {범위 초과 시}
- {동시 요청 시}

### 제약 조건
- 응답 시간: {N}ms 이하
- 의존성: {서비스/라이브러리}
- 금지 사항: {하지 말아야 할 것}
```

---

## AI에게 넘기기 전 체크리스트

- [ ] 기능의 목적이 한 문장으로 정의됨
- [ ] 입력값 타입과 예시가 명시됨
- [ ] 성공 케이스 최소 2개 작성
- [ ] 실패/엣지 케이스 최소 2개 작성
- [ ] 성능 제약 조건 명시
- [ ] 수정하면 안 되는 파일/범위 지정

---

## 프롬프트 구조

### 스펙 주도 프롬프트 패턴

```
# 컨텍스트
{프로젝트 스택, 기존 패턴}

# 구현할 기능
{기능 스펙 내용}

# 제약 조건
- 기존 {파일명}의 패턴을 따를 것
- {라이브러리} 사용 금지
- 테스트 코드 포함

# 완료 기준
- [ ] {테스트 케이스 1} 통과
- [ ] {테스트 케이스 2} 통과
- [ ] 린트 오류 없음
```

## 단계별 스펙 주도 워크플로우

### 1단계: 브레인스토밍 (AI와 함께)

```
"다음 기능을 구현하려고 해. 스펙 작성 전에 놓친 부분이나
엣지 케이스를 먼저 짚어줘: {기능 설명}"
```

### 2단계: 스펙 확정 (혼자)

- 브레인스토밍 결과를 바탕으로 스펙 문서 작성
- EARS 문법 활용: "When {조건}, the system shall {동작}"
- 완료 기준(Done Criteria) 반드시 포함

### 3단계: 구현 위임 (AI에게)

```
"위 스펙대로 구현해줘. 완료 기준을 모두 충족해야 해.
구현 전에 접근 방법을 먼저 설명해."
```

### 4단계: 검증 (혼자)

- 완료 기준 체크리스트 확인
- 엣지 케이스 직접 테스트
- 코드 리뷰 후 머지

---

## EARS 문법 (Easy Approach to Requirements Syntax)

| 패턴 | 문법 | 예시 |
|------|------|------|
| 이벤트 기반 | When {조건}, {시스템}은 {동작}한다 | 로그인 실패 3회 시, 계정을 30분 잠근다 |
| 상태 기반 | While {상태}, {시스템}은 {동작}한다 | 업로드 중, 다른 업로드 요청을 거부한다 |
| 옵션 기반 | Where {기능 포함 시}, {시스템}은 {동작}한다 | 알림 설정 시, 이메일을 발송한다 |
| 무조건 | {시스템}은 {동작}한다 | 시스템은 HTTPS만 허용한다 |

## 스펙 범위 설정

```bash
# CLAUDE.md에 명시하는 범위 예시
## 이번 작업 범위
- 수정 허용: src/api/user.ts, src/types/user.ts
- 수정 금지: src/db/*, src/auth/*
- 새 파일 생성 허용: src/api/user-*.ts

## 완료 기준
- [ ] GET /users/:id 200 응답 확인
- [ ] 존재하지 않는 ID → 404
- [ ] 권한 없는 요청 → 401
- [ ] npm test 전체 통과
```

## 흔한 실수 & 해결

| 실수 | 해결 |
|------|------|
| "이 기능 만들어줘"로 시작 | 스펙 템플릿 먼저 채우기 |
| AI가 다른 파일 건드림 | 수정 허용/금지 범위 명시 |
| 완료 기준 없이 리뷰 | Done Criteria 반드시 작성 |
| 한 번에 큰 기능 요청 | 기능을 독립 단위로 분해 |
| 스펙 없이 즉흥적 수정 | 변경이 크면 스펙 먼저 업데이트 |
