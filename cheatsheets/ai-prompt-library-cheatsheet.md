# AI 코딩 프롬프트 라이브러리 치트시트

> 팀에서 재사용할 수 있는 상황별 프롬프트 템플릿 모음 — 한 페이지 요약

## 왜 프롬프트 라이브러리가 필요한가?

같은 팀에서 AI 코딩 도구를 쓰면서 각자 다른 프롬프트를 쓰면, 결과물 품질이 들쭉날쭉해요. 자주 쓰는 프롬프트를 팀 라이브러리로 관리하면 일관성도 올라가고, 신규 입사자도 바로 쓸 수 있어요. 이 치트시트는 개발 상황별로 검증된 프롬프트 템플릿을 정리한 거예요.

## 프롬프트 라이브러리 구조

| 요소 | 설명 | 예시 |
|------|------|------|
| **카테고리** | 작업 유형 | 리팩토링, 테스트, 문서화, 디버깅 |
| **컨텍스트** | AI에게 전달할 배경 정보 | 파일 경로, 기술 스택, 제약 조건 |
| **지시** | 구체적으로 원하는 동작 | "함수 분리", "엣지케이스 추가" |
| **출력 형식** | 결과물 포맷 | 코드 블록, 표, 체크리스트 |

```
# 팀 프롬프트 기본 골격
[컨텍스트]: {기술 스택, 파일 위치, 관련 코드}
[작업]: {구체적 지시}
[제약]: {코딩 컨벤션, 금지 사항}
[출력]: {원하는 형식}
```

## 리팩토링 프롬프트

### 함수 분리 + 책임 정리

```
이 함수를 단일 책임 원칙에 맞게 분리해줘.

조건:
- 각 함수는 한 가지 일만 담당
- 함수명은 동작을 명확하게 표현 (예: validateInput, formatResponse)
- 원래 함수는 분리된 함수를 호출하는 오케스트레이터로 유지
- 기존 테스트가 깨지지 않도록 인터페이스 유지

변경 전/후를 나란히 보여주고, 각 분리 이유를 한 줄로 설명해줘.
```

### 중복 코드 제거

```
이 파일(또는 이 디렉토리)에서 중복 코드를 찾아줘.

기준:
- 3줄 이상 동일하거나 구조가 비슷한 블록
- 파라미터만 다른 반복 패턴
- 유틸 함수로 추출 가능한 공통 로직

각 중복에 대해:
1. 위치 (파일:라인)
2. 추출할 함수명 제안
3. 리팩토링 코드
```

### 네이밍 개선

```
이 파일의 변수명과 함수명을 리뷰해줘.

체크 항목:
- 축약어를 쓰는 대신 의도가 드러나는 이름인지 (e → event, cb → callback)
- boolean은 is/has/should 접두사를 쓰는지
- 함수는 동사로 시작하는지
- 일관된 네이밍 컨벤션을 따르는지

수정 제안을 표(현재 이름 → 제안 이름 → 이유)로 정리해줘.
```

## 테스트 프롬프트

### 유닛 테스트 생성

```
이 함수의 유닛 테스트를 작성해줘.

규칙:
- 테스트 프레임워크: {Jest/pytest/Go testing}
- 정상 케이스 2개 + 엣지케이스 3개 + 에러 케이스 2개
- 각 테스트 이름은 "should + 기대 동작"으로 작성
- 외부 의존성은 모킹
- AAA(Arrange-Act-Assert) 패턴 사용

테스트 커버리지가 부족한 분기가 있으면 따로 알려줘.
```

### 엣지케이스 발굴

```
이 함수가 처리해야 하는 엣지케이스를 빠짐없이 나열해줘.

카테고리:
- 빈 값: null, undefined, 빈 문자열, 빈 배열
- 경계값: 0, 음수, MAX_SAFE_INTEGER, 매우 긴 문자열
- 타입 불일치: 숫자 대신 문자열, 배열 대신 객체
- 동시성: 중복 호출, 순서 역전
- 외부 실패: 네트워크 타임아웃, 디스크 풀, 권한 없음

각 엣지케이스에 대한 테스트 코드도 함께 작성해줘.
```

### 통합 테스트 시나리오

```
이 API 엔드포인트의 통합 테스트를 작성해줘.

범위:
- HTTP 메서드, 헤더, 바디 전체 검증
- 인증된 요청 / 미인증 요청
- 유효한 입력 / 잘못된 입력 / 빠진 필드
- DB 상태 변경 확인 (before → after)
- 에러 응답 포맷 (status code + error message)

각 시나리오를 describe 블록으로 그룹핑해줘.
```

## 문서화 프롬프트

### README 생성

```
이 프로젝트의 README.md를 생성해줘.

포함할 섹션:
1. 프로젝트 한 줄 설명
2. 주요 기능 (bullet 3~5개)
3. 빠른 시작 (설치 → 설정 → 실행, 복사 가능한 명령어)
4. 환경 변수 표 (변수명 | 필수 | 설명 | 기본값)
5. 프로젝트 구조 (디렉토리 트리)
6. 개발 가이드 (로컬 실행, 테스트, 빌드)

톤: 간결하고 실용적. 불필요한 수식어 없이.
```

### 함수/클래스 JSDoc/docstring

```
이 파일의 모든 공개 함수(export/public)에 문서 주석을 추가해줘.

포맷: {JSDoc / Python docstring / Go godoc}
포함 항목:
- 함수 설명 (한 줄)
- @param 각 파라미터 타입과 설명
- @returns 반환값 타입과 설명
- @throws 발생 가능한 에러
- @example 사용 예시 1개

기존 코드는 변경하지 않고, 주석만 추가해줘.
```

### API 엔드포인트 문서

```
이 라우터 파일을 보고 API 문서를 작성해줘.

포맷: 마크다운 표
열: 메서드 | 경로 | 설명 | 요청 바디 | 응답 | 인증

각 엔드포인트마다:
- curl 요청 예시
- 성공/실패 응답 JSON 예시
- 필수 헤더 목록
```

## 디버깅 프롬프트

### 에러 원인 분석

```
이 에러를 분석해줘.

에러 메시지:
{에러 메시지 붙여넣기}

스택 트레이스:
{스택 트레이스 붙여넣기}

다음 순서로 답변해줘:
1. 에러의 직접 원인 (한 줄)
2. 왜 이 에러가 발생했는지 (코드 흐름 추적)
3. 수정 방법 (코드 포함)
4. 같은 패턴의 에러가 다른 곳에도 있는지 확인
```

### 성능 병목 탐색

```
이 함수(또는 API)가 느린 원인을 찾아줘.

확인 항목:
- N+1 쿼리 여부
- 불필요한 반복문 (O(n²) 이상)
- 큰 데이터를 메모리에 전부 로드하는지
- 캐싱 가능한 연산이 매번 실행되는지
- async/await 병렬 처리 기회 누락

각 병목에 대해:
- 현재 코드 vs 개선 코드
- 예상 개선 효과 (Big-O 또는 정성적)
```

### 로그 분석

```
이 로그를 분석해서 문제 원인을 찾아줘.

{로그 붙여넣기}

정리 형식:
1. 타임라인: 주요 이벤트를 시간순으로 정리
2. 이상 징후: 정상 패턴에서 벗어난 부분
3. 근본 원인: 가장 가능성 높은 원인
4. 조치 방안: 즉시 해결 + 재발 방지
```

## 코드 생성 프롬프트

### 새 기능 구현

```
다음 요구사항에 맞는 {함수/클래스/모듈}을 만들어줘.

요구사항:
- {기능 설명}
- 입력: {타입과 형태}
- 출력: {타입과 형태}
- 에러 처리: {예외 상황과 대응}

제약:
- 기술 스택: {언어/프레임워크}
- 기존 코드 컨벤션을 따를 것
- 외부 라이브러리 추가 없이 구현 (필수 시 이유 설명)
- 유닛 테스트 포함
```

### 설정 파일 생성

```
다음 조건에 맞는 {설정 파일명}을 생성해줘.

환경: {production/staging/development}
요구사항:
- {조건 1}
- {조건 2}
- {조건 3}

주석으로 각 설정의 용도와 변경 시 영향을 설명해줘.
보안에 민감한 값은 환경변수 참조로 처리해줘.
```

## 라이브러리 운영 팁

| 항목 | 실전 팁 |
|------|---------|
| **저장 위치** | 레포 `.prompts/` 폴더 또는 팀 위키에 카테고리별 정리 |
| **버전 관리** | Git으로 변경 이력 추적, PR로 수정 제안 |
| **네이밍** | `{카테고리}-{작업}.md` (예: `test-edge-cases.md`) |
| **변수 표기** | `{중괄호}`로 교체 가능한 부분 표시 |
| **효과 측정** | 수용률(AI 결과를 그대로 쓴 비율) 추적 |
| **갱신 주기** | 월 1회 리뷰 — 안 쓰는 프롬프트 정리, 새 패턴 추가 |
| **공유 방식** | IDE 스니펫 또는 슬래시 커맨드로 등록해서 즉시 호출 |

## 프롬프트 품질 체크리스트

사용하기 전에 이 기준을 확인하세요:

- [ ] **구체적인가?** — "코드 개선해줘" 대신 "함수 분리 + 네이밍 개선"
- [ ] **범위가 명확한가?** — 파일, 함수, PR 등 대상 지정
- [ ] **출력 형식을 지정했는가?** — 표, 코드 블록, 체크리스트
- [ ] **제약 조건이 있는가?** — 기존 인터페이스 유지, 라이브러리 제한
- [ ] **검증 가능한가?** — 결과물이 맞는지 테스트로 확인 가능
