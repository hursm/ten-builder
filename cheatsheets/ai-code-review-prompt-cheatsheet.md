# AI 코드 리뷰 프롬프트 패턴 치트시트

> AI 에이전트에 코드 리뷰를 맡길 때 쓸 수 있는 프롬프트 패턴 — 관점별 정리

## 왜 리뷰 프롬프트가 중요한가?

"코드 리뷰해줘"라고만 하면 AI는 뻔한 피드백만 돌려줘요. 리뷰 관점(보안, 성능, 아키텍처, 테스트)을 명확히 지정하면 실제로 쓸 만한 피드백을 받을 수 있어요. 이 치트시트는 상황별로 바로 복사해서 쓸 수 있는 프롬프트 패턴 모음이에요.

## 기본 구조: 리뷰 프롬프트 3요소

| 요소 | 설명 | 예시 |
|------|------|------|
| **관점** | 어떤 측면에서 볼 것인지 | 보안, 성능, 가독성, 테스트 |
| **범위** | 어디까지 볼 것인지 | 이 파일, 이 PR, 이 함수 |
| **출력 형식** | 피드백을 어떤 형태로 받을지 | 표, 심각도 라벨, 코드 제안 |

```
이 {범위}를 {관점} 관점에서 리뷰해줘.
피드백은 {출력 형식}으로 정리해줘.
```

## 보안 리뷰 프롬프트

### 입력 검증 + 인젝션

```
이 API 엔드포인트를 보안 관점에서 리뷰해줘.
특히 다음을 확인해줘:
1. 사용자 입력이 검증 없이 DB 쿼리에 들어가는 곳
2. SQL 인젝션, XSS, Command Injection 가능성
3. 파라미터 타입 검증 누락

각 이슈를 [심각/주의/개선] 등급으로 분류하고,
수정 코드를 함께 제시해줘.
```

### 인증 + 권한

```
이 인증 플로우를 리뷰해줘.
확인 항목:
- JWT 토큰 검증이 모든 보호 경로에 적용되는지
- 권한 체크가 리소스 소유자를 확인하는지 (IDOR)
- 토큰 만료/갱신 로직에 허점이 없는지
- 민감 데이터가 토큰 페이로드에 포함되지 않는지
```

### 의존성 + 시크릿

```
이 프로젝트의 보안 상태를 점검해줘.
1. package.json(또는 requirements.txt) 의존성 중 알려진 취약점
2. 하드코딩된 API 키, 비밀번호, 시크릿
3. .env 파일이 .gitignore에 포함되어 있는지
4. 환경변수 접근 패턴이 안전한지
```

## 성능 리뷰 프롬프트

### 쿼리 + 데이터 접근

```
이 코드의 데이터베이스 접근 패턴을 성능 관점에서 리뷰해줘.
확인 사항:
- N+1 쿼리 문제가 있는지
- 인덱스가 필요한 WHERE/ORDER BY 절
- 불필요한 SELECT * 사용
- 트랜잭션 범위가 적절한지

각 이슈에 예상 영향도(높음/중간/낮음)와 개선 코드를 달아줘.
```

### 메모리 + 연산

```
이 함수들의 메모리 사용과 연산 효율을 리뷰해줘.
특히:
- 큰 배열/객체의 불필요한 복사
- 루프 안에서 반복되는 비용이 큰 연산
- 캐싱할 수 있는데 매번 계산하는 부분
- 메모리 누수 가능성 (이벤트 리스너, 클로저)
```

### API 응답 최적화

```
이 API의 응답 성능을 개선할 방법을 찾아줘.
체크리스트:
- 응답 페이로드에 불필요한 필드가 포함되는지
- 페이지네이션이 적용되지 않은 목록 API
- 캐싱 헤더(ETag, Cache-Control) 설정 여부
- 병렬 처리할 수 있는 순차 호출
```

## 아키텍처 리뷰 프롬프트

### 의존성 + 결합도

```
이 모듈의 아키텍처를 리뷰해줘.
분석 항목:
1. 순환 의존성이 있는지
2. 상위 레이어가 하위 레이어에 의존하는 올바른 방향인지
3. 인터페이스 없이 직접 구현체에 의존하는 곳
4. 하나의 모듈이 너무 많은 책임을 가진 곳

의존성 그래프를 텍스트로 그려주고, 문제점마다 리팩토링 방향을 제안해줘.
```

### 확장성

```
이 시스템이 트래픽 10배 증가를 견딜 수 있는지 분석해줘.
고려 사항:
- 상태 저장(stateful) 컴포넌트가 있는지
- 수평 확장 시 문제가 될 부분
- 단일 장애점(SPOF)
- 비동기 처리가 필요한 동기 작업

각 병목에 대해 구체적인 해결 방안을 제시해줘.
```

### 패턴 일관성

```
이 코드베이스의 패턴 일관성을 리뷰해줘.
확인 항목:
- 에러 처리 방식이 통일되어 있는지 (try-catch vs Result 타입)
- 네이밍 컨벤션이 일관적인지
- 같은 문제를 다른 방식으로 해결한 곳
- 프로젝트 규칙(CLAUDE.md 등)과 실제 코드의 불일치

패턴이 깨진 곳을 목록으로 정리하고, 통일 방향을 제안해줘.
```

## 테스트 리뷰 프롬프트

### 커버리지 + 엣지케이스

```
이 모듈의 테스트를 리뷰해줘.
확인 사항:
1. 핵심 비즈니스 로직에 테스트가 있는지
2. 빠진 엣지케이스 (빈 입력, null, 경계값, 에러 상황)
3. 테스트가 구현 세부사항에 의존하는지 (깨지기 쉬운 테스트)
4. 테스트 이름이 의도를 명확히 표현하는지

누락된 테스트 케이스를 우선순위별로 정리하고,
가장 중요한 3개의 테스트 코드를 작성해줘.
```

### 모킹 + 격리

```
이 테스트 코드의 모킹 패턴을 리뷰해줘.
확인 사항:
- 외부 서비스 호출이 제대로 모킹되어 있는지
- 모킹 범위가 적절한지 (과도한 모킹 = 의미 없는 테스트)
- 테스트 간 상태 공유가 있는지 (격리 실패)
- 모킹으로 인해 실제 동작과 다른 결과를 내는 테스트
```

## 복합 리뷰 프롬프트

### PR 전체 리뷰 (만능 프롬프트)

```
이 PR을 종합적으로 리뷰해줘.

1단계 — 변경 요약:
변경된 파일과 핵심 변경사항을 3줄로 요약해줘.

2단계 — 이슈 분류:
| 파일 | 라인 | 심각도 | 카테고리 | 설명 | 제안 |
표 형식으로 정리해줘. 심각도는 [차단/주의/개선/칭찬] 4단계.

3단계 — 전체 판단:
머지해도 되는지, 수정이 필요한지 한 줄로 결론.
```

### 레거시 코드 이해 + 리뷰

```
이 파일은 문서가 없는 레거시 코드야.
두 가지를 해줘:

1. 이 코드가 무엇을 하는지 설명 (함수별 역할, 데이터 흐름)
2. 개선이 필요한 부분을 우선순위별로 정리
   - 당장 고쳐야 할 버그
   - 리팩토링하면 좋은 구조적 문제
   - 나중에 개선할 스타일 이슈
```

## 프롬프트 효과를 높이는 팁

| 팁 | 나쁜 예 | 좋은 예 |
|----|--------|--------|
| 관점을 명시 | "리뷰해줘" | "보안 관점에서 리뷰해줘" |
| 체크리스트 제공 | "문제 찾아줘" | "N+1 쿼리, 인덱스 누락, 트랜잭션 범위를 확인해줘" |
| 출력 형식 지정 | "알려줘" | "표로 정리하고 심각도 라벨을 달아줘" |
| 컨텍스트 추가 | "이 코드 봐줘" | "이건 결제 시스템이고 PCI DSS 준수가 필요해" |
| 코드 제안 요청 | "뭐가 문제야?" | "문제점과 수정 코드를 함께 보여줘" |
| 우선순위 요구 | "다 알려줘" | "가장 중요한 3가지만 골라줘" |

## Claude Code에서 바로 쓰기

```bash
# 파일 단위 보안 리뷰
claude "src/auth/login.ts를 보안 관점에서 리뷰해줘. 인젝션, 인증 우회, 시크릿 노출을 확인하고 심각도별로 정리해줘"

# git diff 기반 PR 리뷰
git diff main...HEAD | claude "이 변경사항을 리뷰해줘. 버그, 성능, 보안 순으로 이슈를 정리하고 심각도 라벨을 달아줘"

# 특정 관점만 빠르게
claude "이 폴더의 에러 처리 패턴이 일관적인지만 확인해줘. 불일치하는 곳을 목록으로"
```

## CLAUDE.md에 리뷰 규칙 넣기

```markdown
# 코드 리뷰 규칙

## 리뷰 시 항상 확인
- [ ] SQL 쿼리에 파라미터 바인딩 사용
- [ ] 사용자 입력 검증 (zod/joi 스키마)
- [ ] 에러 응답에 내부 정보 미포함
- [ ] 새 API에 테스트 존재

## 리뷰 출력 형식
심각도: blocker > warning > suggestion > praise
각 이슈에 파일명:라인 + 수정 제안 포함
```

프로젝트 규칙 파일에 리뷰 체크리스트를 넣어두면 "리뷰해줘"만 해도 일관된 기준으로 봐줘요.
