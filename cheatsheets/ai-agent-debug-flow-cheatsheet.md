# AI 코딩 에이전트 디버깅 플로우

> AI 에이전트가 에러를 만들었을 때 — 체계적으로 원인을 찾고 해결하는 5단계 플로우 요약

## 디버깅 5단계 플로우

```
에러 발생 → [1] 재현 → [2] 분류 → [3] 격리 → [4] 수정 지시 → [5] 검증
```

| 단계 | 핵심 질문 | 소요 시간 |
|------|----------|----------|
| 1. 재현 | "정확히 어떤 명령에서 실패했나?" | 1~2분 |
| 2. 분류 | "에이전트 문제인가, 환경 문제인가?" | 1분 |
| 3. 격리 | "어떤 파일, 어떤 라인이 문제인가?" | 2~5분 |
| 4. 수정 지시 | "에이전트에게 어떻게 다시 시킬까?" | 1~3분 |
| 5. 검증 | "같은 실수를 반복하지 않는가?" | 2분 |

## Step 1: 에러 재현

에이전트가 만든 에러를 그대로 재현하는 것이 첫 번째 단계예요. 재현 없이 수정하면 다른 문제를 만들 가능성이 높아요.

```bash
# 빌드 에러 재현
npm run build 2>&1 | tail -20

# 테스트 에러 재현
npm test -- --reporter=verbose 2>&1 | grep -A 5 "FAIL"

# 타입 에러 재현
npx tsc --noEmit 2>&1 | head -30
```

**포인트:** 에러 메시지를 전문 복사해두세요. 에이전트에게 다시 줄 때 필요해요.

## Step 2: 에러 분류 — 의사결정 트리

```
에러 발생
├── 빌드/컴파일 에러?
│   ├── import 경로 틀림 → [환경 문제] Step 3A로
│   ├── 타입 불일치 → [에이전트 문제] Step 3B로
│   └── 없는 패키지 참조 → [환각] Step 3C로
├── 런타임 에러?
│   ├── undefined/null 참조 → [로직 문제] Step 3B로
│   └── API 응답 형식 다름 → [환각] Step 3C로
└── 테스트 실패?
    ├── 기존 테스트 깨짐 → [사이드 이펙트] Step 3D로
    └── 새 테스트 자체가 틀림 → [에이전트 문제] Step 3B로
```

| 분류 | 의미 | 빈도 |
|------|------|------|
| 환경 문제 | 프로젝트 구조를 잘못 파악 | 30% |
| 에이전트 문제 | 로직/구현 실수 | 35% |
| 환각 | 존재하지 않는 API/패키지 사용 | 20% |
| 사이드 이펙트 | 다른 코드에 영향 | 15% |

## Step 3: 원인 격리

### 3A. 환경 문제 격리

```bash
# 프로젝트 구조 확인 — 에이전트가 파악한 구조가 맞는지
find . -name "*.ts" -not -path "./node_modules/*" | head -20

# import 경로 실제 확인
ls -la src/components/  # 에이전트가 참조한 경로가 존재하는지
```

### 3B. 로직 문제 격리

```bash
# git diff로 에이전트가 변경한 부분만 확인
git diff --stat HEAD~1
git diff HEAD~1 -- src/target-file.ts

# 변경 전/후 비교
git stash  # 에이전트 변경 임시 제거
npm test   # 원래 코드로 테스트
git stash pop
```

### 3C. 환각 격리

```bash
# 패키지 실재 여부 확인
npm view <package-name> version 2>&1

# API 엔드포인트 확인
curl -s -o /dev/null -w "%{http_code}" https://api.example.com/endpoint
```

### 3D. 사이드 이펙트 격리

```bash
# 에이전트 변경 파일 목록
git diff --name-only HEAD~1

# 각 파일별 영향 범위 확인
grep -r "import.*targetFile" src/ --include="*.ts"
```

## Step 4: 수정 지시 패턴

분류별로 에이전트에게 다시 지시하는 방법이 달라요.

| 분류 | 지시 패턴 | 예시 프롬프트 |
|------|----------|-------------|
| 환경 문제 | 구조 정보 명시 | `프로젝트 구조를 확인해. src/components/ 아래에 Button.tsx가 있어` |
| 로직 문제 | 에러 + 기대 동작 명시 | `이 함수가 null을 반환하는데, 빈 배열 []을 반환해야 해. 에러: {붙여넣기}` |
| 환각 | 실제 API 문서 제공 | `이 패키지는 존재하지 않아. 대신 {실제 패키지}를 사용해. 공식 문서: {URL}` |
| 사이드 이펙트 | 영향 범위 제한 | `{파일명}만 수정해. 다른 파일은 건드리지 마. 기존 테스트가 깨지지 않게 해` |

**효과적인 수정 지시의 3요소:**

```
1. 에러 메시지 전문 (복사-붙여넣기)
2. 기대하는 동작 (구체적으로)
3. 제약 조건 ("이 파일만", "이 함수만", "기존 테스트 통과")
```

## Step 5: 수정 후 검증 체크리스트

에이전트가 수정을 완료하면 아래 순서로 검증하세요:

```bash
# 1. 원래 에러 해결 확인
npm run build  # 빌드 통과?
npm test       # 테스트 통과?

# 2. 사이드 이펙트 확인
git diff --stat  # 예상 외 파일 변경 없는지

# 3. 타입 체크
npx tsc --noEmit  # 타입 에러 없는지

# 4. 린트 체크
npm run lint  # 코드 스타일 문제 없는지
```

| 검증 항목 | 통과 기준 | 실패 시 |
|----------|----------|--------|
| 빌드 | exit code 0 | Step 3으로 돌아가서 재격리 |
| 테스트 | 기존 + 새 테스트 모두 통과 | 실패 테스트만 에이전트에게 전달 |
| 타입 | 에러 0개 | 에러 메시지 복사 → 재수정 지시 |
| 변경 범위 | 요청한 파일만 변경 | 불필요한 변경 revert 후 재커밋 |

## 반복 실패 시 탈출 전략

같은 에러를 2회 이상 반복하면 접근 방식을 바꿔야 해요:

| 반복 횟수 | 전략 |
|----------|------|
| 2회 | 에러 컨텍스트를 더 자세히 제공 (관련 파일 전체 포함) |
| 3회 | 에이전트에게 "지금까지 시도한 방법을 정리해줘" → 새 접근 요청 |
| 4회 이상 | 새 세션 시작, 또는 수동으로 핵심 부분만 수정 후 나머지 위임 |

```bash
# 새 세션에서 깨끗하게 시작
# (Claude Code 기준)
/clear

# 또는 문제 파일만 수동 수정 후 나머지 위임
git checkout -- src/problem-file.ts  # 에이전트 변경 되돌리기
# 직접 수정 후
# 나머지 작업만 에이전트에게 위임
```
