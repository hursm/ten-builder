# 플레이북 44: AI 에이전트 테스트 주도 개발(TDD) 워크플로우

> 테스트를 먼저 써달라고 하면 AI가 더 좋은 코드를 만든다 — 실전에서 검증된 AI TDD 패턴

## 소요 시간

15-30분 (초기 설정), 이후 기능당 5-10분

## 사전 준비

- Claude Code, Cursor, 또는 터미널 AI 에이전트 설치
- 테스트 프레임워크 설정 (Jest, Pytest, Go test 등)
- 기본 프로젝트 구조 및 CLAUDE.md 또는 에이전트 지시 파일

## 핵심 원칙: 왜 AI TDD가 다른가

일반적인 AI 코딩 방식은 "이 기능 구현해줘"로 시작한다. AI TDD는 반대 순서다:

1. **스펙 먼저** — 무엇을 만들지 자연어로 정의
2. **테스트 먼저** — AI가 실패하는 테스트를 작성
3. **구현 나중** — AI가 테스트를 통과하는 코드를 작성
4. **반복** — 테스트 실행 후 리팩터링

이 순서를 지키면 AI가 만들어내는 코드가 훨씬 명확하고, 범위가 제한되고, 실제로 검증된다.

## Step 1: 스펙 문서 작성

기능 구현 전에 SPEC.md 또는 인라인 주석으로 요구사항을 정의한다.

```markdown
# SPEC: 사용자 인증 모듈

## 기능 요구사항
- 이메일/비밀번호로 회원가입
- JWT 토큰 발급 (만료: 24시간)
- 중복 이메일 가입 시 에러 반환
- 비밀번호: 최소 8자, 영문+숫자 포함

## 에러 케이스
- 잘못된 이메일 형식 → 400 Bad Request
- 중복 이메일 → 409 Conflict
- 비밀번호 정책 미충족 → 422 Unprocessable Entity
```

**AI 프롬프트:**
```
SPEC.md를 읽고, 구현 전에 실패하는 테스트 케이스를 먼저 작성해줘.
긍정 케이스, 부정 케이스, 엣지 케이스를 모두 포함할 것.
테스트만 작성하고, 구현 코드는 아직 작성하지 마.
```

## Step 2: 테스트 파일 생성

AI가 생성한 테스트를 검토하고 실행해서 "전부 실패"하는지 확인한다.

```bash
# Jest (TypeScript/JavaScript)
npx jest --testPathPattern="auth" --verbose 2>&1 | tail -20

# Pytest (Python)
python -m pytest tests/test_auth.py -v 2>&1 | tail -20

# Go
go test ./... -run TestAuth -v 2>&1 | tail -20
```

예상 결과: **모든 테스트 실패** (구현이 없으므로 정상)

```
FAIL  tests/auth.test.ts
  ✗ should register user with valid data
  ✗ should reject duplicate email
  ✗ should reject invalid password
```

> **체크포인트:** 테스트가 맞는 이유로 실패하는지 확인한다. "모듈을 찾을 수 없음"이 아니라 "함수가 존재하지 않음"이어야 한다.

## Step 3: 최소 구현

테스트를 통과시키는 최소한의 코드만 작성하도록 AI에 지시한다.

**AI 프롬프트:**
```
이제 테스트 파일을 기준으로 최소 구현을 작성해줘.
테스트를 먼저 실행하고, 실패하는 테스트부터 하나씩 통과시킬 것.
각 단계마다 테스트를 실행해서 결과를 확인해줘.
```

이 방식의 장점:

| 일반 AI 코딩 | AI TDD |
|-------------|--------|
| 한 번에 많은 코드 생성 | 테스트 단위로 점진적 구현 |
| 검증 없이 완성 선언 | 각 단계마다 실행 확인 |
| 범위가 자주 초과됨 | 스펙 범위만 구현 |
| 리팩터링 두려움 | 테스트가 안전망 역할 |

## Step 4: 테스트 실행 + 피드백 루프

AI에게 테스트 실행과 실패 해결을 자율적으로 반복하도록 지시한다.

```
테스트를 실행하고, 실패하는 테스트가 있으면 코드를 수정해서 통과시켜줘.
모든 테스트가 통과할 때까지 반복할 것.
새로운 테스트 케이스가 필요하다고 생각되면 추가해도 돼.
```

실제 실행 예시:

```bash
# AI가 자동으로 실행하는 루프 패턴
npm test -- --watch
# 또는
pytest -x --tb=short  # 첫 번째 실패에서 멈춤
```

## Step 5: 커버리지 확인 + 리팩터링

모든 테스트 통과 후, AI에게 커버리지와 코드 품질을 검토하도록 요청한다.

```bash
# Jest 커버리지
npx jest --coverage --coverageThreshold='{"global":{"lines":80}}'

# Pytest 커버리지
python -m pytest --cov=src --cov-report=term-missing --cov-fail-under=80

# Go 커버리지
go test ./... -coverprofile=coverage.out
go tool cover -html=coverage.out
```

**AI 프롬프트:**
```
커버리지 리포트를 보고:
1. 테스트되지 않은 중요한 경로가 있으면 테스트 추가
2. 중복 코드나 복잡도 높은 부분을 리팩터링
3. 리팩터링 후 테스트가 여전히 통과하는지 확인
```

## Step 6: CLAUDE.md에 TDD 규칙 추가

프로젝트의 CLAUDE.md 또는 에이전트 지시 파일에 TDD 규칙을 명시한다.

```markdown
## 개발 규칙

### 테스트 주도 개발 필수
1. 새 기능 구현 전에 반드시 테스트 파일을 먼저 작성할 것
2. 구현 코드는 실패하는 테스트를 통과시키는 최소한만 작성
3. 각 구현 단계마다 `npm test` 또는 `pytest` 실행하여 확인
4. 커버리지 80% 미만이면 추가 테스트 작성 후 PR 생성

### 금지 사항
- 테스트 없이 구현 코드 작성 금지
- 모든 테스트가 통과하기 전 다음 기능으로 넘어가기 금지
```

## 체크리스트

- [ ] SPEC.md 또는 인라인 요구사항 문서 작성
- [ ] AI가 테스트를 먼저 작성했는지 확인
- [ ] 테스트가 "올바른 이유로" 실패하는지 확인
- [ ] AI가 테스트 실행 후 구현했는지 확인
- [ ] 커버리지 80% 이상 달성
- [ ] 리팩터링 후 모든 테스트 통과 확인
- [ ] CLAUDE.md에 TDD 규칙 추가

## 흔한 실수

| 실수 | 해결 |
|------|------|
| "테스트도 같이 작성해줘"라고 지시 | 반드시 "테스트를 먼저 작성하고, 구현은 나중에"로 분리 |
| 테스트 실패 확인 없이 진행 | 테스트 실패 스크린샷 또는 로그 확인 필수 |
| 너무 세밀한 테스트 요청 | 스펙 수준의 테스트 먼저, 세부 테스트는 반복 중 추가 |
| AI가 테스트를 건너뜀 | CLAUDE.md에 TDD 규칙 명시로 방지 |
