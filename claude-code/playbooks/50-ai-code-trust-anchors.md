# 플레이북 50: AI 코드 신뢰 앵커 설정

> 팀이 AI 생성 코드를 안심하고 배포하기 위한 신뢰 기준선 수립 — 리뷰 체크포인트, 자동화 게이트, 팀 합의 문서화

## 소요 시간

30-60분 (초기 기준선 설정 기준, 이후 유지보수 5-10분/월)

## 사전 준비

- 팀에서 AI 코딩 도구를 이미 사용 중 (Claude Code, Cursor, Copilot 등)
- GitHub 또는 GitLab PR 기반 워크플로우
- CI/CD 파이프라인 (GitHub Actions, GitLab CI 등)

## 왜 신뢰 앵커가 필요한가

2026년 4월 기준, 개발자의 84%가 AI 코딩 도구를 사용하고 있지만 29%만이 프로덕션 배포에 대해 신뢰한다는 설문 결과가 있습니다. 이 격차의 원인은 AI 도구 자체의 품질 문제가 아닙니다. **팀이 공유하는 신뢰 기준선이 없기 때문입니다.**

신뢰 앵커(Trust Anchor)는 팀이 합의한 체크포인트입니다:

- AI 생성 코드가 어느 기준을 통과하면 PR 머지가 허용되는가
- 어떤 범주의 코드는 반드시 사람이 검토해야 하는가
- 자동화 게이트가 커버하지 못하는 영역은 무엇인가

기준선 없이 운영하면 팀마다 다른 기준으로 리뷰하게 되고, "이 코드 AI가 짠 거잖아요"라는 막연한 불신이 쌓입니다.

## 핵심 개념: 신뢰 3계층

| 계층 | 커버 영역 | 도구/방법 |
|------|----------|----------|
| 자동 게이트 | 정적 분석, 보안 스캔, 테스트 | CI/CD + Linter + SAST |
| 리뷰 체크포인트 | 로직 정확성, 사이드 이펙트 | PR 리뷰 체크리스트 |
| 팀 합의 문서 | 도구 정책, 금지 구역 | TRUST_POLICY.md |

## Step 1: 자동 게이트 구성

CI에서 AI 생성 코드가 통과해야 할 최소 게이트를 설정합니다.

**`.github/workflows/ai-code-gate.yml` 예시:**

```yaml
name: AI Code Gate

on: [pull_request]

jobs:
  gate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      # 정적 분석
      - name: Lint
        run: npx eslint src/ --max-warnings 0

      # 보안 스캔 (하드코딩된 시크릿, 취약 패턴)
      - name: Security scan
        run: |
          pip install semgrep
          semgrep --config=auto --error src/

      # 단위 테스트 + 커버리지
      - name: Test
        run: |
          npm test -- --coverage --coverageThreshold='{"global":{"lines":80}}'

      # 타입 체크
      - name: Type check
        run: npx tsc --noEmit
```

**핵심 원칙:** 게이트는 엄격하게, 체크리스트는 맥락에 맞게.

자동화로 커버되는 영역에서 사람이 반복 검토하는 것은 낭비입니다. 자동 게이트가 통과된 코드는 리뷰어가 "이 로직이 의도한 대로 동작하는가"에 집중할 수 있습니다.

## Step 2: PR 리뷰 체크리스트 작성

팀이 합의한 리뷰 기준을 문서화합니다. AI 생성 코드 비율이 높을수록 체크리스트가 중요합니다.

**`.github/pull_request_template.md` 추가 섹션:**

```markdown
## AI 코드 리뷰 체크포인트

### 자동 게이트 확인
- [ ] CI 모든 체크 통과 (lint, test, security scan)
- [ ] 커버리지 기준 유지 (80% 이상)

### 로직 검토
- [ ] 함수/메서드의 실제 동작이 PR 설명과 일치하는가
- [ ] 엣지 케이스가 처리되어 있는가 (null, 빈 배열, 타임아웃 등)
- [ ] 에러 핸들링이 적절한가

### 맥락 적합성
- [ ] 기존 코드베이스 패턴과 일관된가
- [ ] 불필요한 의존성이 추가되지 않았는가
- [ ] 로그/모니터링 포인트가 충분한가

### 고위험 영역 (해당 시 필수 2인 리뷰)
- [ ] 인증/인가 로직
- [ ] 결제/금융 데이터 처리
- [ ] 외부 API 호출 + 재시도 로직
- [ ] 데이터베이스 마이그레이션
```

**"고위험 영역"의 기준은 팀이 직접 결정합니다.** 스타트업과 금융 서비스 팀의 기준은 달라야 합니다.

## Step 3: TRUST_POLICY.md 작성

팀 전체가 참조할 수 있는 AI 코딩 정책 문서를 레포에 추가합니다.

**`TRUST_POLICY.md` 기본 구조:**

```markdown
# AI 코드 신뢰 정책

> 최종 업데이트: 2026-04-18 | 팀 합의 버전 v1.0

## 허용된 AI 도구

- Claude Code (claude.ai/claude-code)
- Cursor (v0.42+)
- GitHub Copilot

## AI 생성 코드 범위

### 자율 사용 허용
- 유틸리티 함수, 데이터 변환
- 테스트 코드 (단위, 통합)
- 문서화 (JSDoc, 주석)
- 반복 패턴 코드 (CRUD, 폼 핸들링)

### 리뷰 필수 (PR 체크리스트 적용)
- 비즈니스 로직 구현
- API 엔드포인트
- 상태 관리 로직

### 금지 (AI 단독 생성 불가)
- 인증/인가 핵심 로직
- 암호화/복호화 구현
- 프로덕션 DB 마이그레이션 스크립트

## 프롬프트 보안

- API 키, 시크릿, 개인정보를 프롬프트에 포함하지 않는다
- 프롬프트에 프로덕션 DB 데이터를 직접 넣지 않는다
- CLAUDE.md에 민감한 인프라 정보를 명시하지 않는다

## 정책 검토 주기

분기별 1회 팀 회고에서 이 문서를 검토하고 업데이트한다.
```

## Step 4: 신뢰 앵커 온보딩

신규 팀원이 합류할 때 신뢰 앵커를 빠르게 이해할 수 있도록 온보딩 경로를 설정합니다.

```
1. TRUST_POLICY.md 읽기 (10분)
2. 첫 PR에서 체크리스트 직접 적용 (첫 주)
3. AI 생성 코드 포함 PR 1건 리뷰어로 참여 (첫 2주)
4. 월간 팀 리뷰에서 케이스 공유 (지속)
```

**온보딩 확인 스크립트 (선택):**

```bash
#!/bin/bash
# scripts/check-trust-policy.sh

POLICY_FILE="TRUST_POLICY.md"
PR_TEMPLATE=".github/pull_request_template.md"

echo "신뢰 정책 구성 확인 중..."

[ -f "$POLICY_FILE" ] && echo "✓ TRUST_POLICY.md 존재" || echo "✗ TRUST_POLICY.md 없음"
[ -f "$PR_TEMPLATE" ] && echo "✓ PR 체크리스트 존재" || echo "✗ PR 체크리스트 없음"
grep -q "AI 코드 리뷰" "$PR_TEMPLATE" && echo "✓ AI 체크포인트 섹션 포함" || echo "✗ AI 체크포인트 섹션 없음"

echo "완료."
```

## Step 5: 월간 신뢰 검토 루틴

신뢰 앵커는 한 번 만들고 끝이 아닙니다. 실제 버그 사례, 통과된 케이스, 놓친 패턴을 팀이 함께 검토하면서 기준을 정교하게 만들어갑니다.

**월간 검토 항목:**

| 항목 | 확인 방법 |
|------|----------|
| 지난 달 AI 코드 관련 버그 수 | GitHub Issues 태그 필터 |
| 게이트 우회된 케이스 | CI 실패 → 강제 머지 로그 |
| 체크리스트 미준수 PR | PR 리뷰 히스토리 |
| 금지 영역 침범 사례 | 코드 리뷰 코멘트 검색 |

30분 회의로 4개 항목을 검토하고, 변경이 필요하면 TRUST_POLICY.md에 반영합니다.

## 체크리스트

- [ ] CI/CD에 자동 게이트 추가 (lint + security + test)
- [ ] PR 템플릿에 AI 체크포인트 섹션 추가
- [ ] TRUST_POLICY.md 초안 작성 + 팀 검토
- [ ] 고위험 영역 정의 + 2인 리뷰 규칙 설정
- [ ] 신규 팀원 온보딩 경로 정리
- [ ] 월간 검토 일정 캘린더에 등록

## 흔한 실수와 해결

| 실수 | 해결 |
|------|------|
| 체크리스트가 너무 길어 형식적으로 변함 | 5개 이하 핵심 항목만 유지, 나머지는 가이드 문서로 분리 |
| 자동 게이트만 믿고 리뷰를 건너뜀 | 고위험 영역 2인 리뷰를 브랜치 보호 규칙으로 강제 |
| 정책이 팀 일부만 알고 있음 | TRUST_POLICY.md를 README에 링크, 온보딩 체크리스트에 포함 |
| 정책이 오래되어 현실과 다름 | 분기별 검토 일정을 캘린더에 반복 이벤트로 등록 |
