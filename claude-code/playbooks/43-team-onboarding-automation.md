# 플레이북 43: AI 에이전트 온보딩 자동화

> 신규 팀원이 합류한 첫날부터 첫 번째 PR까지 — AI 에이전트로 온보딩 시간을 3일에서 반나절로 줄이는 실전 플레이북

## 소요 시간

30-60분 (초기 설정) + 신규 팀원당 2-4시간

## 사전 준비

- AI 코딩 에이전트 설치 완료 (Claude Code, Cursor 등)
- 레포지토리에 `AGENTS.md` 또는 `CLAUDE.md` 파일 존재
- 개발 환경 스크립트(`make setup` 또는 `./setup.sh`) 준비
- Git 접근 권한 및 기본 워크플로우 이해

## 왜 이 플레이북이 필요한가

새 팀원이 합류했을 때 가장 많은 시간이 소요되는 작업은 "코드베이스 이해"와 "환경 설정"이에요. 시니어 개발자가 옆에서 하루 종일 붙어 설명해줘야 하고, 그래도 "왜 이렇게 구조가 되어 있는지"는 며칠이 지나야 감이 오죠.

AI 에이전트를 활용하면 달라요. 코드베이스 전체를 읽고 즉각 답변하며, 작업 단위로 분해해서 첫 태스크까지 직접 안내해 줘요. 시니어 개발자의 부담도 줄고, 신규 팀원의 ramp-up 속도도 올라가요.

## Step 1: 온보딩 AGENTS.md 작성

기존 `AGENTS.md`와 별개로, 신규 팀원용 컨텍스트를 담은 `ONBOARDING.md`를 레포 루트에 추가해요.

```markdown
# ONBOARDING.md — 신규 팀원 가이드

## 프로젝트 개요
이 레포지토리는 [프로젝트 이름]을 위한 [간단한 설명]이에요.

## 아키텍처 핵심 결정
- **왜 [프레임워크]를 쓰는가:** [결정 이유]
- **왜 [폴더 구조]로 되어 있는가:** [배경]
- **절대 건드리면 안 되는 것:** [레거시 또는 민감한 코드 위치]

## 개발 환경 설정 순서
1. `make setup` 실행
2. `.env.example`을 `.env`로 복사 후 값 채우기
3. `make dev`로 로컬 서버 실행
4. `make test`로 테스트 통과 확인

## 첫 번째 PR 추천 작업
- `good-first-issue` 레이블이 붙은 이슈 확인
- 문서 개선 (README, 인라인 주석)
- 기존 테스트에 케이스 추가

## AI 에이전트에게 물어볼 수 있는 것
- "이 함수가 하는 일이 뭔가요?"
- "이 버그를 어떻게 재현할 수 있나요?"
- "테스트를 어떻게 작성하면 되나요?"
```

## Step 2: 코드베이스 탐색 태스크 자동화

신규 팀원이 AI 에이전트에게 줄 수 있는 표준 탐색 프롬프트 세트를 준비해요.

### 초기 오리엔테이션 프롬프트

```
ONBOARDING.md와 AGENTS.md를 읽은 뒤, 이 코드베이스의 전체 구조를
다음 순서로 설명해줘:

1. 폴더 구조와 각 폴더의 역할
2. 데이터 흐름 (요청이 어떻게 처리되는지)
3. 주요 의존성과 이유
4. 테스트 전략
5. 로컬에서 기능 하나를 추가한다면 어떤 파일을 건드려야 하는지

설명 후, 내가 이해를 확인할 수 있는 질문 3개를 내줘.
```

### 환경 설정 검증 프롬프트

```
make setup을 실행했는데 아래 에러가 발생했어:
[에러 메시지 붙여넣기]

이 에러의 원인을 분석하고, 해결 방법을 단계별로 알려줘.
해결 후 확인해야 할 사항도 포함해줘.
```

## Step 3: 첫 번째 태스크 안내 자동화

신규 팀원의 첫 번째 PR은 두 가지 목표를 동시에 달성해야 해요.
- 실제 코드 기여 (자신감 확보)
- 워크플로우 경험 (PR 프로세스 학습)

```bash
#!/bin/bash
# scripts/generate-first-task.sh
# 신규 팀원에게 맞는 첫 번째 태스크를 AI로 추천

REPO_CONTEXT=$(find . -name "*.md" -maxdepth 2 | head -5 | xargs cat)
OPEN_ISSUES=$(gh issue list --label "good-first-issue" --json title,number,body --limit 5)

echo "코드베이스 컨텍스트와 열린 이슈 목록을 바탕으로,
신규 팀원에게 적합한 첫 번째 태스크를 추천해줘.
난이도 순으로 3개, 각각 예상 소요 시간과 수정해야 할 파일 목록 포함.

이슈 목록: $OPEN_ISSUES"
```

### 태스크 실행 가이드 프롬프트

```
이슈 #[번호]를 구현하려고 해. AGENTS.md와 기존 코드를 참고해서:

1. 이 기능을 구현하려면 어떤 파일을 수정해야 하나?
2. 기존 코드 패턴과 일관성을 유지하려면 어떻게 작성해야 하나?
3. 어떤 테스트를 작성해야 하나?
4. PR 설명은 어떻게 쓰면 되나?

단계별로 안내해줘. 각 단계에서 내가 직접 작성하고 네가 검토하는 방식으로 진행하자.
```

## Step 4: 팀 지식 이전 세션

코드베이스 곳곳에 숨어있는 "구두로만 전달되던 지식"을 문서화해요.

| 지식 유형 | 전통적 이전 방식 | AI 보조 방식 |
|-----------|----------------|-------------|
| 코드 의도/배경 | 1:1 대화 1-2시간 | AI가 코드 읽고 추론 + 시니어 확인 10분 |
| 버그 재현 방법 | Slack 검색 + 구두 설명 | AI가 git blame + 이슈 분석 |
| 배포 절차 | 위키 + 구두 보완 | AI가 Makefile/CI 읽고 단계별 설명 |
| 코드 리뷰 기준 | PR 코멘트 경험 축적 | `AGENTS.md`에 명시 + AI 사전 검토 |

### 팀 노하우 추출 프롬프트

```
이 코드베이스에서 신규 개발자가 자주 실수하거나 의아해할 부분을
코드를 분석해서 목록으로 만들어줘.

각 항목에 대해:
- 어떤 혼란이 생기는지
- 실제 의도가 무엇인지
- 어떻게 설명하면 이해하기 쉬운지

이 내용을 ONBOARDING.md에 추가할 수 있는 형태로 작성해줘.
```

## Step 5: 온보딩 체크포인트 자동화

AI 에이전트가 신규 팀원의 이해도를 주기적으로 점검하는 루틴이에요.

```bash
#!/bin/bash
# scripts/onboarding-checkpoint.sh
# 첫 번째 PR 이후 실행

echo "온보딩 1주차 체크포인트를 진행할게요.
다음 질문에 답변해보세요:

1. 이번 PR에서 가장 어렵게 느꼈던 부분은?
2. 코드베이스에서 이해가 안 되는 부분이 있다면?
3. 개발 환경에서 불편하거나 고쳐지면 좋겠다고 생각한 것은?

답변을 기반으로 다음 태스크를 추천하고,
팀에게 공유할 온보딩 개선 사항을 정리할게요."
```

## 체크리스트

- [ ] `ONBOARDING.md` 파일 레포 루트에 추가
- [ ] 아키텍처 결정 사항과 배경 문서화
- [ ] `good-first-issue` 레이블 이슈 3개 이상 준비
- [ ] 환경 설정 스크립트(`make setup`) 신규 환경에서 테스트
- [ ] 온보딩 프롬프트 세트 팀 위키에 공유
- [ ] 1주차 체크포인트 일정 캘린더에 등록

## 기대 효과

| 지표 | 기존 | AI 보조 온보딩 |
|------|------|--------------|
| 환경 설정 시간 | 4-8시간 | 1-2시간 |
| 첫 번째 PR까지 | 3-5일 | 1-2일 |
| 시니어 개발자 투입 시간 | 8-16시간 | 2-4시간 |
| 코드베이스 파악 수준 (1주 후) | 기초 | 중급 |
