# 플레이북 52: AI 스페셜리스트 에이전트 역할 분담

> 보안, 성능, QA 전문가 역할을 서브에이전트에 부여해 코드를 다각도로 검증하는 플레이북

## 소요 시간

15-25분

## 사전 준비

- Claude Code 최신 버전
- 검토할 코드 또는 PR 브랜치
- 최소 2개 이상의 전문가 역할 정의

## 개념: 스페셜리스트 에이전트란?

일반적인 코드 리뷰는 한 사람(또는 하나의 에이전트)이 모든 관점에서 검토합니다. 문제는 모든 것을 잘 보려다 보면 아무것도 깊이 못 보는 경우가 생깁니다.

스페셜리스트 에이전트 방식은 다릅니다. 각 에이전트에게 **하나의 전문가 역할**만 부여하고 그 역할에 집중하게 합니다.

```
오케스트레이터
  ├── 보안 전문가 에이전트  → 인젝션, 인증, 시크릿 유출 검토
  ├── 성능 엔지니어 에이전트 → N+1, 알고리즘 복잡도, 메모리 누수 검토
  └── QA 에이전트           → 엣지 케이스, 테스트 커버리지, 회귀 위험 검토
```

각 에이전트는 자신의 전문 영역에서만 집중하기 때문에 일반 리뷰보다 놓치는 게 적습니다.

## Step 1: 역할 정의 파일 작성

프로젝트 루트에 `review-agents.md` 파일을 만들고 각 역할의 책임을 명확히 정의합니다.

```markdown
## 보안 전문가
- SQL 인젝션, XSS, CSRF 취약점
- 하드코딩된 시크릿 또는 API 키
- 인증/인가 로직의 허점
- 입력값 검증 누락
- 의존성 CVE 노출 여부

## 성능 엔지니어
- N+1 쿼리 패턴
- 불필요한 루프 또는 중첩 반복
- 캐시 가능한 연산 미캐시
- 메모리 누수 위험 패턴
- 비동기 처리 비효율

## QA 전문가
- 엣지 케이스 처리 누락
- 기존 테스트와의 충돌
- 회귀 위험이 있는 변경사항
- 에러 핸들링 일관성
- 명세와 구현의 불일치
```

## Step 2: 오케스트레이터 프롬프트 작성

Claude Code에 다음 프롬프트로 병렬 리뷰를 시작합니다.

```
review-agents.md의 역할 정의를 참고해서 다음 파일들을 3개의 서브에이전트로 병렬 검토해줘:
- src/auth/login.ts
- src/api/users.ts
- src/db/queries.ts

에이전트 1: 보안 전문가 — 보안 취약점만 집중 검토
에이전트 2: 성능 엔지니어 — 성능 문제만 집중 검토
에이전트 3: QA 전문가 — 테스트 품질과 엣지 케이스만 집중 검토

각 에이전트는 자신의 역할 외 항목은 언급하지 않는다.
결과는 역할별로 구분해서 보고한다.
```

## Step 3: 역할별 결과 병합

각 에이전트의 결과를 취합해 우선순위 매트릭스를 만듭니다.

| 이슈 | 발견 에이전트 | 심각도 | 파일 |
|------|-------------|--------|------|
| SQL 인젝션 위험 | 보안 전문가 | 높음 | queries.ts:42 |
| N+1 쿼리 패턴 | 성능 엔지니어 | 중간 | users.ts:87 |
| 빈 배열 처리 누락 | QA 전문가 | 낮음 | login.ts:23 |

**병합 규칙:**
- 같은 파일에서 여러 에이전트가 이슈를 발견하면 → 수정 우선순위 상향
- 보안 이슈 → 항상 최우선 수정
- 성능 + QA 이슈 → 비즈니스 영향도 기준으로 우선순위 결정

## Step 4: 검증 사이클

수정 후 해당 이슈를 발견한 에이전트에게만 재검토를 요청합니다.

```
[보안 전문가 에이전트에게]
queries.ts의 SQL 인젝션 이슈를 수정했습니다.
parameterized query 방식으로 변경했는데, 보안 관점에서 충분한지 검토해줘.
```

전체 재검토가 아닌 **발견자 재확인** 방식이기 때문에 토큰 소비가 적습니다.

## 역할 프리셋 템플릿

자주 쓰는 역할 조합을 미리 정의해두면 반복 작업이 줄어듭니다.

### 프론트엔드 검토 세트

```markdown
## 접근성 전문가
- WCAG 2.1 AA 기준 준수 여부
- 키보드 네비게이션 지원
- 스크린 리더 호환성
- 색상 대비율

## 번들 최적화 전문가
- 불필요한 import
- 코드 분할 기회
- 이미지 최적화 여부
- lazy loading 적용 가능 영역
```

### 백엔드 API 검토 세트

```markdown
## API 설계 전문가
- REST 규칙 준수
- 상태 코드 적절성
- 페이지네이션 누락
- 버전 관리 일관성

## 데이터 엔지니어
- 인덱스 활용 여부
- 트랜잭션 경계 적절성
- 데이터 일관성 위험
- 마이그레이션 안전성
```

## 체크리스트

- [ ] 역할별 책임 범위가 명확히 분리되어 있음
- [ ] 각 에이전트 프롬프트에 "역할 외 항목 언급 금지" 조건 포함
- [ ] 이슈 병합 시 심각도 기준을 미리 정의함
- [ ] 수정 후 발견 에이전트로 재검토 완료
- [ ] 반복적으로 쓰는 역할 세트를 파일로 저장함
