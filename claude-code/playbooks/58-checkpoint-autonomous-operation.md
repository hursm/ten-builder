# 플레이북 58: AI 에이전트 체크포인트 기반 장기 실행

> 장시간 자율 실행 중 상태를 저장하고 실패 시 복구하는 실전 패턴 — 체크포인트 전략, git 스냅샷, 롤백 흐름

## 소요 시간

20-30분 (초기 설정 기준)

## 사전 준비

- Claude Code 최신 버전 (체크포인트 기능 내장)
- 작업 중인 프로젝트 git 저장소
- 장시간 실행할 태스크 (빌드, 마이그레이션, 리팩터링 등)

---

## 체크포인트가 필요한 상황

AI 에이전트로 30분 이상 걸리는 태스크를 실행하다 보면 중간에 실패할 경우가 있다. 네트워크 끊김, 컨텍스트 한도 초과, 잘못된 방향으로의 진행 등 이유는 다양하다.

체크포인트가 없으면 두 가지 문제가 생긴다:

| 문제 | 결과 |
|------|------|
| 실패 후 처음부터 재시작 | 이미 완료된 작업까지 다시 수행 → 시간 낭비 |
| 에이전트가 방향을 잃었을 때 돌아갈 지점 없음 | 코드가 원래보다 나빠지는 방향으로 계속 진행 |

Claude Code의 체크포인트는 이 문제를 두 가지 방식으로 해결한다: **자동 체크포인트**(파일 수정 전 자동 저장)와 **수동 git 스냅샷**(직접 커밋해두는 체크포인트).

---

## Step 1: Claude Code 자동 체크포인트 이해

### 1-1. 자동 체크포인트 동작 방식

Claude Code는 파일을 수정하기 전에 자동으로 체크포인트를 생성한다. `Esc` 키를 두 번 누르면 가장 최근 체크포인트로 즉시 복구된다.

```bash
# 체크포인트 복구 단축키
Esc + Esc   # 직전 체크포인트로 즉시 롤백
```

체크포인트는 로컬 git 히스토리에 저장되므로 세션이 종료되어도 유지된다.

### 1-2. 체크포인트 히스토리 확인

```bash
# Claude Code가 생성한 체크포인트 커밋 목록
git log --oneline | head -20

# 특정 체크포인트와 현재 상태 비교
git diff HEAD~3
```

---

## Step 2: 수동 git 스냅샷 체크포인트 전략

자동 체크포인트만으로는 부족한 경우에 직접 저장 지점을 만든다.

### 2-1. 태스크 시작 전 스냅샷

```bash
# 태스크 시작 전 현재 상태 저장
git stash push -m "checkpoint: 태스크명 시작 전 $(date '+%Y-%m-%d %H:%M')"

# 또는 커밋으로 저장
git add -A
git commit -m "checkpoint: 대규모 리팩터링 시작 전"
```

### 2-2. 단계별 중간 스냅샷

긴 태스크는 논리적 단계별로 커밋을 나눈다.

```bash
# 1단계 완료 후
git add src/
git commit -m "wip: 1단계 완료 — 데이터 모델 변환"

# 2단계 완료 후
git add tests/
git commit -m "wip: 2단계 완료 — 테스트 업데이트"
```

에이전트에게 단계별 커밋을 직접 지시할 수도 있다:

```
각 주요 단계가 끝날 때마다 git commit을 해줘.
커밋 메시지 형식: "wip: {단계 번호} — {완료된 내용 한 줄 요약}"
```

### 2-3. CLAUDE.md에 체크포인트 규칙 등록

```markdown
## 자율 실행 규칙

- 파일 30개 이상 수정 전: git commit으로 중간 저장
- 에러가 3회 반복되면: 작업 중단 후 현재 상태 보고
- 테스트 실패 시: 자동 수정 시도 2회 초과 금지
```

---

## Step 3: 복구 흐름

### 3-1. 자동 체크포인트에서 복구

에이전트가 잘못된 방향으로 가고 있을 때:

```bash
# 1. 에이전트 실행 중단
Ctrl+C

# 2. 현재 변경사항 확인
git diff

# 3. 원하는 지점으로 롤백
git reset --hard HEAD~2    # 2개 커밋 전으로 되돌리기
# 또는
git stash pop              # stash로 저장한 체크포인트 복구
```

### 3-2. 부분 복구

일부 파일만 롤백하고 나머지는 유지할 때:

```bash
# 특정 파일만 이전 버전으로 복구
git checkout HEAD~1 -- src/database/schema.ts

# 복구 확인
git diff --staged
```

### 3-3. 복구 후 재시작 지시

```
직전 시도는 취소됐어. 현재 코드베이스 상태에서 다시 시작해줘.
이번엔 src/database/ 디렉토리만 수정하고, 다른 파일은 건드리지 마.
```

## Step 4: 장기 실행 태스크 설계 패턴

### 4-1. 태스크 분해

8시간짜리 태스크를 한 번에 맡기면 실패 확률이 높다. 1-2시간 단위로 나눠서 순차 실행하는 게 효과적이다.

| 태스크 크기 | 권장 방식 |
|------------|----------|
| 30분 이하 | 한 번에 실행 |
| 30분~2시간 | 중간 스냅샷 1-2개 |
| 2시간 이상 | 서브태스크로 분해 + 단계별 검증 |

### 4-2. 검증 게이트 설정

각 단계 완료 후 에이전트가 스스로 검증하도록 지시한다:

```
각 단계 완료 후 다음을 확인해:
1. 기존 테스트가 통과하는지 (npm test)
2. 변경된 파일 목록을 출력
3. 예상과 다르면 작업 중단 후 보고

모든 체크 통과 시에만 다음 단계로 진행해.
```

### 4-3. 타임아웃 설정

```bash
# 일정 시간 초과 시 에이전트 자동 종료
timeout 3600 claude "리팩터링 작업 실행..."

# 실행 후 상태 확인
echo $?   # 0이면 정상 완료, 124면 타임아웃
```

## 문제 해결

| 상황 | 해결 |
|------|------|
| stash pop 충돌 발생 | `git stash drop`으로 stash 삭제 후 `git diff`로 수동 병합 |
| 체크포인트 커밋이 너무 많아 히스토리 지저분 | 태스크 완료 후 `git rebase -i`로 wip 커밋 정리 |
| 에이전트가 체크포인트 무시하고 계속 진행 | CLAUDE.md에 중단 조건 명시 + `--no-continue` 플래그 사용 |
| 어떤 체크포인트로 돌아갈지 모를 때 | `git log --oneline --graph`로 분기점 시각 확인 |
