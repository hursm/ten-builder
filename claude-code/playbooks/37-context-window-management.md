# AI 에이전트 컨텍스트 윈도우 관리 플레이북

> 대규모 코드베이스에서 AI 에이전트의 컨텍스트 한계를 넘지 않고 최대 성능을 뽑아내는 실전 가이드

## 왜 컨텍스트 윈도우 관리가 중요한가?

AI 코딩 에이전트에게 프로젝트 전체를 한 번에 넘기면 성능이 떨어져요. 토큰 수가 늘어날수록 응답 품질이 낮아지고, 비용은 올라가고, 속도는 느려져요. 10만 줄짜리 프로젝트를 다룬다면 **무엇을 넣고 무엇을 빼는가**가 결과를 결정해요.

| 문제 | 증상 | 원인 |
|------|------|------|
| 응답 품질 저하 | 엉뚱한 파일을 수정 | 관련 없는 코드가 컨텍스트를 차지 |
| 할루시네이션 | 존재하지 않는 함수 참조 | 파일 구조 파악 실패 |
| 느린 응답 | 30초 이상 대기 | 토큰 처리량 과부하 |
| 비용 폭증 | 단순 수정에 $2+ | 불필요한 파일 반복 로딩 |
| 컨텍스트 초과 | 대화 중간에 잘림 | 누적 토큰이 한계 초과 |

## 소요 시간

15-30분 (프로젝트 규모에 따라 다름)

## 사전 준비

- AI 코딩 에이전트 설치 (Claude Code, Cursor, Windsurf 등)
- 50개 이상 파일이 있는 프로젝트
- 프로젝트의 디렉토리 구조 기본 이해

## Step 1: 컨텍스트 예산 계획

프로젝트 규모에 따라 컨텍스트 전략이 달라져요.

| 프로젝트 규모 | 파일 수 | 전략 |
|--------------|---------|------|
| 소형 | < 20개 | 전체 로딩 가능, 별도 관리 불필요 |
| 중형 | 20-100개 | 선택적 로딩 + 구조 파일 활용 |
| 대형 | 100-500개 | 레이어링 + 서브에이전트 분할 |
| 초대형 | 500개+ | 맵-리듀스 방식 필수 |

**컨텍스트 예산 분배 원칙:**

```
전체 컨텍스트 (100%)
├── 시스템 프롬프트 + 규칙 파일  (10-15%)
├── 프로젝트 구조 맵             (5-10%)
├── 작업 대상 파일               (40-50%)
├── 참조 파일 (타입, 인터페이스)   (15-20%)
└── 대화 히스토리                 (10-20%)
```

## Step 2: 프로젝트 구조 맵 작성

에이전트가 전체 파일을 읽지 않아도 프로젝트를 파악할 수 있게 **구조 맵**을 만들어요.

```markdown
# PROJECT_MAP.md

## 디렉토리 구조
src/
├── api/          # REST API 엔드포인트 (Express)
├── services/     # 비즈니스 로직
├── models/       # DB 모델 (Prisma)
├── utils/        # 공통 유틸리티
└── types/        # TypeScript 타입 정의

## 핵심 파일 (우선 참조)
- src/types/index.ts — 전체 타입 정의
- src/services/auth.ts — 인증 로직
- prisma/schema.prisma — DB 스키마

## 의존성 흐름
API → Service → Model → DB
```

**프로젝트 맵을 `CLAUDE.md`나 `.cursorrules`에 포함하면** 에이전트가 매번 `find` 명령을 실행하지 않아도 돼요.

## Step 3: 선택적 파일 로딩 패턴

### 패턴 1: 타입 우선 로딩

```bash
# 에이전트에게 먼저 타입 파일만 읽게 하기
"src/types/index.ts를 먼저 읽고,
 그 다음 src/services/payment.ts를 수정해줘"
```

타입 파일(200줄)을 먼저 로딩하면 에이전트가 프로젝트의 데이터 구조를 이해한 상태에서 작업해요. 타입 없이 바로 서비스 파일을 수정하면 잘못된 필드명을 쓸 확률이 높아져요.

### 패턴 2: 관련 파일만 지정

```bash
# 나쁜 예 — "전체 프로젝트에서 버그 찾아줘"
# 좋은 예 — 범위를 좁혀서 지시
"src/services/payment.ts와 src/api/payment.route.ts에서
 결제 실패 시 재시도 로직을 추가해줘.
 참조: src/types/payment.ts"
```

### 패턴 3: 점진적 확장

```bash
# 1단계: 구조 파악
"프로젝트 구조를 파악하고 주요 파일 목록을 정리해줘"

# 2단계: 대상 좁히기
"인증 관련 파일만 나열해줘"

# 3단계: 구체적 작업
"src/services/auth.ts의 refreshToken 함수에
 만료 시간 검증 로직을 추가해줘"
```

## Step 4: 컨텍스트 리셋 전략

대화가 길어지면 컨텍스트가 누적돼서 초기 지시사항을 잊어요. 이걸 방지하려면 적절한 시점에 리셋해야 해요.

**리셋 타이밍:**

| 신호 | 조치 |
|------|------|
| 응답에서 이전에 삭제한 코드를 다시 생성 | 즉시 리셋 |
| "어떤 파일을 수정하고 있었죠?" 같은 질문 | 즉시 리셋 |
| 같은 실수를 3번 반복 | 즉시 리셋 |
| 5개 이상 파일 수정 완료 | 예방적 리셋 |
| 대화 턴 20회 초과 | 예방적 리셋 |

**리셋 방법:**

```bash
# Claude Code — compact 명령으로 대화 요약
/compact

# 또는 새 세션 시작 + 이전 작업 요약 전달
"이전 세션에서 src/services/auth.ts에
 refreshToken 만료 검증을 추가했어.
 이어서 테스트 코드를 작성해줘."
```

## Step 5: 대규모 프로젝트 분할 패턴

500개 이상 파일이 있는 프로젝트에서는 **작업 단위를 분할**해야 해요.

### 맵-리듀스 패턴

```bash
# Map: 각 모듈별로 독립 작업
"src/modules/auth/ 디렉토리의 모든 함수에
 입력 검증을 추가해줘"

"src/modules/payment/ 디렉토리의 모든 함수에
 입력 검증을 추가해줘"

# Reduce: 결과 통합
"auth와 payment 모듈의 검증 패턴을
 src/utils/validator.ts로 통합해줘"
```

### 서브에이전트 패턴

```bash
# 메인 에이전트: 계획 수립
"프로젝트 전체 구조를 파악하고,
 성능 최적화가 필요한 파일 목록을 만들어줘"

# 서브에이전트 1: 특정 모듈 작업
"src/api/ 폴더의 라우터 성능을 최적화해줘"

# 서브에이전트 2: 다른 모듈 작업
"src/services/ 폴더의 DB 쿼리를 최적화해줘"
```

### CLAUDE.md 컨텍스트 압축

프로젝트 루트의 규칙 파일에 핵심 정보를 압축해서 넣으면 매 세션마다 파일을 다시 읽을 필요가 줄어요.

```markdown
# CLAUDE.md

## 기술 스택
- Next.js 15 + TypeScript + Prisma + PostgreSQL

## 코딩 규칙
- snake_case (DB) ↔ camelCase (API) 자동 변환: src/utils/transform.ts
- 에러 처리: 반드시 AppError 클래스 사용
- API 응답: { success, data, error } 형식

## 자주 수정하는 파일
- src/services/*.ts — 비즈니스 로직
- src/api/routes/*.ts — API 엔드포인트
- prisma/schema.prisma — DB 스키마 변경 시
```

## Step 6: `.claudeignore`로 불필요한 파일 차단

에이전트가 자동으로 읽어들이는 파일 중 불필요한 것을 차단해요.

```gitignore
# .claudeignore
node_modules/
dist/
build/
.next/
coverage/
*.lock
*.log
*.min.js
*.map
__pycache__/
.pytest_cache/
*.pyc
```

**차단 효과:**

| 차단 대상 | 절감 토큰 (대략) |
|-----------|----------------|
| `node_modules/` | 수백만 토큰 |
| `dist/`, `build/` | 수십만 토큰 |
| `*.lock` 파일 | 5-10만 토큰 |
| 소스맵 (`*.map`) | 10-50만 토큰 |
